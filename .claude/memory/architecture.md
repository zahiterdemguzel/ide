# Architecture

Standard Electron main/preload/renderer split. The renderer is pure UI with **no** Node or OS access — everything that touches the OS lives in the main process and is reached over IPC declared in `preload/`.

```
renderer/  ──IPC(preload/)──►  main/  ──►  node-pty / git / http hook server
  (UI)                          (OS)
```

## The main process — three subsystems

1. **PTY manager.** One `pty.spawn(claude, ...)` per session, keyed by a generated UUID. The UUID is passed as `--session-id` so it round-trips inside hook payloads — that is how a hook event is matched back to its sidebar row. Sessions live in the `sessions` Map; `pty-input` / `pty-resize` / `kill-session` operate on it by id. Each record stores the **`repo`** it was created in (`getRepoPath()` at `new-session`), so a session is bound to its project: it spawns/resumes with that folder as `cwd` regardless of which folder is currently open, the snapshot persists it, and the renderer shows only the open folder's sessions. Each record also stores the session's **agent model choice** (`model`/`subagentModel`), which `sessionEnv()` turns into the CLI's `ANTHROPIC_MODEL`/`CLAUDE_CODE_SUBAGENT_MODEL` env at spawn (and re-applies on resume), so a session keeps running the model it was created with — see [settings.md](settings.md#agent-models). The app does not set the reasoning-effort env; a user changes effort with the CLI's own `/effort` command. Switching folders (`setSessionsRepo()`) re-filters the list (on top of the active/archived tab) without tearing down the other projects' live terminals. **Archiving** a session (the sidebar ×) doesn't kill it — `suspend-session` kills only the Claude PTY to free its resources while keeping the session record (and all its tracked-file state) with `pty: null` and `suspended: true`; the `onExit` handler sees `suspended` and skips the usual teardown. **Restoring** it (`resume-session`) respawns with `claude --resume <id>`, so the conversation continues under the same id and `recordSessionActivity()` keeps accumulating `edits`/`fileOps` against the same record — any uncommitted work tracked before archiving survives and stays committable. The renderer disposes the xterm terminal on archive and rebuilds it on restore (`attachTerminal()`), keeping its `files` list intact throughout. Each session record also accumulates **what the agent did**: `firstPrompt` (its first `UserPromptSubmit`); `edits` — a `Map<absPath, op[]>` recording every `PostToolUse` of a file tool (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) as a replayable op (`editOp()`); and `fileOps` — a `Map<absPath, 'add'|'delete'>` of **path-level changes a text op can't express** (a binary file a `Bash`/MCP tool created, or a file it renamed/moved/deleted), filled by diffing the git working tree across a tool call (see [Tracking filesystem changes](#tracking-filesystem-changes)). `recordSessionActivity()` fills all of these from the hook stream and pushes a `session-meta` event to the renderer (the file list is `trackedFiles()` = the union of `edits` and `fileOps` keys); `commit-session` replays the ops and folds the path-level changes to commit just that session's work (see [Per-session commit](#per-session-commit)). On the first prompt it also fires `generateSessionName()` — a Haiku call via the shared `runHaiku()` helper (prompt capped at 2000 chars) asking for a 2-4 word title, pushed to the renderer as `session-name`. See [Haiku generation](#haiku-generation) for how `runHaiku()` reaches the model.
2. **Hook → status server.** A Node `http` server on a random `127.0.0.1` port. See [status-detection.md](status-detection.md).
3. **Git.** Plain `git` porcelain via `execFile` in `repoPath` (no git library). Every call goes through the shared `git()` helper, which prepends `-c core.quotePath=false` (so non-ASCII paths come back verbatim and round-trip to `add`/`diff` instead of being C-quoted), sets `GIT_TERMINAL_PROMPT=0` (a remote that wants credentials with no helper fails fast instead of hanging on an unanswerable prompt), and applies a 120s `timeout` backstop so a stuck network op can't wedge the UI. `git status --porcelain=v1` is parsed into staged/unstaged lists **and a separate `conflicts` list** — unmerged entries (`DD/AU/UD/UA/DU/AA/UU`, both columns set) are pulled out so they aren't double-listed with wrong-direction stage/unstage actions; the renderer shows them in a dedicated Conflicts section (click to view, ✓ to `git add` once resolved). stage = `git add`, unstage = `git reset HEAD` (falls back to `git rm --cached` for a repo with no commits); also `git commit -m` (called with an empty message, it stages-all-if-needed then asks Haiku to author the message from the staged diff — `generateCommitMessage()`, the shared `runHaiku()` helper in `claude.js` (see [Haiku generation](#haiku-generation)), capped at 12000 chars in / 1000 out; the message used is returned to the renderer), `git-undo` (`git reset --soft HEAD~1` — undo the last commit, keep its changes staged), `git push` (a branch with no upstream fails with "has no upstream branch", so the handler retries once with `push -u origin HEAD` — the common first-push case), `git-fetch`, and `git-pull` (`pull --ff-only`, so a pull that would need a merge fails cleanly instead of opening an editor with no tty or leaving conflicts — the user can then hand the merge to a Claude session, see [Hand a merge to Claude](#hand-a-merge-to-claude)). Both `git-pull` and `git-push` tag their result with **`needsMerge`** — true when the failure is a *divergence the remote moved past* (the pure `pullNeedsMerge`/`pushNeedsMerge` in `git-parse.js` match git's ff-only refusal / push-rejection wording, and *only* those, so auth/network/no-upstream failures aren't mistaken for a merge), which the renderer turns into the Claude offer. `commit-session` commits **only the hunks a single session edited** — see [Per-session commit](#per-session-commit). `git-diff` returns a file's unified diff (`--cached` for staged, `--no-index /dev/null <file>` for untracked) for the center-pane diff viewer. `git-revert` discards a file's changes — `git clean` for untracked, `git restore --staged --worktree` otherwise. The **History tab** is backed by `git-log` (last 100 commits, unit-separator-delimited `hash/short/subject/author/relDate` fields), `git-commit-diff` (`git show --format=<hash>` — the bare patch for the center diff viewer), `git-revert-commit` (`git revert --no-edit <hash>` — a non-destructive revert commit, the right tool for **pushed** history), and `git-undo-commit` (drops an **unpushed** commit by rewriting history: a soft reset for the HEAD commit so its changes stay staged, else `git rebase --onto <hash>^ <hash>`, with a `rebase --abort` fallback if the replay conflicts). `open-folder` runs `git rev-parse --show-toplevel` (`repoRoot()`) so porcelain paths and add/reset line up regardless of which subfolder the user picks. The chosen folder is prepended to `recent-folders.json` in the shared data dir (`instance.js`'s `sharedDataDir`, not the per-instance profile — so it persists across runs and is shared by all instances). **No project opens by default on startup**: `repoPath` stays `null` until the user picks a recent project or browses (the recent-projects menu auto-opens on launch — see the renderer's `index.js` bootstrap); `git()`/`gh()` short-circuit with `no folder open`, and the explorer/run-config watchers skip registration until a folder is opened. A **`--folder <path>` CLI flag** (alias `--dir`, both `--folder X` and `--folder=X` forms; parsed by the Electron-free `cli-args.js`'s `parseFolderArg`) overrides this for a single launch: when present and the path exists it becomes the initial `repoPath` directly — it is **not** written back to the recent list, so the override is ephemeral (it rides through `npm start` via `npm start -- --folder <path>`). The **Start IDE** launch config passes no `--folder`, so a dev launch starts with no project open — the recent-projects picker — matching the real first-run experience. **Branches**: `gitStatus` also returns the checked-out branch name (`currentBranch()` = `git rev-parse --abbrev-ref HEAD`, `'HEAD'` when detached) for the header indicator; `git-branches` lists local branches (`git for-each-ref --sort=-committerdate refs/heads`, most-recent first) plus the current one for the selector popover, `git-checkout` switches to a branch (`git checkout <branch>`, failing cleanly when the worktree would be clobbered), and `git-create-branch` creates and switches to a new branch off the current HEAD (`git checkout -b <branch>`, surfacing git's rejection of invalid ref names). **Non-git folder**: `git-is-repo` (`git rev-parse --is-inside-work-tree`) tells the renderer whether to show the normal pane or the **create-repository panel** (see [Create a repository](#create-a-repository)); `create-repo({name, description, isPrivate})` runs the whole publish flow — `git init` + `symbolic-ref HEAD refs/heads/main` (default branch **main**, set this way so it works on git versions predating `init -b`), `git add -A`, an `Initial commit` (`--allow-empty` fallback for an empty folder), then `gh repo create <name> --private|--public [--description …] --source . --remote origin --push` via the GitHub CLI — returning `{ ok, step, error }` so the renderer can report which stage failed (a missing `gh` comes back as an install hint). The name validation + `gh` argv assembly are the pure, unit-tested `repo-create.js`.

## Haiku generation

`runHaiku()` (in `claude.js`) backs every short generation (commit messages, session titles). It has two paths:

1. **Fast path — direct Messages API.** A one-shot `claude -p` cold-starts the whole CLI (Node + plugins + hooks + MCP) and sends its full agentic system prompt + tool definitions on every call — ~5s for a one-line commit message. Instead `runHaiku()` calls `https://api.anthropic.com/v1/messages` directly with the OAuth token Claude Code already stores (`claudeAiOauth.accessToken`), a one-line system prompt, and no tools: **~1s, no process boot, no new dependency or API key.** `readOauthToken()` reads that token from wherever the CLI keeps it for the platform — the `~/.claude/.credentials.json` file on Linux/Windows, or the **macOS Keychain** (generic password, service `Claude Code-credentials`) on darwin, where no file exists. It tries the file first, then falls back to the Keychain via `security find-generic-password`; both yield the same JSON blob. (Miss this and the whole fast path — and the usage meter below — silently fails on macOS, since every call falls back to the slow CLI and `readUsage()` returns null.) Subscription (OAuth) tokens are only accepted when the **first system block is the Claude Code identity string** (`"You are Claude Code, …"`) — the same one the CLI sends; a second system block carries the "output only the requested text" rule. Sent with headers `authorization: Bearer <token>`, `anthropic-version: 2023-06-01`, `anthropic-beta: oauth-2025-04-20`.
2. **Fallback — the CLI.** The OAuth token is short-lived; `readOauthToken()` returns null when it's missing or within 60s of `expiresAt`, and any non-200/transport error also yields null. In those cases `runHaiku()` falls back to the original one-shot `claude -p --model haiku` (prompt over stdin). That path is slow but always works **and refreshes the stored token on disk**, so the fast path heals itself on the next call. Both callers already treat a null result as "Haiku unavailable" and fall back further (the session title for a per-session commit, an error for the main git pane).

The model id is pinned (`claude-haiku-4-5-20251001`) because the raw API needs a full id, not the `haiku` alias; if it ever 404s the call returns null and the CLI fallback (which does accept the alias) takes over.

## Usage meter

The top toolbar shows the user's remaining **Claude subscription usage** against the two rolling windows — the **5-hour** session limit and the **weekly** limit — as two **vertically stacked** sliders (5-hour on top, weekly below) with a "resets in …" countdown, just left of the browser/settings buttons. The run strip (`#toolbar-runs`) flex-grows and scrolls internally, so the meter (and the browser/settings buttons) stay pinned to the right edge no matter how many run buttons there are. The meter is **hidden entirely when there's no usable OAuth token** — a machine without Claude Code installed, or one not yet signed in — so it never shows during first-run setup.

There is **no CLI command or dedicated endpoint** for this — `/usage` is interactive-only, and `count_tokens` returns no rate-limit headers (verified live). The data rides on the **unified rate-limit response headers of a real `/v1/messages` call**: `anthropic-ratelimit-unified-5h-utilization`/`-5h-reset`, the `7d` pair, and `-representative-claim` (`five_hour`/`seven_day`, the current bottleneck). Utilization is a `0..1` fraction; reset is **epoch seconds**. So `readUsage()` (in `claude.js`) makes the **smallest possible** call — `max_tokens: 1` (≈ one output token), the same OAuth token + Claude Code identity system block as `runHaiku()` — and reads the headers off the response; it returns null when there's no usable OAuth token or the call fails. The renderer polls `get-usage` (sessions.js) **every 30s, but only while the window has focus** (an unfocused/background app skips the tick so it doesn't keep spending a token per round-trip; a `window` `focus` listener refreshes immediately on return) and hides the meter on null (an API-key user, or a transient failure), so the cost is one token and no perceptible latency.

**Out of usage (429).** The rate-limit headers ride on the **429** you get once you've exhausted a window too — that's exactly when the meter matters most, so `fetchUsage` parses the headers **regardless of HTTP status** (was a 200-only gate that made the meter vanish precisely when you hit the limit). A hard rejection can drop the per-window `-utilization` headers and report only the overall `anthropic-ratelimit-unified-status: rejected` + `-reset`; `parseUsageHeaders` then **synthesizes a fully-used window** (utilization `1`, keyed off `-representative-claim`, default `5h`) so the meter still shows `100% · resets in …`. It requires a concrete `-reset` for that fallback, so a bare status with no data still yields null (meter hides).

The header→view-model parsing is the Electron-free, unit-tested `usage-parse.js` (`test/usage-parse.test.js`): `parseUsageHeaders` (both windows, utilization clamped to `0..1`, claim mapped to the `5h`/`7d` key, null when the headers are absent), `formatResetShort` (ms-until-reset → compact `24m`/`13h`/`2d`/`now`), and `usageView` (the assembled `{windows:[{key, utilization, resetIn, resetAt, representative}]}`). The renderer (`usage-meter.js`) renders each window's fill width from utilization and colours it by level — green (<50%) / `warn` yellow (≥50%) / `crit` red (≥80%) off the `--green`/`--yellow`/`--red` palette vars — and re-translates labels via `data-i18n` (`usage.*` keys); both windows' labels are styled identically. The two rows share a 3-column grid (`label · bar · reset`, each `.um-win` is `display:contents`) so the bars line up regardless of label width or locale. reset countdowns refresh on the next poll.

## Per-session token meter

Distinct from the subscription [Usage meter](#usage-meter) above: each session's terminal shows its **own** cumulative token count and dollar cost on a status line at the very bottom, driven by Claude Code's native **statusLine** feature — no app UI, no terminal scraping. It rides on the same `claude --settings <json>` flag that already injects the hooks (`hooksSettings(port, statusLineCommand)` in `hook-events.js`), so the user's global settings are never touched: when a command is supplied, a `statusLine: { type: 'command', command }` entry is added alongside the `hooks`.

Claude pipes the session's status JSON to that command's stdin on each refresh and renders whatever it prints. The command is `node <script>`, where the script (`statusline-script.js`) reads stdin and emits `model · $cost · tokens` via the pure, unit-tested `statusline-lib.js` (`formatStatusLine`). The **metrics are right-aligned**: the model sits on the left and the `$cost · tokens` block is pushed to the terminal's right edge using the width Claude exposes in the `$COLUMNS` env var (stdout is piped, so `process.stdout.columns` is unset). The padding is placed *between* the two halves rather than leading, so the alignment survives even if Claude trims leading/trailing whitespace; when `$COLUMNS` is absent or too narrow it falls back to a plain ` · `-joined inline line. The **cost is placed *before* the token count**: in a right-aligned line Claude clips the right edge, so the (less critical) token count absorbs any truncation while the dollar — to its left — stays fully visible. A 1-column right `margin` plus `padding: 0` on the statusLine config further keep the metrics off the edge. The cost renders in the same default colour as the rest of the line (no ANSI emphasis). Claude Code's `$COLUMNS` and `padding` support are what make the alignment work (`statusline-lib.js`'s `formatStatusLine` takes `{ width, margin }`).

**Toggle.** A **Settings → General → Token cost meter** switch (default on) gates the whole thing. Because the statusLine is injected in main at spawn, the renderer persists the flag in `localStorage` *and* pushes it to main over IPC (`set-statusline-enabled` → `statusline.setEnabled`); `statusLineCommand()` returns null when off, so the next session spawns with no statusLine while live sessions keep theirs. See [settings.md](settings.md#general-preferences). **Tokens are summed from the transcript, not stdin**: the stdin `context_window` counts are *current context size*, not the session total, so `sumTranscriptTokens()` reads `transcript_path` and totals all four token classes (`input`/`output`/`cache_creation`/`cache_read`) across every assistant call — the same basis the cost is computed on. **Cost is taken verbatim** from stdin's `cost.total_cost_usd` (already cumulative), so there's no pricing table to maintain. A leading BOM is stripped before parsing and any bad/empty stdin degrades to `0 tokens · $0.00`.

**Why a script staged on disk.** Claude runs the statusLine command as an external process, so the script must live on real disk — an external `node` can't read inside the packaged asar archive. `statusline.js` copies both files (`statusline-script.js` + its `statusline-lib.js` sibling) out of the app bundle into `sharedDataDir/statusline/` once per launch (Electron's fs reads the asar source transparently) and returns the `node "<path>"` command; it returns null if staging fails, in which case no statusLine is injected. **`node` is assumed on PATH** (a developer's machine running a Node CLI); if it's absent Claude simply renders no status line and the session is otherwise unaffected.

**On a subscription the dollar figure is notional** — `total_cost_usd` is computed at API rates, but a Pro/Max plan isn't billed per token (the truthful "how much of my plan" number is the [Usage meter](#usage-meter)); this line answers "how heavy is this session".

## Claude Code setup gate

The whole app is a front-end for the `claude` CLI, so a user who hasn't installed Claude Code is guided through it before any session can spawn. The check runs on **every** app launch (nothing is persisted to skip it after the first run), so an install that later breaks or is removed is caught again. `claudeAvailable()` (in `claude.js`) is the detector: it clears the cached `resolveClaude()` path (so a fresh install is picked up on re-check) and runs `claude --version` with a 10s timeout — the truest "actually usable" test, since a stale PATH entry that no longer launches still counts as missing; any failure (not on PATH, non-zero exit, timeout) resolves to `{ installed: false }`. The `check-claude` IPC handler (`sessions.js`) returns that result plus `installGuide()` from the Electron-free, unit-tested `claude-install.js` (`test/claude-install.test.js`).

`installGuide(platform)` gives, as one object: `installArgs`, `authArgs`, `installOk`, and `installFail`. The crucial design choice is **how the commands run**: not by *typing* them into an interactive shell prompt, but by spawning the shell with the command as **argv** (`zsh -ilc '<cmd>'`). Typing into a freshly-spawned interactive shell races its line editor — zsh's ZLE on macOS swallowed the command, so the install terminal looked alive but did nothing. Passing the command as a shell argument runs it directly and reliably. `installArgs`/`authArgs` are therefore arrays handed straight to `pty.spawn` (via the new `args` field on `term-create`); the shell path itself stays main's choice (`availableShells()[0]` — `$SHELL` on POSIX, `powershell.exe` on Windows).

The native installer needs nothing pre-installed on any platform (Windows ships PowerShell's `irm` + .NET, macOS/Linux ship curl + bash), so there is a single install path and no npm fallback. POSIX uses `-il` (login + interactive) so the shell sources the user's profile/rc and inherits their **real** PATH/env — a GUI-launched macOS app otherwise gets a stripped-down launchd environment (not what the user has in Terminal.app), which is why an install could appear to run yet not take. Re-sourcing the rc also puts the freshly-installed `claude` on PATH for the sign-in step (`authArgs` = `['-ilc', 'claude']`). Windows PowerShell has no login concept and inherits the full env, so `-NoExit -Command` is enough, with the auth command re-reading PATH from the registry (the installer wrote it there) before launching `claude`. Only the non-translatable shell commands live in the helper; all surrounding prose is i18n (`setup.*` keys).

**`installOk` / `installFail` — knowing the outcome.** The install runs in an embedded terminal whose shell stays alive afterward, so there's no exit code to read. The install command echoes `installOk` on success and `installFail` otherwise, keyed on the installer's exit status (`… && echo OK || echo FAIL` on POSIX, `…; if ($?) { echo OK } else { echo FAIL }` on Windows). The renderer watches the terminal output for whichever appears: **OK** enables Next, **FAIL** keeps it disabled and shows a retry message. (An earlier single "completion" marker chained with `;` printed even on failure, so a failed install was reported as success — the bug this replaced.) Because the command is spawned as argv rather than typed, the shell never echoes the command line, so no quoting trick is needed to keep the marker out of a command echo.

The renderer side is `claude-setup.js` — a **three-step wizard** (`#claude-setup-dialog`) with a numbered stepper: **Install → Sign in → Done**. `initClaudeSetup()` runs on every launch (after `initSettings` so it's translated, after `initConsoles` so the terminal subsystem is up) and opens the wizard when Claude Code is missing. The flow is deliberately easy — **Next → Next → authenticate in the terminal → Finish**:

- **Intro** (step 1) just states Claude Code isn't installed and offers **Next**; the detected platform is shown as a hint.
- **Install** (step 2): entering it spawns an xterm terminal *embedded in the dialog* (the same `shared/terminal.js` helpers the sessions/consoles use) with `guide.installArgs` as the shell argv via `term-create`. **Next stays disabled** until the renderer sees `guide.installOk` in the terminal output; `guide.installFail` keeps it disabled and shows a retry message. The note tells the user to wait and click Next when it finishes.
- **Sign in** (step 3): the install terminal is **torn down** (killing that shell) and a fresh terminal is spawned with `guide.authArgs` — i.e. `claude` running directly — so the login flow (browser or paste-code) runs inline. **Finish** verifies with a real `claudeAvailable()` probe before closing; on failure it surfaces the "you may need to restart the app" note (a just-installed CLI often isn't on the already-running process's PATH). The setup-docs link opens in the system browser via `openExternal`, never navigating the app window.

Auth completion itself isn't programmatically tracked (there's no reliable cross-OS signal short of a real API call) — the user drives it in the terminal and clicks Finish, which is gated on the install probe as the source of truth. The embedded terminal's `term-data`/`term-exit` listeners filter by its own `termId`, so they coexist with the git-pane consoles' listeners on the same broadcast channels. Going **Back**, or closing the dialog (×, Esc, or a successful Finish), tears the embedded terminal down (kills the PTY, disposes the term) so no background install/auth shell is left running. `newSession()` also routes through `ensureClaude()`, which re-probes and re-opens the wizard (returning `false` so the caller bails) whenever Claude Code is still absent — so every entry point to opening a session is gated, not just startup.

## Per-session commit

Each session is tracked independently so its work can be committed on its own, **even when two sessions edit the same file** — each commit then contains only the lines that session changed.

**Tracking (record).** Every hook payload carries the `--session-id` we spawned with, so `recordSessionActivity()` can attribute it. Per session it keeps:

- `firstPrompt` — the first `UserPromptSubmit` (set once). Seeds the Haiku-generated session **title**, and is the last-resort fallback for a per-session **commit message** (see below).
- `edits` — a `Map<absPath, op[]>`. Every `PostToolUse` of a file tool becomes a *replayable op* via `editOp()`: `Write` → `{t:'write', content}`, `Edit` → `{t:'edit', old, new, all}`, `MultiEdit` → `{t:'multi', edits}`, `NotebookEdit` → `{t:'opaque'}` (can't be replayed as text). A `.gitignore`d file is **never** recorded here — `recordSessionActivity()` runs `isIgnored()` (`git check-ignore`) on the edited path first and skips it, so a session editing an ignored file (a log, `.env`, a build artifact) never tracks or commits it. This mirrors the filesystem-diff path, which already excludes ignored files because they don't appear in `git status`.
- `fileOps` — a `Map<absPath, 'add'|'delete'>` of path-level changes no text op can express (a binary a `Bash`/MCP tool produced, or a rename/move/delete), filled by diffing the working tree across each tool call. See [Tracking filesystem changes](#tracking-filesystem-changes).

**Commit (replay).** `commit-session(id)` rebuilds, for each touched file, *only this session's version*:

1. `replayEdits(base, ops)` replays the session's ops onto the file's committed (`HEAD`) contents — `base` comes from `git show HEAD:<path>` (empty for a new file). String replacement, in order. It returns `{content, clean}`; `clean` is false if an `old_string` is no longer present (the other session moved/overwrote it) or the op was opaque.
2. If `clean`, the synthesized `content` is committed. If not, that file falls back to its **whole current working-tree contents** (best effort — overlapping same-line edits are inherently ambiguous). This is `commitContent(base, ops, working)` in `edit-ops.js` (pure, unit-tested): replay-or-fall-back, returning the blob to commit. **Empty patches are dropped here**: when the resolved content is byte-identical to the `HEAD` blob (the session edited a file then undid the change, or rewrote identical text) it returns `null` and the file is committed as *nothing* — so the commit and the `Commit N files` count never include diff-less phantom changes. `fileOps` get the same empty-patch filter: an `'add'` whose current bytes already match `HEAD` (compared by `hash-object` blob sha, binary-safe) and a `'delete'` of a path not in `HEAD` are both dropped. Dropped phantom paths are **forgotten from `edits`/`fileOps` unconditionally** (even when no real commit follows) and a `session-meta` is re-pushed, so a button that read `Commit 13 files` corrects to the real count (or `Nothing to commit`) instead of "committing" files git then silently ignores.
   Then the session's `fileOps` are folded in as extra commit entries (skipping any path already covered by an edit): an `'add'` reads the file's **current bytes as a `Buffer`** (so binary stays intact) and an `'delete'` becomes a removal entry. This is what gets binary files, renames/moves, and deletions into the per-session commit alongside the text hunks.

   **Godot sidecars.** Finally, for every entry the commit already holds, `sessionEntries` folds in the resource's Godot sidecar files — `<file>.uid` (every `.gd` script) and `<file>.import` (every imported asset: `.png`, `.glb`, … — the suffix appends to the *full* filename, so `icon.png` → `icon.png.import`). The Godot editor writes these next to the file; the agent's edit tools never touch them, so they live in neither `edits` nor `fileOps` and would otherwise be left uncommitted, breaking the import on checkout. The companion paths come from the pure, unit-tested `companion-files.js` (`companionPaths(rel)`; a sidecar has no sidecar of its own). Each is included only if it exists on disk and its bytes **differ from `HEAD`** (the same blob-hash check binary adds use, so an unchanged sidecar doesn't inflate the commit); on a resource *deletion* an orphaned sidecar still committed at `HEAD` is removed too. Sidecars are recomputed from disk on every commit/diff (never tracked in `edits`/`fileOps`), so there is nothing to forget afterward, and a `pathClaimedByOther` guard keeps a sidecar another live session is editing out of this one.

   **Commit message.** A session's first prompt is conversational ("make the git panel nicer"), not a commit message, so the message is authored from the session's **own diff** instead — the same patch the Diff dialog renders. `sessionCommitMessage(s, id, patch)` feeds that patch to Haiku via `runHaiku()` (`commitMessagePrompt`/`cleanCommitMessage` in the pure, unit-tested `commit-msg.js`, identical phrasing to the main git pane's `generateCommitMessage` in `git.js`). The prompt encodes the two widely-used standards — the [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/) structure (`<type>(scope): <subject>`, body, `BREAKING CHANGE:` footer) and [Chris Beams' seven rules](https://cbea.ms/git-commit/) (imperative ≤50/72-char subject, blank line, a body wrapped at 72 explaining what & why) — so the log reads professionally instead of as one-line stubs. `cleanCommitMessage` strips a stray ``` fence / wrapping quotes and caps the result at 4000 chars so a full subject+body+footer survives. The call is bounded by a 30s timeout (`Promise.race`); on timeout, an empty diff, or any failure it falls back to `fallbackCommitMessage` — the session **title** (`s.name`), then the first prompt's opening line, then `session <id8>`. Because the call adds latency, the work is **snapshotted at click**: the committed paths are removed from `edits`/`fileOps` and a `session-meta` is pushed *before* the await, so a session that keeps running during generation accumulates its new edits as a separate batch that this commit never folds in (the frozen `entries` already hold the click-time blob content). If the commit then fails, the snapshot is restored (newer edits kept ahead) so nothing is silently dropped. The renderer's commit button shows a `Writing commit message…` spinner and blocks re-clicks until it resolves (see [Renderer](#renderer)).
3. `commitBlobs(entries, msg)` commits those entries **without touching the working tree or the real index's other entries**: it seeds a throwaway index (`GIT_INDEX_FILE` in a tmp file) from `HEAD`, `hash-object -w`s each blob entry (`content` may be a string or a `Buffer`, so binary is byte-exact) and `update-index --add`s it — or `update-index --force-remove`s a `delete` entry — into that throwaway index, then `write-tree` → `commit-tree -p HEAD` → `update-ref HEAD`. Finally it syncs the **real** index for *just those paths* (new blobs via `--cacheinfo`, removals via `--force-remove`), so the committed changes read as clean and only the *other* session's edits remain as unstaged changes.

Because step 3 commits a synthesized blob rather than the shared working file, the other session's edits to the same file are never swept in — they stay in the working tree for that session to commit later (replaying onto the now-updated `HEAD`).

On success `commit-session` **forgets the committed paths** from that session's `edits` *and* `fileOps` maps and re-pushes `session-meta` with the shrunk file list (`trackedFiles()` = union of the two). The renderer's per-session commit button shows `Commit N files` while changes remain, and a disabled **`Nothing to commit`** once there are none. Crucially **`N` is the committable count, not the raw tracked count**: `trackedFiles()` includes *empty-patch* files — ones whose net change vs `HEAD` is now nothing (already committed via the main git pane, or edited-then-reverted) — which `commit-session` drops, so counting them would make the button read `Commit 20 files` while the commit only produces 5. The button therefore reads `N` from the session's **diff stat** (`s.diffStat.files`, from `session-diff-stat` → the same `sessionEntries` filter the commit and the Diff badge use), falling back to `trackedFiles().length` only until the first stat arrives. `refreshDiffStat` runs off every `session-meta` and on select/restore and re-renders the button when the fresh count lands, so the label, the Diff badge, and the actual commit always agree. So the button doubles as the commit-state indicator — after a commit it goes disabled until the session edits again, and can be re-used many times across a session. There is no separate persistent "Committed" status label; `#session-commit-msg` only carries the transient *revert* result and commit/revert *errors* for the active session.

**Known ceilings.** Two sessions editing the *exact same lines* → the second replays unclean and falls back to a whole-file commit. `NotebookEdit` is always whole-file. A version of a session's file the user manually staged is overwritten in the index by the synthesized blob. This relies on Claude Code's `PostToolUse` payload exposing `tool_input.old_string`/`new_string`/`content`. For `fileOps`, the executable bit isn't preserved (everything commits as mode `100644`), `.gitignore`d files are never tracked (the text-edit path filters them via `git check-ignore`, the filesystem-diff path because they don't appear in `git status` — the intended behaviour in both), and attribution is per-burst against a global `git status`. That global snapshot can't tell which session wrote a path, so when two sessions run concurrently the filesystem diff is reconciled against **exact text-edit ownership**: a path another session already recorded — via its precise per-session `edits` (authoritative) or, for two fs-only sessions, a `fileOp` it recorded first — is **never** attributed here (`pathClaimedByOther()` in `sessions.js`, applied both when recording the diff and again when building a session's commit/diff entries, so a `fileOp` recorded before the other session's edit landed self-heals). What remains a genuine ceiling is two *different* sessions whose `Bash`/MCP calls both create or move the **same** path with no text-edit signal from either — inherently ambiguous, so it goes to whichever session's burst closes first. Within a single session, parallel tool calls are handled correctly via the in-flight ref-count (see [Tracking filesystem changes](#tracking-filesystem-changes)).

## Per-session revert

The symmetric operation: `revert-session(id)` undoes **only what this session did in the working tree**, leaving another session's edits to the same file intact. It does *not* snapshot — it de-applies the session's own ops.

For each touched file, `inverseEdits(working, ops)` is the inverse of `replayEdits`: it walks the session's ops newest-first and backs each substitution out of the *current* working contents (`Edit` `old→new` is reversed to `new→old`). Because each reversal only rewrites this session's own `new` strings, another session's edits (different text regions) are untouched. The reverted contents are written straight to the working file.

`clean=false` when an op can't be inverted — a full `Write` or opaque op (no stored pre-image), a pure deletion (can't relocate the removed text), or a `new` string that's since been overwritten. Then a hard reset to `HEAD` is only done when **no other live session also edited that file** (`git show HEAD:<path>`, or delete if the file was new this session); if another session shares the file it's **skipped** and reported, since clobbering its work is worse than leaving ours. Successfully reverted files are dropped from the session's `edits` map (skips stay tracked so the user can retry after resolving); a `session-meta` event refreshes the bar's file count.

The session's `fileOps` are undone the same way: a path tracked in `HEAD` is restored with `git checkout HEAD -- <path>` (binary-safe — git writes the committed bytes directly), an `'add'` with no `HEAD` version was new this session so it's `unlink`ed, and a path another live session also touched (`edits` **or** `fileOps`) is **skipped**. Reverted `fileOps` are forgotten too; the `trackedFiles()` count covers both maps.

## Per-session diff

The session bar's **Diff** button shows what a single session changed, scoped the same way commit/revert are — only that session's own hunks vs `HEAD`, never another session's edits to the same files. `session-commit.js` factors the blob-building out of the commit handler into `sessionEntries(s, repoPath)` (the replay-or-fall-back per file, plus the binary/rename/delete `fileOps`, identical to what [Per-session commit](#per-session-commit) commits), so commit and diff agree exactly. `sessionDiff(s, repoPath, withPatch)` then seeds a **throwaway index** from `HEAD` (the same `GIT_INDEX_FILE` trick as `commitBlobs`, so the real index and working tree are untouched), `update-index`es each entry's blob into it, and runs `git diff --cached` against `HEAD`: `--numstat` gives the badge counts (summed by the pure, unit-tested `sumNumstat` in `git-parse.js` — a binary file counts as a changed file with 0 lines), and the full patch (only when `withPatch`) feeds the dialog. An empty entry set is "no change", so the button disables.

Two IPC handlers: `session-diff-stat(id)` returns just `{additions, deletions, files}` for the button badge; `session-diff(id)` adds the `patch` for the dialog. Each stat spawns several git processes and requests arrive in bursts (startup, a tab switch, a git refresh), so `session-diff-stat` runs through a limiter — `createLimiter(3)` from the pure, unit-tested `concurrency.js` — instead of all at once. The renderer refreshes the badge off **every `session-meta`** (debounced per id) — which main pushes for background sessions too, so a session working out of view keeps its `+added/-removed` current — plus per restored session on startup (no push fires for an idle restored session) and on select. Startup and the bulk refreshes (`refreshAllDiffStats()`) **skip archived sessions**: hundreds of restored sessions can sit in the Archived tab, their badge isn't shown there, and computing every one's stat on launch is what used to make the app crawl on open. An archived row keeps its tracked-file-count fallback (`s.files.length`) and gets its real stat only when it's **unarchived** (`setArchived()` calls `refreshDiffStat(id)` on the resume path). A session's stat is computed **against `HEAD`**, so it also goes stale whenever `HEAD` moves underneath it without that session editing anything — a commit from the main git pane, a commit from *another* session, or a file changed on disk — leaving a button stuck on `Commit 2 files` that reads `Nothing to commit` the moment the user opens the diff (which forces a refresh). To close that gap, `refreshGit()` (the single choke point hit after any commit, on the 3s poll, and on window focus) re-validates every **non-archived** session's stat via `refreshAllDiffStats()` (in `sessions.js`; archived sessions are skipped and refreshed only on unarchive) whenever the working-tree state actually changed — gated on a compact `gitStateSignature(r)` fingerprint (branch + ahead/behind + every staged/unstaged/conflicted path-and-status) so the poll doesn't fan out a diff-per-session when nothing moved. Each per-session refresh is the same debounced `refreshDiffStat`, so this and a concurrent `session-meta` coalesce. Clicking **Diff** opens `#session-diff-overlay`, a panel **over the live terminal** (the terminal is never torn down, so the session is one dismiss away); the overlay **darkens + blurs the backdrop** (`rgba(0,0,0,.45)` + `backdrop-filter: blur`) like the approval/confirm dialogs. It closes on its ×, on any `mousedown` outside the panel (a capture-phase document listener, so clicking the git pane / Commit button dismisses it and the user can commit straight away), on Escape, and on switching sessions.

The dialog has two view modes, toggled by a segmented **Unified / Split** control in its head (the choice persists across opens; the held `patch` re-renders without re-fetching): **Unified** is the single-column view via `renderDiffInto()`, and **Split** is a two-column old-|-new side-by-side via `renderDiffSplitInto()` — within a hunk it pairs consecutive deletions and additions row-for-row (del left, add right; an unpaired one leaves the opposite side a blank filler), context lines show on both sides, and file/`@@` headers span full width. Both are factored out of `diff.js` (`fileHeaders` on for the per-file headers a multi-file session patch carries) and render into the dialog's own body rather than the center diff overlay — that overlay hides the terminal, which is exactly what this dialog must not do.

## Hand a merge to Claude

A diverged remote is the one git situation the porcelain buttons deliberately *don't* resolve themselves (a real merge needs judgement and can conflict). Instead the IDE offers to hand it to a fresh Claude session, so a non-expert user never has to drop to a terminal. Three entry points, all funneling through `offerClaudeMerge(title, message, errorText, operation)` in `git-pane.js`:

- **A failed pull** — `git-pull` came back `needsMerge` (the branches diverged, so `--ff-only` refused).
- **A rejected push** — `git-push` came back `needsMerge` (the remote has commits this branch doesn't; the push bounced). Other push failures still show the plain git-error dialog.
- **The Conflicts section's "Let Claude resolve" button** — for conflict markers already sitting in the working tree (`#conflicts-resolve`, in the section header).

`offerClaudeMerge` first `confirmDialog`s (so it's opt-in, never automatic), then calls `newSessionWithPrompt(prompt)` (exported from `sessions.js`). The prompt is **one generic instruction** — pull and push the current branch, resolve any conflicts, run the tests, don't break a feature (`git.mergePrompt`) — followed by **which operation the user was running** (`git.mergePromptOp` + a per-op phrase: `git.opPull`/`git.opPush`/`git.opResolve`, passed as the `operation` arg by each call site) and the **exact git error** (`git.mergePromptError` + `r.stderr`), so the agent knows both what was being attempted and the real failure text instead of re-deriving either. One prompt covers all three cases; the dialog's title/message and the operation phrase differ per entry point. Both the operation line and the error line are omitted when their value is absent (the conflicts entry passes no `errorText`). `newSessionWithPrompt` spawns a normal session and queues the text, typing it in once the session emits its first PTY output (Claude's input box isn't ready the instant the PTY spawns), then **submits it** with a separate Enter write a beat later — bundling the `\r` with the text submits before the TUI finishes ingesting the multi-line paste, so the Enter is deferred and sent on its own. All strings are i18n'd (`git.*` keys); the divergence detection itself is the unit-tested `pullNeedsMerge`/`pushNeedsMerge` in `git-parse.js`.

## Create a repository

When the open folder **isn't a git repo**, the git pane swaps its whole body for a create-repository panel (`#git-create-repo`) instead of the empty stage/commit lists. `refreshGit()` gates on `git-is-repo` *before* any porcelain call (they'd all fail with "not a git repository"): a `false` shows the panel via `showCreatePanel(repoPath)` (which also hides the Changes/History tabs and the branch/sync header buttons, since none apply yet) and returns; a `true` calls `hideCreatePanel()` and restores the active tab. The panel has a **name** field (prefilled with the folder's basename), an optional **description**, a **Private/Public** segmented toggle (default Private, reusing the session-diff `Unified/Split` control's look), and a **Create repository** button.

Clicking it calls `create-repo` (see the [Git](#the-main-process--three-subsystems) subsystem) — `git init` on `main`, the initial commit, then `gh repo create … --push`. On success it just `refreshGit()`s, which now sees a real repo and renders the normal pane (the remote, the pushed commit, a clean tree). On failure the panel's status line shows the step's error (e.g. the "gh not installed / not signed in" hint) and the button re-enables. UI is themed entirely off the CSS-variable palette (`--accent`, `--bg`, `--border`, …) and every string is i18n'd (`git.create.*`). The name rules + `gh` argv live in the pure, unit-tested `repo-create.js`.

## Native recent-projects menus

Right-clicking the app's **taskbar icon (Windows)** or **Dock icon (macOS)** shows a list of recently opened projects — the same `recentFolders` the in-app Open-folder reverse-combobox uses, kept in sync. `repo.js` calls `refreshNativeRecent(recentFolders, openRecentInPlace)` once at load and again from `setRepoPath`/the stale-entry pruning/`removeRecentFolder`, so the native menu always matches the persisted list. Each recent row in the in-app combobox carries a trash button that arms on the first click and, on the second, DOM-removes the row and calls the `remove-recent-folder` IPC (`removeRecentFolder` in `repo.js` → the Electron-free `removeRecent` in `recent-folders.js`) — same two-click arm pattern (`shared/arm-hint.js`) as the session-delete button. The menu structure is built by the Electron-free, unit-tested `native-recent-menu.js` (`jumpListCategories`/`dockMenuItems`/`folderLabel`); `native-recent.js` does the Electron wiring.

The two platforms differ because of how each OS hands the click back:

- **Windows — Jump List** (`app.setJumpList`): a custom **Recent** category of `task` items, each relaunching `<exe> --folder="<path>"`. The taskbar can only *launch a program*, not message a running window, and this app is deliberately multi-instance (no single-instance lock — see [instance.js](#the-main-process--three-subsystems)), so a pick opens the project in a **new window** via the existing `--folder` CLI override (`cli-args.js`'s `parseFolderArg`). No callback is needed.
- **macOS — Dock menu** (`app.dock.setMenu`): real `MenuItem`s whose click runs `openRecentInPlace(folder)` in main — it `switchToFolder`s (the same validate-and-repoint path `open-folder-path` uses) and pushes a `folder-changed` event to the renderer, which reloads exactly like the in-app recent menu (`applyRepoChange`). So a Dock pick switches the **running** window's folder in place.

Both native APIs require a ready app, so `native-recent.js` ready-gates: a pre-`ready` refresh (the load-time call) is stashed and replayed on `app.whenReady`. The relaunch works in dev too: when `app.isPackaged` is false the exe is the bare Electron binary, so `native-recent.js` prepends the app directory (`app.getAppPath()`, quoted) as the Jump List item's `extraArgs` — otherwise `electron.exe --folder=…` has no app to run. Packaged, the exe is the app itself, so `extraArgs` is empty.

## Git stash

Setting the whole working tree aside (and bringing it back) rounds out the porcelain buttons, for the common "I need to switch context but don't want to commit half-done work" case. It's modelled as a **section in the Changes view** (`git-stashes-title`) alongside Staged/Changes/Conflicts — that, like the Conflicts section, **renders only when there's something to show** (`stashes.length`). Creation is a single **Stash** button (archive icon) in the Changes header beside stage-all/discard-all: `git stash push --include-untracked` (so new files are saved too). The renderer reads git's "No local changes to save" out of the result to report *Nothing to stash* rather than a misleading success.

Each stash row reuses the commit-row layout (a message line over a muted `ref · relDate` meta line) and three hover actions: **Apply** (`stash apply`, keep it), **Pop** (`stash pop`, apply then remove), and **Drop** (`stash drop`) — Drop being destructive uses the same two-click armed rotate-ccw as the discard buttons (`armHint.dropStash`). Clicking the row opens the stash's patch in the center diff overlay (`git-stash-show -p` → `openStash` → `showStash`, the same multi-file `renderDiff` path commits use). Each op funnels through `runStash()` which surfaces a git error via the shared dialog and then `refreshGit()`s — and `refreshGit()` always calls `refreshStashes()` (an independent `git-stash-list`) so the section, the file lists, and the per-session counts re-render together; it's unconditional because a `drop` changes the stash list without touching the working tree (so the `gitStateSignature` gate would miss it). The selector (`stash@{N}`, from `%gd`) is what targets a specific stash; the message (`%s`) and date (`%cr`) come from `git stash list`, parsed by the Electron-free, unit-tested `parseStashList` in `git-parse.js`. All labels/tooltips are i18n'd (`git.stash*` / `armHint.dropStash`).

## Tracking filesystem changes

The text-edit tools (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) are the only ones whose `PostToolUse` payload tells us *what* changed in a file. Everything else a session does to the filesystem — a binary a `Bash` command produced (a build artifact, a downloaded asset, an image an MCP tool generated), a `mv`/`git mv` rename, a file moved in or out of the repo, an `rm` — leaves no replayable op. Those are caught instead by **diffing the git working tree around the tool call** (`sessions.js`):

- On **`PreToolUse`** for any file-touching tool — `tracksFs(payload)` (pure, in `fs-track.js` with `TEXT_EDIT_TOOLS`/`READONLY_TOOLS`, unit-tested): not a text-edit tool and not in the read-only set (`Read`/`Grep`/`Glob`/…, and the subagent spawners `Task`/`Agent` — see *Subagents* below) — `recordSessionActivity()` snapshots `git status --porcelain=v1 --untracked-files=all --no-renames` into `s.preStatus` as a `Map<relPath, "XY">`. `--no-renames` makes a rename surface as a separate delete + add, so each path is attributed on its own.
- On **`PostToolUse`** for the same tool, it snapshots again and `applyFsDiff()` compares: a path whose status changed and that **now exists on disk** is recorded `fileOps[abs] = 'add'`; one that's **gone** is recorded `'delete'` — unless it was only ever untracked (never reached `HEAD`, so there's nothing to commit as a deletion), in which case it's simply forgotten. A path already covered by an `edits` op is left to that machinery; a path an edit tracked but that a later tool then moved/removed is dropped from `edits` (its text ops are now void) and recorded as a `fileOp`. A path **another** session already owns is skipped entirely (`pathClaimedByOther()`), since the global `git status` diff would otherwise pull a concurrent session's edit into this session — see the per-session commit [*Known ceilings*](#per-session-commit).

**Bulk-VCS commands are excluded.** A `Bash` command that only *moves git state* — `git pull`/`merge`/`rebase`/`reset`/`stash`/`cherry-pick`/`revert`/`clone`/`switch`/branch-`checkout` — changes the working tree wholesale, but those files are not the session's authored work; attributing them would inflate the per-session commit with dozens of files the agent never wrote (the bogus "Commit 20 files"). This is especially live because the IDE itself spawns sessions to resolve merges (see [Hand a merge to Claude](#hand-a-merge-to-claude)), so an un-skipped `git pull` was the main offender. `tracksFs(payload)` therefore returns false for such a Bash command — classified by the pure, unit-tested `isBulkVcsCommand()` in `fs-track.js` (`test/fs-track.test.js`), which token-scans the command (handling compound `&&` chains, global options like `-C <path>`, and a `--` pathspec separator that marks `git checkout -- file` as a real file restore, not a branch switch). `git mv`/`git rm`/`git add` are **not** excluded — they're genuine path-level edits the tracker must catch.

The snapshot must be taken **before** the tool runs, so the hook server `await`s `recordSessionActivity()` *before* answering the hook — a command hook blocks its tool until the `curl` returns, so awaiting the `PreToolUse` snapshot guarantees the "before" state predates the tool's writes. This adds one `git status` of latency to each file-touching tool (none to read-only tools or non-tool events), which is negligible for an interactive session. Because the diff is scoped to `git status`, `.gitignore`d paths are invisible (correctly never committed).

**Subagents are tracked through their own hooks.** A subagent's tool calls fire the same hooks with the **parent `session_id`** plus an `agent_id` (Claude Code sets `agent_id` only inside a subagent's context), so its text edits land in `edits` and its fs tools ref-count and diff exactly like main-thread ones — no special casing, and nothing to merge afterwards. Two rules keep that safe: (1) the wrapping spawner tool — named `Task` historically, `Agent` since CLI 2.1.63 — is in `READONLY_TOOLS`, because on recent CLIs subagents run in the background and the wrapper's `PostToolUse` can fire at spawn while the subagent keeps working; fs-tracking the wrapper would pin `fsInFlight` for the subagent's whole run (and a stuck count suppresses every later snapshot until the next prompt). (2) The `UserPromptSubmit` reset of `fsInFlight`/`preStatus` (and the first-prompt capture that names the session) applies **only to main-thread prompts** (`!payload.agent_id`) — a subagent's `UserPromptSubmit` arrives while main-thread fs tools are mid-flight, and letting it wipe the pending snapshot silently lost those tools' changes. `NotebookEdit` names its target `notebook_path`, not `file_path` — `editedFilePath()` (`fs-track.js`) reads both. A subagent spawned with worktree isolation edits under its own worktree; those paths aren't in the main working tree, so the tracker correctly ignores that work until it's merged back.

**Parallel tool calls.** Claude runs tool calls in parallel within a turn, so the `PreToolUse`/`PostToolUse` hooks of several fs tools interleave — a single `s.preStatus` slot would let a second `Pre` clobber the first, and a `Post` arriving with no live snapshot would drop its tool's changes entirely (the "session changed files but the button says nothing to commit" bug). Instead `recordSessionActivity()` **ref-counts the fs tools in flight** (`s.fsInFlight`): it snapshots when the count goes 0→1 and runs `applyFsDiff()` once it returns to 0, diffing every concurrent tool's changes against that one consistent baseline exactly once. The attribution window is therefore *a burst of concurrent tool calls*, not a single call — see the per-commit *Known ceilings* for the cross-session concurrency caveat.

## Session persistence

Sessions survive closing the app: each is stored as **its own file** under `sessions/<id>.json` in the shared data dir (`instance.js`'s `sharedDataDir`, not the disposable per-instance profile), so a single session's change rewrites only that one small file and the whole set is never serialized at once. `serializeSession()` (pure, in `session-persist.js`) snapshots a record to JSON — dropping runtime-only fields (the PTY, the transient `preStatus`) and flattening the `edits`/`fileOps` Maps to arrays; `deserializeSession()` rebuilds it on load with `pty: null` and `suspended: true` (a restored session has no live process until resumed) but its tracked-file state intact, so it's committable immediately.

**Only sessions with something to restore are kept.** `isSessionPersistable(s)` (pure, in `session-persist.js`) gates every write and every load: a session is persistable only once it carries a conversation (`firstPrompt`, set on its first prompt) or tracked work (`edits`/`fileOps`). An **empty husk** — created but never prompted — has no transcript on disk, so resuming it would run `claude --resume <id>` against a conversation that was never created and the CLI prints *"No conversation found with session ID: &lt;id&gt;"*. `new-session` still persists on create, but that write is a no-op until the first prompt; `loadPersistedSessions()` drops (and `removeSessionFile()`s) any non-persistable snapshot so an already-saved husk doesn't resurrect every launch; and the PTY `onExit` deletes the file for a non-persistable session that just exited (the failed-`--resume` case). This is why an unnamed never-used session — even one you archived — no longer reappears with the "No conversation found" error after a restart. Each file also carries a `_seq` — a monotonic creation index — because per-file storage loses the Map's insertion order across a restart; load reads every `*.json`, sorts by `_seq`, and rebuilds the Map oldest-first (and bumps the live counter past the max seen). A file that **can't be parsed** (empty/truncated — e.g. a write interrupted by a crash or kill) is unrecoverable: load reports it once and then **deletes** it, so the *"Unexpected end of JSON input"* error can't resurface on every launch; leftover `*.json.tmp` files (an interrupted atomic write) are cleaned up on load too. An existing single-file `sessions.json` store is **migrated** into per-session files on first load (each entry gets a fresh `_seq`), then the legacy file is deleted.

**Every change syncs.** A save is triggered after each mutation: discrete user/state actions — creating a session, archiving (`suspend-session`) and dearchiving (`resume-session`), a `setSessionState()` dot change, a per-session commit (marks `pushed`, and clears the committed edits) or revert (clears the reverted edits), and the Haiku-generated name — call `persistSession(id)` for an **immediate** single-file write; the high-frequency edit stream (`recordSessionActivity()` on every `PostToolUse`) is the one path that **debounces** (`schedulePersist(id)`, ~1s per session) so a burst of edits coalesces into one write. Every write goes through `writeSessionFileAtomic()` (write to a sibling `<id>.json.tmp`, then `renameSync` over the real file) so a crash or kill mid-write leaves the old complete file intact rather than an empty/half-written one that fails to parse next launch. Killing a session (`kill-session`) and eviction delete its file (`removeSessionFile()`); on quit `persistSessions()` writes the whole set synchronously as a backstop. The on-disk set is capped at **100 MB** (`MAX_PERSIST_BYTES`): after each write `evictOverBudget()` drops the oldest *evictable* sessions (`enforceLimit()`, never one with a live PTY) from memory, disk, and the UI (`session-evicted`) until it fits. `serializeSession`/`deserializeSession`/`isSessionPersistable`/`sessionBytes`/`enforceLimit`/`persistedState` are all Electron-free and unit-tested (`test/session-persist.test.js`); the dot-state collapse on persist is covered in [status-detection.md](status-detection.md#persistence-across-restarts).

## Renderer

The renderer is split across `renderer/` modules (`sessions.js`, `git-pane.js`, …); this section covers the session bar + git pane. It keeps a `Map` of sessions, each owning its own xterm.js `Terminal` in a hidden container div. Switching sessions just toggles which container is visible, preserving scrollback. `setState()` sets the status-dot CSS class. Clipboard handling for every terminal (session and git-pane console alike) is the shared `attachClipboard()` in `renderer/shared/terminal.js`: Ctrl+C copies the selection (passing SIGINT through to the PTY only when nothing is selected — key matched case-insensitively so it still fires with Shift held), Ctrl+V / right-click paste, and right-click copies when a selection exists. Copies go through `copyToClipboard()` → main's `clipboard-write` handler (`clipboard.writeText`), because the renderer's async Clipboard API is unreliable under `file://` (not a secure context, so `navigator.clipboard` can be `undefined` and throw synchronously — which is why Ctrl+C silently did nothing) and otherwise needs document focus the canvas can lack right after a selection; a hidden-textarea `execCommand('copy')` remains only as a last resort if the IPC bridge is absent. Text paste likewise reads through main's `clipboard-read` handler. All three clipboard handlers (`clipboard-read`, `clipboard-write`, `paste-image`) retry through the pure `clipboard-lib.js` `withClipboardRetry()`: on Windows the clipboard is a single lock another process can briefly hold, so a bare `readText()`/`writeText()`/`readImage()` intermittently throws "OpenClipboard failed"; without the retry that throw **rejected the IPC call and the renderer's `paste()` swallowed the rejection, silently dropping the paste** (the intermittent "can't paste, time to time" bug). The retry returns a fallback (`''`/`false`/`{ok:false}`) after a few short-delayed attempts instead of rejecting, and `paste()` itself is wrapped in try/catch as a final guard so a stubborn lock degrades to a harmless no-op. Paste checks the clipboard for an **image** before falling back to text: a PTY can't carry image bytes, so main's `paste-image` handler (`clipboard.readImage()`) spills the bitmap to a temp PNG under `os.tmpdir()/claude-ide-pastes` and returns its path, which the renderer pastes formatted by `opts.formatImagePath` — sessions emit an `@<path>` mention (so Claude reads the image), console shells get the raw path quoted-if-spaced. Because the Claude CLI turns on terminal mouse reporting, a plain mouse drag is forwarded to the app instead of selecting; **hold Shift while dragging** to force a local selection (xterm's built-in override). The flip side is that the Claude CLI does its *own* drag-to-select and copies via the **OSC 52** clipboard escape sequence (it reports "auto copy succeeded") rather than a key event — xterm has no built-in OSC 52 handler, so those copies used to vanish before reaching the OS clipboard. `attachClipboard()` registers `term.parser.registerOscHandler(52, …)`, decoding the base64 payload through the pure, unit-tested `renderer/shared/osc52.js` `decodeOsc52()` (UTF-8 aware; declines a read "?" / empty / malformed payload) and routing the text through the same `copyToClipboard()` path as Ctrl+C, so the CLI's own copy lands. So both a local Shift-drag/double-click selection (Ctrl+C / right-click) and the CLI's mouse-drag-copy (OSC 52) now reach the clipboard. The git pane re-fetches on a 3s focus-gated poll, on **window focus** (so changes made while the app was in the background show without waiting for the next tick), on every **`session-meta`** event (debounced — a session just touched the working tree, so the Changes list refreshes within a beat instead of up to 3s later), after stage/unstage, and after open-folder (there is no manual refresh button — `git-fetch`/`git-pull` and the branch selector cover the remaining needs). A **background `git-fetch` fires automatically** — once on startup, on opening another folder, and on switching branches — through `autoFetch()` (git-pane.js): it `git-fetch`es and, only on success, re-`refreshGit()`s so a freshly discovered ahead/behind count shows without the user reaching for Sync. It's a convenience, so failures (no remote, offline, a non-git folder) are swallowed and never surface a dialog. A top `#session-bar` shows the active session's name on the left and, on the right, a two-click **Revert** button (`revert-session` — de-applies just this session's edits, see [Per-session revert](#per-session-revert)) a commit button that calls `commit-session` for that session only, and a **Diff** button carrying a green `+added`/red `-removed` badge that opens a floating diff dialog over the live terminal (see [Per-session diff](#per-session-diff)); `updateSessionBar()` refreshes them on select (also disarming Revert) and on each `session-meta` event. The commit button shows `Commit N files` from the session's **committable** count (`s.diffStat.files` — the same filtered count the commit and the Diff badge use, not the raw tracked-file count, so it never overcounts empty-patch files), or a disabled `Nothing to commit` when that count is zero (see [Per-session commit](#per-session-commit)). While a commit is in flight, the button shows a disabled `Writing commit message…` spinner (the per-session `s.committing` flag, so only *this* session's button blocks — the rest of the app stays live) until the Haiku message call resolves and the commit lands. The Diff button disables when the session changed nothing. The name prefers the Haiku-generated title from `session-name` (which also updates the sidebar row label), falling back to the first prompt, then `session <id8>`. Clicking a git row calls `openFile()`, which routes by extension: images/audio open `#asset-view` (see Asset viewer), everything else opens `#diff-view` over the center pane — `renderDiff()` parses git's unified diff into coloured rows with old/new line-number gutters. Both overlays share `closeOverlay()`; selecting a session also hides them. Each row's status badge carries a tooltip from `statusLabel()` (`renderer/shared/git-status.js`, unit-tested) that spells out which column it's in, because the same porcelain letter reads differently across the staged/unstaged split — most confusingly `D`: a **staged deletion** vs a **file deleted on disk but not staged**. Unstaging a staged deletion does *not* clear it — `git reset` restores the file's index entry but leaves it gone on disk, so git re-reports it as an unstaged deletion and the same red `D` just hops to the Changes list; the tooltip points an unstaged deletion at Discard, which is what actually restores the file. Each row also has a two-click discard (rotate-ccw icon) button left of stage/unstage — the first click arms it red, the second calls `git-revert`. Every two-click "arm then confirm" button across the app (this discard, discard-all, the History tab's revert, the session-bar Revert, the archived-session permanent **Delete**, and the inline browser's **Terminate**) also pops a small speech-bubble callout next to the button on the first click prompting the second — the shared `renderer/shared/arm-hint.js` (`showArmHint(anchor, message?)` / `hideArmHint()`, styled by `styles/arm-hint.css` with the same popover palette as the branch menu) surfaces the otherwise-hidden two-click contract without the user having to discover the tooltip on hover. The bubble shows a short generic "Click again to approve" (`armHint.clickAgain`, two lines) by default; the per-button `armHint.*` keys stay as the buttons' detailed `title` tooltips. It auto-fades, retargets when another button arms, tracks the button on scroll, and is dismissed when the action fires or the button disarms (e.g. a git refresh rebuilds the rows). (The archived-session Delete previously deleted on a single click — it now arms first, matching the rest.) The "Changes" header carries matching stage-all (`+`) and discard-all (same rotate-ccw icon, same two-click arm) buttons that loop those calls over every unstaged file. Stage/unstage/discard now surface a failed result through the shared git-error dialog instead of silently no-op'ing. When `gitStatus` returns conflicts, a **Conflicts** section (`#conflicts-section`, hidden when empty) renders above Staged: each row opens the file's diff and carries a ✓ button that `git add`s it once the markers are resolved; it moves out of the section on the next refresh. The section header also carries a **"Let Claude resolve"** button (`#conflicts-resolve`) that hands the whole resolution to a fresh Claude session (see [Hand a merge to Claude](#hand-a-merge-to-claude)). The commit box has **Commit** (calls `git-commit`: commits the staged changes, but if nothing is staged it first runs `git add -A` so a bare click behaves like "commit all"; clicking with an **empty** message box has main author one from the staged diff via Haiku — the button disables and shows `Writing commit message…` during the call, then reports `Committed: <subject>`), **Undo** (calls `git-undo` → `git reset --soft HEAD~1`, keeping the changes staged — but only for **unpushed** commits: if HEAD is already on the upstream, `git-undo` refuses with "Last commit is already pushed — revert it instead of undoing" rather than dropping a commit the remote still has, mirroring the History tab's undo-vs-revert split), and **Push** (calls `git-push`, then re-refreshes; a push the remote rejected because it diverged comes back `needsMerge` and offers a Claude session instead of the error dialog — see [Hand a merge to Claude](#hand-a-merge-to-claude)). The Push button carries a green pill **ahead badge** pinned to its top-right corner (`#git-ahead`) showing the number of unpushed commits — `gitStatus` returns this as `ahead` (`git rev-list --count @{u}..HEAD`, `0` when there is no upstream or no commits yet), and `refreshGit()` hides the badge when the count is zero. The **Pull** button (pane header) mirrors this with its own green **behind badge** (`#git-behind`) showing how many commits a pull would bring in — `gitStatus` returns this as `behind` (`git rev-list --count HEAD..@{u}`, reflecting the last fetch's view of the remote, so it populates after a fetch/poll); while pulling, the button shows the same sweeping progress bar as Push (`.loading`) and disables; a pull that can't fast-forward comes back `needsMerge` and offers a Claude session (see [Hand a merge to Claude](#hand-a-merge-to-claude)). Toolbar/action icons are inline Lucide SVGs chosen to read distinctly: discard = `rotate-ccw` (single arrow), undo = `undo-2` (hook arrow), branch = `git-branch`.

**GPU terminal rendering.** xterm's default renderer is the DOM (one node per cell) — the slowest path. Every terminal (session, console, the setup wizard) instead gets a GPU renderer through `attachRenderer(term)` in `renderer/shared/terminal.js`: it loads the **WebGL** addon, registering `onContextLoss` to dispose and fall back to the **Canvas** addon (GPU-composited 2D, no WebGL context consumed) if the context is lost — a driver reset, GPU-process recycle, or the window being backgrounded — and uses Canvas from the start if WebGL is unavailable. Because Chromium caps live WebGL contexts at ~16 and this app can hold many terminals at once, the renderer is attached **only to the on-screen terminal** and disposed when it's hidden: `selectSession`/`selectConsole` attach to the newly visible terminal and dispose every other's (storing the handle as `s.renderer`/`cc.renderer`), and the suspend/close/teardown paths dispose before `term.dispose()`. So live WebGL contexts stay at roughly *visible session + visible console + setup terminal* — far under the cap. The renderer addons only paint cells; input, selection, OSC 52, and clipboard all live on the `Terminal` itself, so this is transparent to `attachClipboard()`. The GPU rasterization for web-content compositing is enabled separately by main-process switches (`enable-gpu-rasterization`/`enable-zero-copy`, see [platform-notes.md](platform-notes.md)), and the 3D asset viewer requests the discrete GPU via `powerPreference: 'high-performance'`.

**Archived-session search.** The session list's filter tabs (Active/Archived/All) gain a `#session-search` box that is shown **only on the Archived tab** (archived sessions accumulate over time, so they're the set worth filtering). Typing filters the visible archived rows live (debounced 150ms, like the explorer's file search) by matching each session's **title + first prompt** against the query. The matching is the pure, unit-tested `renderer/shared/name-match.js` (`compileQuery`/`matchesTerms`), which **mirrors the explorer's filename search** (main's `search-names`): the query is folded (NFC + case-insensitive, via the shared `text-fold.js`) and split on whitespace into terms that **all** must match, each a `.ext` suffix or a plain substring. The filter is folded into `sessionVisible()` (so `applyTabFilter()` applies it) and only ever engages on the Archived tab; leaving that tab clears the box and the filter. Selection is left untouched while typing — the filter is transient and non-destructive.

**Branch selector.** The git pane head carries a `#git-branch` button showing the current branch name (a `git-branch` icon + `#git-branch-name`, refreshed from `gitStatus().branch` on every `refreshGit()`). Clicking it toggles `#branch-menu`, a popover that is **`position: fixed`** (not absolute) so the git pane's `overflow:hidden` can't clip it — `positionBranchMenu()` aligns its right edge just inside the git pane's right edge (below the button) from the live rects, so it grows leftward into the wider terminal area rather than being cut off at the pane border or running off the window. It's re-aligned on window resize while open. Branch names, especially remote ones, run long, so the popover and its search box are sized generously (`width: 380px`, capped to the viewport). Opening it fetches `git-branches` fresh (so new branches appear), which now returns **both local (`refs/heads`) and remote-tracking (`refs/remotes`) branches** so the search covers everything: the pure, unit-tested `parseBranches()` (`git-parse.js`) lists locals most-recent-first, then appends each remote whose name has no local counterpart (deduping a branch carried on two remotes, dropping the `origin/HEAD` pointer), tagging each entry `{ name, remote }`. The for-each-ref `--sort=-committerdate` gives a latest-*edited* order; `orderBranchesByUsage()` then re-ranks by latest-*used* — the order branches were last checked out, parsed from the HEAD reflog (`git reflog --pretty=%gs`, matching `checkout: moving from … to <b>`) — so with an empty search a branch you switch to often but rarely commit on (e.g. `main`) still sits near the top; branches never checked out (or aged out of the reflog) keep their committerdate order behind the used ones. Remote-only rows read muted with a `☁` prefix (`.branch-item.remote`). The current local branch is marked, and the search box substring-filters the combined list live (matching the full `origin/…` name). The list scrolls past ~15 rows (`max-height`), so a repo with many branches stays usable. Clicking a row calls `git-checkout` and re-refreshes (errors surface through the shared git-error dialog); a remote-only row is checked out by its **short name** (`origin/feature/x` → `feature/x`) so git's DWIM creates a local tracking branch instead of detaching HEAD. The popover dismisses on outside click or Escape. Each non-current **local** row also carries a **delete** (trash) button (`.branch-del`) that reveals on row hover (remote-only rows have none — they aren't ours to delete locally): it's the same two-click arm-then-confirm as the discard/stash-drop buttons — first click arms it red and pops the `armHint.deleteBranch` callout, second click calls `git-delete-branch` (`git branch -D <name>`, force-delete so the two-click approval — not git's merged-only `-d` check — is the guard). The checked-out branch gets no delete button (git refuses to delete it). After a delete the list is re-fetched and re-rendered **without closing the popover**, so several branches can be removed in a row; errors surface through the git-error dialog. The search box doubles as a **create** field: whenever the typed name isn't already an exact branch, a `+`-prefixed "Create branch" row (`.branch-create`) appears at the top of the list — clicking it, or pressing Enter on a non-matching name, calls `git-create-branch` (`git checkout -b <name>` off the current HEAD) and re-refreshes; pressing Enter on a name that matches an existing branch switches to it instead. Git branch names can't contain spaces, so the typed name is first run through `normalizeBranchName()` (`renderer/shared/git-status.js`, unit-tested) which turns each run of whitespace into a single hyphen and trims the ends; the "Create branch" row shows the normalized name so the user sees exactly what will be created (e.g. `fix login bug` → `fix-login-bug`).

The git pane has two tabs (`#git-tabs`, styled like the session filter tabs): **Changes** (the staged/unstaged lists + commit box, all wrapped in `#git-changes-view`) and **History** (`#git-history-view`). Switching to History calls `refreshHistory()` → `git-log` (which returns both the local `commits` and the not-yet-pulled `incoming` commits), caching them in `allCommits` / `incomingCommits` and rendering one row per commit (subject + `short · author · relDate` meta line) via `renderHistory()`, which lists the **incoming** commits above the local log so a pull can be previewed before syncing. A `#history-search` box atop the list filters those cached commits live (no re-fetch): every whitespace-split term must appear in a commit's subject, author, or hash (full or short), case-insensitively — the same rule as the unit-tested `filterCommits()` in `git-parse.js`, replicated in the renderer since it can't `require` main. Clicking a row opens that commit's full patch in the diff overlay via `openCommit()` → `showCommit()` (`renderDiff` with its `fileHeaders` flag on, so each `diff --git` emits a file-name header row and re-derives the highlight language from that path). A commit's direction drives both its look and its action, marked by a **leading direction icon** (`.commit-dir`) left of the subject, colour-matched to a left stripe; a fixed-width slot is reserved on every row (empty for pushed commits) so subjects stay aligned. An **incoming** commit (from `git-log`'s `incoming` list — `git log HEAD..@{u}`, tagged by the pure `markIncoming()`) gets an accent stripe + Lucide **download** icon and has **no** action button — it isn't in local history yet, so there's nothing to undo/revert (clicking the row still previews its diff, since the fetched objects are already local). For the local commits, the `pushed` flag (from `markPushed()`) decides: an **unpushed** commit gets a green stripe + Lucide **upload** icon (matching the green Push ahead badge — the same set of commits) and its button **undoes** it (`git-undo-commit`, a history rewrite), while a **pushed** commit has no icon/stripe and its button **reverts** it (`git-revert-commit`, a new undo commit on top — pushed history can't be safely rewritten). The icons carry the `git.unpushedTitle` / `git.incomingTitle` tooltips. For the two actionable cases it's the same two-click rotate-ccw arm-then-confirm as the discard buttons (arm message `armHint.undoCommit` vs `armHint.revertCommit`); on failure it surfaces the shared git-error dialog (retitled per call via `#git-error-title`). `refreshHistory()` also runs after a commit/undo, and after a successful background fetch (`autoFetch`) or **Sync** while the History view is open — so the incoming preview appears once a fetch surfaces it and clears once a pull lands it; it is **not** on the 3s poll (which only refreshes the Changes lists).

Its background is a pure-CSS pattern (`#empty-hint::before`): a faint dot field — the app's own status-dot vocabulary scattered as texture — under a soft `--accent` glow, masked to fade at the edges; it's theme-aware via the palette vars (`color-mix`), no image asset. The card holds a three-dot wordmark (a cluster of `--green`/`--yellow`/`--purple` dots — the product is *many parallel sessions*), a tagline, a **Start** action list, and a **Session status** legend mapping each dot colour to its meaning. ## File explorer

The left sidebar is split 50/50 by default: the sessions list on top, a file tree (`#file-tree`) below; a drag-gutter between them resizes the split (see [Resizable panes](#resizable-panes)). The tree is **lazy** — `loadDir(rel, container, depth)` fetches one directory level over `list-dir` (main reads `repoPath/<rel>` and returns `{name, dir}` entries, folders first then alphabetical, `.git` hidden) and only loads a folder's children the first time it's expanded (▸/▾ twisty). Each row carries a **file-type icon** between the twisty and the name: `loadDir` builds it with `fileIcon(name)`/`folderIcon()` (`shared/file-icon-img.js`), which renders an `<img>` (loaded under the page's `img-src 'self'` CSP, like every other app-local asset) for one of the SVGs in `src/renderer/file-icons/`, chosen by the pure `iconForFile()`/`iconForFolder()` mapping (`shared/file-icons.js`); a folder's icon flips open↔closed with its twisty (and resets on **Collapse all**). The icons are **real artwork fetched from the internet, not hand-drawn** — [Devicon](https://devicon.dev) brand logos for programming languages (kept in their own colours) and [Lucide](https://lucide.dev) glyphs (recolored to read on the dark tree) for data/asset/config families and folders; `iconForFile()` matches a few exact filenames first (`Dockerfile`→docker, `package.json`→nodejs, `.gitignore`→git, …), then the extension, then falls back to a generic file glyph. The same `fileIcon()` renders on **search-result** rows. Clicking a row selects it (`select(row, rel, dir)` records the last-clicked item in `selected`, for files and folders alike); clicking a file also calls `openFromTree()`, which reuses the git pane's viewers: images/audio → `showAsset()`, everything else → `showFile()` — the **file editor** (see [File editor](#file-editor)), which fetches the file's text over `read-text` and mounts an editable view in the `#diff-view` container (binary files, sniffed by a NUL byte, show `(binary file)`). The tree refreshes on startup, on **Open folder** (button at the pane's bottom), after any in-app file op, and **automatically when the repo changes on disk** — there's no manual refresh button. `explorer.js` keeps one recursive `fs.watch` over `repoPath` (re-pointed on **Open folder** via `onRepoChange`); each event whose top path segment is a skipped dir (`shouldSkipDir`: `node_modules`/`.git`/build output) is dropped, and the rest are debounced (250ms, trailing-edge so the last event in a burst always fires) into a single `tree-changed` push to the renderer, which calls `refreshTree()`. `refreshTree()` **preserves expansion**: before rebuilding it snapshots which folders are open (rows whose twisty is `▾`, keyed by the `data-rel` it stamps on every row), reloads the root, and re-expands those paths recursively, then re-applies the selection highlight — so a refresh (whether triggered by an on-disk change or by creating/renaming/deleting a file) leaves the tree looking unchanged apart from the edit, and (since it only re-fetches expanded folders) stays cheap. The header icons: **+** (`#files-new`) prompts for a filename (with extension, e.g. `run.py`) through the shared `promptText` modal (`renderer/shared/prompt.js`), then creates the file over `create-file` (main joins it under `repoPath`, refuses paths escaping the root, `mkdir -p`s the parent, writes with the `wx` flag so it fails if the file already exists, and re-prompts with the error otherwise). A **folder+** (`#files-new-folder`) icon sits beside it: it prompts for a folder name and creates it over `create-folder` (same path guards; refuses to clobber an existing entry; `mkdir -p`) via the parallel `createFolderIn(dir)` loop, then refreshes the tree. Next comes a `chevrons-up` **Collapse all** button (`collapseAll`, which just flips every open `▾` twisty back to `▸`). The prompt-and-create loop is `createFileIn(dir)` (re-prompts on error, opens the file on success); `dirOf(rel, dir)` computes a tree item's destination dir — inside it when it's a folder, alongside it (its parent) when it's a file — and `targetDir()` applies that to the toolbar button's `selected`, falling back to the repo root when nothing is selected.

**Context menu & file ops.** Right-clicking a tree row (`contextmenu`) selects it and opens a single floating `#tree-ctx-menu` (built once in `tree.js`, positioned at the cursor and nudged back inside the viewport, dismissed by any outside click or Escape). Its items, each with a leading glyph icon: **New file**, **New folder**, a separator, **Rename**, **Add to chat**, **Copy path**, **Show in file browser**, a separator, then a red **Delete** (`.ctx-danger`). The opener is also exported as `showTreeContextMenu(ev, rel, dir)` so the search-results list can raise the same menu on its (always-file) rows. **New file** runs the shared `createFileIn(dirOf(rel, dir))` against the right-clicked row — so it lands inside a folder or alongside a file (the same flow as the toolbar **+** button); **New folder** does the same through `createFolderIn`. Rename prompts for the new name through `promptText` (pre-filled, **Rename** confirm button) and calls `rename-file` (main `renameSync`, both old/new paths guarded against escaping `repoPath`, refuses to clobber an existing name). Delete runs the shared `deleteEntry(rel, dir)`: it confirms through `confirmDialog` (danger variant) and calls `delete-file`, which moves the target to the OS Recycle Bin via `shell.trashItem` (recoverable, not a permanent unlink); the same `deleteEntry` is bound to the **Delete key** — `#file-tree` carries `tabIndex = -1` and `select()` focuses it, so pressing Delete while the tree has focus deletes the selected row (with the same confirm); failures (rename or delete) report through `confirmDialog` as a one-button notice. Both refresh the tree afterward (expansion preserved, per above) and fix up `selected`. **Copy path** writes the repo-relative path with `navigator.clipboard`; **Add to chat** sends a Claude `@<rel>` mention into the active session via `sendToActiveSession()` (exported from `sessions.js`, writes to the session PTY and refocuses it). **Show in file browser** calls `reveal-in-folder`, which guards the path against escaping `repoPath` and opens the OS file browser with the item selected via `shell.showItemInFolder`.

**Drag to terminal.** Tree **file** rows are `draggable` and put `@<rel>` on the drag's `text/plain`; the git pane's **changes/staged/conflict rows** do the same (`makeFileDraggable()` in `git-pane.js`). `#terminal-host` accepts the drop (`dragover` sets `dropEffect = 'copy'`; `drop` reads `text/plain` and feeds it to `sendToActiveSession()`), so dragging a file into the terminal inserts its `@<rel>` mention into the active Claude session — the same text the **Add to chat** menu item sends. The same drop target also accepts **OS file drops** (from Explorer/Finder): those carry no `text/plain`, so the handler falls back to `dataTransfer.files`, resolves each to an absolute path via `api.pathForFile()` (preload-exposed `webUtils.getPathForFile`, since Electron 31 dropped `File.path`), and sends them as space-separated `@<abs>` mentions. A `#file-search` box atop the pane (debounced 150ms) replaces the tree with a `#search-results` list while it has text: **Files** come first from `search-names` (an async recursive walk in main, skipping `.git`/`node_modules`, case-insensitive and NFC-normalised (so a precomposed query matches a filename a macOS volume hands back decomposed), capped at 500 — the query is split on whitespace into terms that **all** must match the file's full repo-relative path, so `enemy .mp3` finds files whose path contains both `enemy` and the `.mp3` extension (folder names count too); a `*.png`/`.png` term matches by extension, any other term is a substring of the path, so a bare `png`/`js` gathers that file type too), then a **References** section is filled in asynchronously from `search-refs` (`git grep -n -I -F -i --untracked`, a subprocess so it never blocks the UI; capped at 500) — the filename results render immediately and don't wait on it. A run-token guards against a slow refs response landing after the query changed. Clicking a result opens it via `openFromTree()`; a reference result also passes `{ line, term }` so `showFile()` calls `jumpToLine()` to centre that line in the editor and select the matched word in the textarea. Right-clicking a result raises the same context menu as the tree (via `showTreeContextMenu`), treating the hit as a file. The opened folder isn't labelled in the sidebar; instead `setWindowTitle(repoPath)` sets the OS window title to `IDE / <project name>` (the path's basename), updated on startup (via `refreshGit`) and on **Open folder**.

## Quick Open

A keyboard-driven fuzzy file switcher (`quick-open.js`), the IDE staple for jumping to a file without the tree. **Ctrl/Cmd+P** from anywhere opens a centered palette — the listener is **capture-phase** (like [terminal links](#terminal-links--inline-browser)) so a focused xterm terminal can't swallow the key first, and it `preventDefault`s the browser's own print binding. On open it fetches the repo's flat file list **once** over `list-files` and caches it for that session of the palette, then fuzzy-matches **client-side on every keystroke** — no per-keystroke IPC, so typing is instant. `list-files` (in `explorer.js`) walks the tree once with the `.git`/`node_modules` skip set (capped at 10k) and **memoizes the result** — the same cached list backs `search-names`, and both re-walk only after the repo's `fs.watch` (the explorer tree watcher) fires or the folder changes, so repeated opens/searches on a big repo never re-walk. Concurrent requests share the one in-flight walk.

The matcher is `renderer/shared/fuzzy.js`, kept DOM-free so it's unit-tested the same way as `git-status.js` (see [testing.md](testing.md)). `fuzzyMatch(query, target)` requires the query to be a subsequence of the path (case-insensitive via the shared `text-fold.js` `fold()`, so non-ASCII paths match regardless of case; whitespace ignored) and scores it by what makes a guess feel right: runs of consecutive characters, matches on a word/path boundary (a separator, `.`/`_`/`-`, or a camelCase hump), and matches inside the **basename** rather than a parent dir; ties break toward the shorter path and earlier first match. It returns the matched indices so the palette can **bold** them. `fuzzyFilter(query, items, limit)` ranks the whole list and keeps the top 100. ↑/↓ move the highlight (wrapping, `scrollIntoView`), Enter opens the highlighted file through the same `openFromTree()` the tree and search use, and Esc / backdrop click dismiss. The list shows `quickOpen.noMatches` when nothing matches and `quickOpen.empty` when the repo has no files.

## Command Palette

The action sibling of Quick Open (`command-palette.js`): where Quick Open jumps to files, this runs app commands. **Ctrl/Cmd+Shift+P** from anywhere opens the same centered palette shell (it reuses the `.qo*` overlay styles, plus `.cmdp-row`/`.cmdp-title`), with the listener again **capture-phase** so a focused terminal can't swallow it. The command set is registered **once** from `renderer/index.js` — the only place every action function (`newSession`, Quick Open's `open`, `browseForFolder`, `cycleTheme`, `refreshGit`, `refreshTree`, opening Settings) is in scope — via `registerCommands([{ id, titleKey, keywordsKey?, run }, …])`. Commands carry i18n **keys**, not resolved strings, so the palette re-localizes every title each time it opens; switching language in Settings is reflected with no re-registration. `cycleTheme()` (exported from `settings.js`) advances the theme in the `THEMES` registry and persists it, so "Color Theme: Next" flips themes without opening Settings.

The matcher is `renderer/shared/command-match.js`, DOM-free and unit-tested like `fuzzy.js`. `matchCommands(query, commands, limit)` reuses `fuzzyMatch` against each command's **visible title** first (a title hit supplies the highlight positions and ranks by its own score); when the title misses, it falls back to the command's hidden **keywords** (synonyms like "preferences" → Settings), scored below any title hit (a fixed `KEYWORD_PENALTY`) and without highlights. An empty query lists every command in registration order. ↑/↓ move the highlight, Enter runs the highlighted command (errors are caught and logged, never thrown into the palette), and Esc / backdrop dismiss; the list shows `command.noMatches` when nothing matches.

## File editor

Opening any text file (`showFile()` in `viewer/file.js`) mounts an **always-editable** IDE-style editor in the shared `#diff-view` container — there is no separate read/edit mode. A transparent `<textarea>` (`.editor-ta`) overlays a syntax-highlighted `<pre>` (`.editor-pre`) pixel-for-pixel: the textarea owns native editing (caret, selection, undo/redo), its text is `color: transparent` with a visible `caret-color`, and the highlight behind it shows through. A line-number gutter (`.editor-gutter`) sits left and is `position: sticky; left: 0` so it stays pinned during horizontal scroll. On every `input`, `paint()` re-highlights the whole text (capped at 5000 lines, then plain) and rebuilds the gutter so the overlay stays aligned.

**Edits live in memory until saved.** A `buffers` `Map<file, {value, saved}>` holds each file's current text and last-known on-disk content; `dirty = value !== saved`. Switching files or selecting a session never writes to disk and never discards the buffer — reopening a file restores its unsaved edits. A non-dirty buffer is re-synced from disk on open, so external changes (e.g. a Claude session editing the file) show up. **Save** (the `#diff-save` button in the head, or **Ctrl/Cmd+S**) writes `value` over `write-text` (main guards the path against escaping the repo), sets `saved = value`, and calls `refreshGit()` so the new diff appears immediately. The button is disabled when clean, accented when dirty, and the title shows a `•` while unsaved; it is shown only while a file is mounted (`hideDiff()` retires it, so the diff view never shows it). `Tab` inserts a tab and `Enter` carries the current line's leading indent, both via `document.execCommand('insertText', …)` to preserve the native undo stack.

**HTML preview.** For `html`/`htm` files (`HTML_EXT` in `shared/ext.js`) the head carries a **Code | Preview** segmented switch (`#diff-preview`) **absolutely centred** in `#diff-head` (`position: absolute; left: 50%`), so its placement never shifts. **Preview** saves the buffer first (so the rendered page mirrors what you see), hides the editor DOM, and mounts a `<webview>` (`.editor-preview`) over `#diff-body` showing the page from its `file://` URL — out of process, so the host CSP doesn't restrict the local page; **Code** removes the webview and restores the editor with its caret/scroll intact (the editor is *hidden*, not unmounted, and `previewEl` doubles as the "previewing" flag). Both segment labels stay rendered in equal `1fr 1fr` grid columns with a 50%-wide CSS thumb that slides to the active one (`.previewing`), so the control's width is identical in either state and across languages — toggling reflows nothing. The labels are localized by their own `data-i18n` attributes (not flipped in JS); `syncToggle()` only moves the active class/thumb and updates the localized `data-i18n-title` tooltip. Clicking a segment selects that mode, clicking the track flips, and Enter/Space toggle (the switch is `tabindex`-focusable with `role="switch"`). The `file://` URL is resolved in main (`file-url` in `explorer.js` → Node's `pathToFileURL`, which adds the drive letter and percent-encodes spaces/non-ASCII) and guarded to stay in-repo like the other path handlers. `hideDiff()` retires the switch alongside Save, and `mountEditor()`/`showNotice()` reset `previewEl`, so non-editor views never carry preview chrome.

**Find in file.** **Ctrl/Cmd+F** while a file is mounted opens the floating `#editor-find` bar (query box, `current / total` count, match-case toggle, prev/next, close). The pure matcher is `renderer/shared/find.js` — `findMatches(text, query, caseSensitive)` returns all non-overlapping `{start,end}` ranges, `nearestMatch()` picks the hit at/after the caret on open, and `stepMatch()` walks them with wraparound; all three are unit-tested (`test/find.test.mjs`). Case-insensitive matching runs through the shared **`renderer/shared/text-fold.js`** `fold()` (used by every search surface), which is length-preserving so a folded offset is a valid offset into the raw text — the `<mark>` ranges line up even past a Turkish "İ" (whose plain `toLowerCase` is two code units and used to shift every later highlight); it also folds "İ"/"I" → "i" so a lowercase query like `istanbul` finds `İstanbul`. Crucially the matches are **not** shown via the textarea's selection (the next keystroke would overwrite a selected range). Instead a third layer — the `.editor-hl` div, mounted behind the `<pre>` (`z-index` 0 < pre 1 < textarea 2) — mirrors the full text with each hit wrapped in a `<mark>` (current hit brighter); because it shares the `<pre>`'s text, font, padding, `tab-size`, and `white-space: pre`, the marks line up exactly with the rendered code without measuring glyph widths. `showCurrentMatch()` repaints the marks, parks a **collapsed** caret on the current hit (so closing the bar lands there with nothing selected), and scrolls it to centre via `scrollOffsetIntoView()`. Enter / Shift+Enter (or the ↑/↓ buttons) step while focus stays in the query box; Esc / × close. `Alt+C` toggles match-case. The same overlay backs **reference-search jumps**: `jumpToLine(line, term)` (called when a `search-refs` result is clicked, [File explorer](#file-explorer)) paints a single transient yellow mark on the term and clears it after 2s, replacing the old select-the-term behaviour for the same overwrite reason. `hideDiff()` resets the bar (hidden) and clears the overlay alongside the Save button, so a diff/asset view never carries editor chrome.

**JSON structure actions.** For `.json` files a floating pill (`#editor-json-actions`, styled after the find bar, bottom-right of the editor body) offers **+ Item** or **+ Key** for the innermost array/object the caret sits in — hidden outside any container, in an unclosed one, or for non-JSON files. The edit itself is the pure `renderer/shared/json-edit.js` `jsonInsertion(text, offset)` (unit-tested, `test/json-edit.test.mjs`): a string/escape-aware bracket scan finds the innermost container and its close, then builds the appended `""` element or `"key": ""` entry matching the container's existing layout — inline containers append inline (`[1, 2]` → `[1, 2, ""]`), multiline ones add a properly indented line (indent taken from the last element's line, or bracket indent + one detected tab/space step when empty), with an existing trailing comma tolerated. `file.js` applies the insert through `insertText` (native undo intact) and then selects the placeholder (`key`, or a collapsed caret between the new value's quotes) so typing immediately replaces it. The bar's visible button tracks the caret (`keyup`/`mouseup`/`input`), its `mousedown` is prevented so clicking doesn't move focus off the textarea, and it's retired by `hideDiff()`/preview/notices like the rest of the editor chrome.

## Syntax highlighting

Both the diff (`renderDiff`) and the file editor (`paint`) colour code with **highlight.js** — loaded as the prebuilt browser bundle `@highlightjs/cdn-assets` via plain `<script>`/`<link>` in `index.html` (no bundler), exposing `window.hljs`. The `vs2015` theme stylesheet supplies the `.hljs-*` token colours; rows keep their existing `.diff-code` background, so add/del/hunk colouring still shows through.

`langFor(file)` maps a file extension to an hljs language via `EXT_LANG` (covers python, js/ts, c#, c/c++, rust, go, swift, java, kotlin, ruby, php, bash, sql, json, yaml, xml, css, markdown, objc, …). Two grammars aren't in the common bundle and are loaded as extra self-registering scripts: **dos** (`.cmd`/`.bat`) and **powershell** (`.ps1`). Godot is handled without a dedicated grammar: `.gd` → python (GDScript is python-shaped) and `.import`/`.tscn`/`.tres`/`.cfg`/`.godot` → ini. Unknown extensions return `null` → the file viewer auto-detects (`hljs.highlightAuto`); the diff viewer leaves them plain.

- **File editor** highlights the whole text once, then `hlLines()` splits the HTML into per-line fragments, re-opening any span left open across a newline so each line stays balanced (correct for multi-line strings/comments); the fragments are re-joined with `\n` into the highlight `<pre>`.
- **Diff viewer** highlights each line in isolation via `hlLine()` — a hunk is a fragment with no whole-file context, so multi-line constructs may colour imperfectly; hunk headers (`@@`) stay plain. Diff text is selectable/copyable (`.diff-code` opts back in to `user-select: text` against the global `body { user-select: none }`), while line numbers (`.diff-ln`) stay unselectable and the `+`/`-`/space markers are `::before` generated content — so a copy yields clean code without gutters or change markers. Applies everywhere the shared renderers run: the center diff overlay (git-pane changes and History commit/stash patches) and the per-session Diff dialog, in both unified and split modes.

## Asset viewer

Binary assets have no meaningful text diff, so clicking one in the git pane (or opening it from the tree) opens `#asset-view` instead. The set of routed extensions lives in `shared/ext.js` (`IMG_EXT` / `AUDIO_EXT` / `MODEL_EXT` / `VECTOR_EXT` / `SCENE_EXT`, kept disjoint); `viewer/center.js` checks them to route to the asset viewer rather than the text/diff view. The renderer fetches the file's bytes as base64 over `read-asset` (`explorer.js` reads it from `repoPath`, returns `{ base64, mime }`), then picks a view by type (image/audio build a `data:` URL; the 3D and vector editors decode the base64 themselves — to an `ArrayBuffer` / to text; the Godot scene editor branches before `read-asset` and reads the file as text over `read-text`):

- **Pixel editor** — PNGs under 200×200. Drawn onto a `<canvas>`, blown up by an integer scale. A floating tool rail pinned to the viewport's left-center edge (Blender-style — three big icon buttons stacked vertically, overlaid on the canvas; the rail forwards pointer events only through the buttons so its gaps don't block painting) picks one of three tools, each binding the mouse buttons differently: **Pen** (left paints, right erases), **Eraser** (the same brush inverted — left erases, right paints), and **Fill** (left flood-fills, right flood-erases). The pen/eraser stamp a square block sized by a **brush-size** slider in the bottom panel (1–32 px, default 1); selecting Fill re-labels that one slider as a **threshold** slider (0–100 %, default 0) — each tool keeps its own value, switching tools never bleeds one into the other. Painting uses the current palette/`<input type="color">` colour (the colour wheel); the −/+ toolbar buttons darken/lighten it by lerping toward black/white. The flood fill itself is the pure `floodFill()` helper in `shared/pixel-ops.js` (4-connected scan over the `ImageData` buffer, matching the seed colour within the per-channel threshold) so it's unit-tested without a canvas. The canvas's context menu is suppressed since right-click is the erase stroke. Mouse wheel changes the integer zoom; middle-button drag pans (via `assetBody` scroll), while a middle click without dragging eyedrops the pixel underneath. Undo/redo (↶/↷ buttons or Ctrl+Z / Ctrl+Y, capped at 50 `ImageData` snapshots) is pushed once per stroke or fill. **Save** sends `canvas.toDataURL()` bytes back over `write-asset` and refreshes git.
- **Image zoom** — any other image. An `<img>` with −/+/reset buttons scaling its width. Left-button drag pans a zoomed image by scrolling `#asset-body` (grab/grabbing cursor); the move/up listeners live on the document so the drag keeps tracking past the image edges and are torn down via the asset view's cleanup hook. For canvas-encodable raster formats (`png`/`jpg`/`jpeg`/`webp`) the toolbar also carries an **Adjust** button that switches to the adjustment view below; the two views (zoom ↔ adjust) share the one decoded `<img>` and a coordinator (`viewer/asset/index.js`'s `renderImage`) that keeps each sub-view's header buttons in their own `.asset-view-tools` slot — separate from the shared **Open externally** button — and runs the active sub-view's cleanup on switch.
- **Image adjustment** — a non-destructive editor for the same raster formats. A live-preview `<canvas>` over a panel of `−100..100` sliders — **exposure, brightness, contrast, saturation, vibrance, temperature, tint** — each re-rendering the preview as it's dragged (double-click a slider to zero it; **Reset all** zeros everything). The pixel maths is the pure, unit-tested `shared/adjust-ops.js` (`applyAdjustments(src, dst, values)`): exposure/brightness/contrast/temperature/tint collapse into one 256-entry LUT per channel (`buildChannelLuts`), then saturation + vibrance mix the channels per pixel (vibrance weighted by a pixel's existing saturation so near-grey pixels boost most). The view (`viewer/asset/image-adjust.js`) reads the source pixels **once** and recomputes from them every change (so adjustments never compound), coalescing slider input to **one pipeline run per animation frame** (`requestAnimationFrame`) since a multi-megapixel pass per `input` event would stutter. **Apply** bakes the current canvas back into the file via `canvas.toDataURL(<original mime>)` → `write-asset` (the same path the pixel editor saves through, keeping the file's format) and refreshes git; the applied pixels become the new baseline and the sliders return to neutral. **Done** returns to the zoom view.
- **Icon viewer** — `ico` (in `IMG_EXT`, but branched before the generic image path in `viewer/asset/index.js`). An `.ico` bundles several independently-encoded frames (16px…256px, PNG or BMP), and a plain `<img>` shows only the one Chromium picks — so `viewer/asset/ico-view.js` shows them all: the container is parsed by the pure, unit-tested `shared/ico-parse.js` (`parseIco` reads the ICONDIR + entries, trusting a PNG frame's IHDR over the directory's 1-byte sizes, largest-first; recognizes `.cur` type-2 containers) into a summary line ("Windows icon · N images · bytes") plus a grid of cards, one per frame, each labelled `describeEntry`-style ("32×32 · 32-bit · PNG") on a checkerboard so transparency reads. Frames are never pixel-decoded in JS (`extractFrame`): a PNG frame is a complete PNG stream served **as-is** as `image/png` (Chromium's ICO decoder rejects PNG frames past the format's 256px cap, and real icons embed e.g. 2048px PNGs behind a 0 directory byte — its PNG decoder has no such limit), while a headerless BMP DIB frame is repackaged verbatim as a **single-image .ico blob** (`buildSingleIco`) for Chromium's ICO decoder; clicking a card opens that exact frame in the shared zoom view with an "All sizes" back button. A file that fails to parse (usually a renamed PNG) falls back to a plain `<img>` zoom of the raw bytes. Blob URLs are revoked by the view's cleanup. View-only — no editing/adjust.
- **Audio player** — `wav`/`ogg`/`mp3`. A fully custom player (the native `<audio controls>` is hidden and driven programmatically): a circular play/pause transport, current/duration readout, a speed-cycle button, and a mute toggle + volume slider. Above it an interactive mirrored bar waveform, decoded once from `AudioContext.decodeAudioData` (base64 → `ArrayBuffer`, no network) and reduced to one normalised peak per bar. The played portion fills with an accent gradient as it plays (driven by a `requestAnimationFrame` loop only while playing); clicking or dragging the waveform seeks, and hovering shows a time bubble. Bars recompute on panel resize via a `ResizeObserver`. The view registers a cleanup that cancels the rAF loop, disconnects the observer, and pauses playback.
- **3D model viewer** — `glb`/`gltf`/`fbx`/`obj`/`usdz`/`stl`/`ply` (`MODEL_EXT`). A [three.js](https://threejs.org) scene whose common parts — scene/camera/renderer/lights/grid/`OrbitControls`/render-loop, the per-format loaders, `frameObject`, the outliner, and disposal — live in the shared **`viewer/asset/model-scene.js`** core, used by both the read-only entry (`viewer/asset/model.js`) and the editor (`viewer/asset/model-editor.js`). `OrbitControls` (drag to orbit, scroll to zoom, right-drag to pan), a hemisphere + key light so an unlit mesh stays readable, and a ground grid. Each format has its own loader (`GLTFLoader`/`FBXLoader`/`OBJLoader`/`STLLoader`/`PLYLoader`/`USDZLoader`); they all reduce to a single `Object3D` that `frameObject()` recentres on the origin and pulls the camera back to fit the bounding sphere (a **Reset view** button re-applies that framing). The bare-geometry formats (STL/PLY) are wrapped in a default `MeshStandardMaterial`, honouring per-vertex colours and flat-shading a normal-less mesh. A **scene-graph outliner** (`buildHierarchy()`, `.model-tree`) floats over the canvas's top-left as a translucent panel: a collapsible tree mirroring the `Object3D` hierarchy (loader-injected lights/cameras filtered out), with a short type tag per node (`M`/`G`/`B`), per-node caret to collapse a branch, an eye toggle for that branch's `visible` flag, a wireframe toggle (walks the node's subtree setting each mesh material's `wireframe`), and click-to-select that parks the orbit pivot (`controls.target`) on the node's bounding-box centre. Any node with **`collider` in its name** (or under one) is forced to wireframe on load (`applyColliderWireframe`) so a physics volume reads as a cage instead of occluding solid textured geometry, and its row's wireframe toggle starts on. **Empty nodes** — transforms with no mesh anywhere beneath them (locators, bone roots) — would otherwise be invisible, so `addEmptyMarkers` drops a small black 3-axis cross marker at each (a `LineSegments` child, `depthTest` off, tagged `userData.isEmptyMarker` so the outliner skips it as a row); selecting an empty flips its marker to white. Hovering or selecting a node draws a `THREE.BoxHelper` (accent-coloured, `depthTest` off so it shows through geometry) around it so you can see which part it is; the hover box falls back to the selected node's box on mouse-leave. A glTF mesh that uses several materials loads as a group of per-material primitive meshes; the outliner **folds those primitives into one row** (detected via the loader's `associations`, tracked in a `WeakSet` so the tags never leak into a saved file) so a multi-material mesh reads as one object rather than one row per material. The panel header collapses the whole tree. `model.js` is **dynamically imported** the first time a model opens, so three.js (a large dependency) never costs app startup. The module is loaded from `node_modules` via an **import map** in `index.html` (`three` → `three.module.js`, `three/addons/` → `examples/jsm/`); the import map must precede the renderer's module script. Cleanup cancels the render loop, disconnects the `ResizeObserver`, disposes every geometry/material/texture, and force-loses the WebGL context. **Limitation**: a model is loaded from its own bytes alone, so a `.gltf`/`.obj` referencing external `.bin`/`.mtl`/texture files renders without them.

**Editor mode (glTF/GLB only).** For `glb`/`gltf` the viewer's header carries an **Edit** button (gated on `EDITABLE_MODEL_EXT` — the only formats three.js's `GLTFExporter` can write back); other formats stay view-only. A coordinator in `viewer/asset/index.js` (`renderModelCoordinator`, mirroring the image view's `renderImage` zoom↔adjust switch) toggles between the read-only `renderModel` and the editor `renderModelEditor` over the same loaded bytes, each owning its own header-button slot (`.asset-view-tools`) and cleanup. Entering edit mode tags `#asset-tools` with `asset-edit-mode`, whose CSS hides "Open externally" so **only a Save button** shows; **Ctrl+S** also saves, and Save is **disabled until the first edit** (a `dirty` flag set by every committed change, cleared on save). The editor adds a **transform gizmo** (`TransformControls`, attached to the current selection and operating in the object's **local** space — its own origin/axes — so move/rotate/scale pivot on the part itself, not the world or parent frame; **W/E/R** switch mode; the gizmo helper lives on the scene, never the model root, so it's excluded from export, and dragging it freezes `OrbitControls`), **click-to-select** in the viewport (a raycast on a click that isn't a gizmo grab or orbit drag, rolling a primitive hit up to its folded mesh; clicking empty space deselects), a floating **edit dock** (`.model-edit-panel`, top-right, mirroring the outliner) with **texture adjustments** (the model's base-color `material.map` textures via the pure `shared/model-textures.js` `enumerateTextures`, adjusted live with the same `shared/adjust-ops.js` pipeline as the [image adjustment](#asset-viewer) view, the edited `<canvas>` assigned back as `texture.image`), **material** props (base color / metalness / roughness / emissive), and **node** duplicate/delete; the outliner rows are **drag-and-drop reparentable** (`Object3D.attach` preserves world transform, drops onto self/descendants rejected). A command-stack **undo/redo** (Ctrl+Z / Ctrl+Y) covers every gesture. **Save** re-exports the live model root through `GLTFExporter.parseAsync(root, { binary: ext==='glb', onlyVisible: false })` — empty-marker crosses are detached first so they don't export as junk meshes, and a framing-pivot wrapper keeps `frameObject`'s recentring out of the saved root transform — then base64-encodes it (the chunked, stack-safe `shared/base64.js` `arrayBufferToBase64`) and writes via `write-asset` + `refreshGit()`. **Limitations**: editing is glTF/GLB only; a `.gltf` saves with embedded buffers/images; `GLTFExporter` re-encodes all textures to PNG on GLB save; texture adjustments touch only the sRGB base-color map; a multi-material mesh edits its first slot.

**Godot scene editor (`.tscn`, `SCENE_EXT`).** Godot scene files open a 3D editor (`viewer/asset/scene-editor.js`, dynamically imported by `renderSceneCoordinator` in `viewer/asset/index.js` — one mode, no view↔edit switch, and "Open externally" stays visible since it hands the scene to Godot itself). Switching between the 3D view and the raw text reuses the [HTML preview](#file-editor)'s **Code | Preview segmented switch**: the scene editor's header carries the same control (`.asset-seg` in `asset.css`, Preview active — Code runs `hideAsset()` + `showFile()`), and the text editor's `#diff-preview` toggle also shows for `SCENE_EXT`, where Preview saves the buffer and routes back through `center.js`'s `openFromTree` (imported lazily to avoid an eager file.js ↔ center.js cycle) instead of mounting a webview. The parsed document is the **source of truth**: the pure, unit-tested `shared/tscn.js` models the `.tscn` text (descriptor + `ext_resource`/`sub_resource`/`node`/`connection` sections, every attribute/property kept as its raw string so unknown types, scripts, signals and properties **round-trip untouched**), and the three.js graph — built on the same `model-scene.js` core (viewer, outliner, framing pivot, empty markers) — is just a live view of it: one `THREE.Group` per node section (tagged `userData.nodePath`), primitive meshes (`BoxMesh`/`SphereMesh`/`CylinderMesh`/`CapsuleMesh`/`PlaneMesh`/`QuadMesh`/`TorusMesh` with `StandardMaterial3D` albedo/metallic/roughness) built from their sub-resources, lights (`DirectionalLight3D`/`OmniLight3D`/`SpotLight3D`) as real three lights pointing local −Z plus a marker sphere, `Camera3D` as a cone, and instanced scenes / undecodable meshes (ArrayMesh, `.res`) as translucent placeholder boxes — all visual children tagged `isEmptyMarker` so the outliner lists only Godot nodes. Transforms map 1:1: Godot's 12 `Transform3D` numbers are the basis **rows** + origin, `Matrix4.set()` is row-major, and both engines are Y-up right-handed; `transformOfNode` also composes split `position`/`rotation` (Euler **YXZ**)/`quaternion`/`scale` props. Editing mirrors the glTF editor — local-space `TransformControls` gizmo with its **Move/Rotate/Scale** mode buttons and **Delete** in the header toolbar rather than the dock (W/E/R and Del still work; the dock keeps only the Add-child picker), click-to-pick (rolls a hit up to its owning node), an **Add child** dock section (mesh primitives create a `sub_resource` + `MeshInstance3D`, plus `Node3D`/lights/camera; names deduped Godot-style `Box`, `Box2`, …; `load_steps` kept honest), **Delete** (removes the subtree and any `connection`s into it), **drag-and-drop reparenting** in the outliner (Godot's reparent-keep-global-transform: `reparentNode` moves the section subtree depth-first under the new parent, renames on sibling collision, rebases descendant parent paths + connection endpoints; the three side uses `Object3D.attach` and writes the compensating local transform back for 3D nodes), and undo/redo where each command stores the serialized doc before/after (re-parsed wholesale — scene files are small) plus closures mirroring the step onto the three graph. Gizmo release writes the node's new local matrix back as a single leading `transform` prop (dropping split TRS props, matching Godot's own editor). **Ctrl+S** serializes the doc through `write-text` + `refreshGit()`. Only nodes Godot would let carry a 3D transform get the gizmo (`*3D` types, instances, or typeless override nodes that already store TRS). A **resources panel** (`.scene-res-panel`, a collapsible translucent strip pinned to the viewport's bottom) lists the project's 3D model files as draggable cards, built from the flat `list-files` listing by the pure `shared/scene-assets.js`: `godotRootOf` finds the nearest ancestor `project.godot` above the scene as the `res://` root (repo root as best-effort fallback; models outside the project are excluded), and a card's thumbnail is only ever a **pre-existing sibling image** sharing the model's basename (`robot.glb` → `robot.png`, loaded over `read-asset`) — never generated; without one the card shows a glyph. Dropping a card on the canvas raycasts the drop point onto the ground plane (converted to root-local coordinates) and instances the model there: `addExtResource` inserts a `PackedScene` ext_resource (**deduped by type+path**, Godot-shaped `<n>_<slug>` id, `load_steps` kept honest) and `addNode` emits a **typeless** `instance=ExtResource(...)` node under the root (the `type` attribute is omitted when `instance` is given, matching Godot), rendered as the usual instanced-scene placeholder box and undoable like every other edit. **Referenced model files actually load**: any node whose `instance=ExtResource(...)` (or a `MeshInstance3D`'s `mesh` ext_resource) points at a `MODEL_EXT` file (`.glb`/`.gltf`/`.obj`/…) starts as a placeholder box and asynchronously swaps in the real mesh — bytes over `read-asset` at the res-root-resolved repo path, parsed by `model-scene.js`'s `loadModel`, **one parse per file** cached and `clone(true)`d per instance, framing re-applied once after the initial loads settle (never on later drops), and a failed load (missing file, `.tscn`/`.res` target) just keeps the placeholder. **Limitations**: `.tscn` instances and `.res`/ArrayMesh meshes still render as placeholders; non-3D branches (Control, Node2D) list in the outliner but render nothing.

## Vector editor

`svg` and `ai` (`VECTOR_EXT`) route to a vector coordinator (`renderVectorCoordinator` in `viewer/asset/index.js`, a clone of `renderModelCoordinator`) instead of the image view. It toggles a read-only **preview** and — for SVG only (`EDITABLE_VECTOR_EXT`) — a full **editor**, dynamically importing each module so [paper.js](http://paperjs.org) never costs app startup.

- **Preview** (`viewer/asset/vector-view.js`). For SVG it renders the markup as a `data:image/svg+xml` `<img>` and reuses the [image zoom](#asset-viewer) view (`renderZoom`) — no paper.js loaded — plus an **Edit** button. For `.ai` it shows a **view-only info card**: the magic bytes are sniffed by the pure `shared/svg-ops.js` `aiKind`/`aiInfo` (`%PDF-` → PDF-based, `%!PS-Adobe` → PostScript; `/MediaBox` or `%%BoundingBox` for size, `%%Creator` for the app) and the user is pointed at the header's **Open externally**. Modern `.ai` wraps a PDF and legacy `.ai` is PostScript — neither round-trips to editable paths in-browser, and an embedded PDF preview would need `plugins:true` + a CSP `object-src` loosening, so it stays an info card.

- **Editor** (`viewer/asset/vector-editor.js`, built on `vector-paper.js` + `vector-tools.js` + `vector-panels.js`). paper.js is the `-core` build (API only, no PaperScript/`eval`, CSP-clean), lazy-loaded by **injecting its `<script>`** on first Edit (`loadPaper()`): it ships a UMD/global bundle, not clean ESM, so it can't ride the three.js import map — but an external same-origin script is permitted by the page CSP `script-src 'self'` (the nonce is only required for *inline* scripts). `createCanvas` sets up an isolated `PaperScope` on a `<canvas>`; `importSVG`/`exportSVG` round-trip the document. The shell wires a single paper `Tool` that delegates to whichever tool is active, plus a left **tool rail**, the canvas, and a right **dock** of panels.
  - **Tools** (`vector-tools.js`, each a `(ctx) => toolObject` factory): **select/move** with an 8-handle scale box + rotate knob and marquee; **direct node edit** (drag anchor points and bézier handles); **pen** (click corners, click-drag handles, click the start to close); **pencil** (freehand → `path.simplify`); **shapes** (rectangle / rounded / ellipse / line / polygon / star); **text** (`PointText` with an inline `<input>` editor); **eyedropper**; **fill bucket**; **eraser**. Pan (middle-drag) and zoom (wheel, toward the cursor) are handled by the shell in screen space so they never fight a tool gesture. The transform handles and marquee live on a dedicated overlay `Layer` that is pulled out of the project for every export/snapshot, so they never leak into the saved SVG.
  - **Dock panels** (`vector-panels.js`): a **Style** section bound to the selection (fill solid/linear/radial-gradient/none, stroke colour/width/cap/join/dash, opacity, blend mode, rect corner radius); **Arrange** (z-order, group/ungroup, duplicate, delete, align/distribute, flip, rotate 90°); **Path** (boolean `unite`/`intersect`/`subtract`/`exclude`/`divide` via `PathItem`, plus join/close/reverse/simplify/smooth/flatten); and a **Layers** list (add/select/rename/visibility/lock/reorder/delete). The structural operations are the editor's `actions` object; z-order/align use the pure `shared/svg-ops.js` helpers (`bringToFront`/`forwardOne`/… and `alignOffsets`).
  - **Undo/redo** is a `project.exportJSON()` **snapshot ring** (≤40 entries, the [pixel editor](#asset-viewer)'s `ImageData`-ring approach, not a command stack — robust against booleans/grouping that exact inverse closures handle poorly); a restore re-imports the JSON and resets selection. **Save** exports SVG, re-stamps the document's original `width`/`height`/`viewBox` and guarantees `xmlns` (`svg-ops.js`'s `applySvgSize`/`ensureSvgXmlns`), then writes it UTF-8-base64 (`shared/base64.js`'s `textToBase64`) via `write-asset` + `refreshGit()`. **Ctrl+S** saves; Ctrl+Z/Y, Delete, `[`/`]`, Ctrl+G/Shift+G/Ctrl+D, and single-key tool shortcuts are wired in a capture-phase keydown.

  **Limitations**: save is **reconstructive** — it rewrites the SVG from paper's model, so filters/masks/clip-paths/`<use>`/CSS `<style>`/markers/patterns/embedded fonts/scripts and comments/metadata may be dropped or altered (the *preview* is always faithful; only *Save* rewrites). `.ai` is view-only. The fill bucket sets the clicked item's fill (no multi-path region flood); the eraser deletes items; text is single-line `PointText`; blend modes are best-effort on export; undo clears the live selection after a restore.

`data:` URLs for `<img>`/`<audio>` require the CSP in `index.html` to allow `img-src`/`media-src 'self' data:`; the 3D loaders create `blob:` URLs for embedded textures, so `img-src`/`connect-src` also allow `blob:`. The editor's `GLTFExporter` and `TransformControls` load from `node_modules` under the **existing** `three/addons/` import-map entry, so editor mode adds **no** CSP or import-map change. A strict `script-src 'self'` **blocks an inline import map**, so the CSP whitelists it with a **nonce** (`'nonce-threejsimportmap'`, matched by the import map tag's `nonce` attribute) rather than a content hash — a hash over the block's exact bytes silently breaks whenever an editor reformats it or a CRLF checkout rewrites the line endings, whereas the nonce is content-independent. (A static nonce is safe here: the app is a local `file://` page, so the per-request randomness a nonce normally provides is moot — anyone who can alter the local HTML already has code execution.)

**Open externally.** The asset view's head toolbar (`#asset-tools`) carries an **Open externally** button for every asset type that hands the file to the OS's default program for its extension (a `.glb` in the system 3D viewer, an image in the photo app). It routes through `open-asset-external` (`explorer.js`), which resolves the repo-relative path against `repoPath` and calls `shell.openPath` — distinct from `open-external` (which takes an already-absolute path / URL and backs the terminal-link feature).

## PDF viewer & page editor

`pdf` (`PDF_EXT` in `shared/ext.js`) routes to a PDF coordinator (`renderPdfCoordinator` in `viewer/asset/index.js`, a clone of `renderModelCoordinator`) that toggles a read-only **view** and a page-level **editor**, each dynamically imported so neither library costs app startup. Both resolve their library through the `index.html` import map (`"pdfjs-dist"`/`"pdfjs-dist/worker"` → the **`legacy/build`** files, `"pdf-lib"` → `dist/pdf-lib.esm.js`). The legacy pdf.js build is required: the modern build uses `Promise.try` (Chromium 128+), which this app's Electron 31 (Chromium 126) lacks — with it, opening a PDF fails as an unhandled rejection and the pane stays blank; the legacy build ships the polyfill.

- **View** (`viewer/asset/pdf-view.js`). Renders every page with **[pdf.js](https://mozilla.github.io/pdf.js/)** as a `<canvas>` in a vertical scroll (`.pdf-pages`), sequentially so page 1 shows immediately; −/+/Reset zoom re-renders at the new width (a render token cancels a superseded pass), page-count readout, and an **Edit** button. The shared loader/render glue is `viewer/asset/pdf-doc.js`: `loadPdfjs()` imports pdf.js **and its worker module into the main thread** (import-map entry `"pdfjs-dist/worker"`) — the worker module registers `globalThis.pdfjsWorker`, which pdf.js detects and uses directly instead of spawning a Web Worker. **Do not switch this to a real worker**: on a `file://` page `location.origin` is `"null"`, so pdf.js's same-origin check fails against its own worker file and it wraps it in a `blob:` worker, which the CSP blocks — the document then never opens (a blank pane). `openPdf()` passes `isEvalSupported: false` so pdf.js never reaches for `new Function` (CSP-clean); `renderPageCanvas()` renders one page devicePixelRatio-sharp at a CSS width.
- **Editor** (`viewer/asset/pdf-editor.js`). **Page assembly, not content authoring**: a grid of page thumbnails (each source page rendered *once* by pdf.js; rotation is a CSS transform, reorder just moves the canvas) with rotate ⟲/⟳, delete (blocked on the last page), and reorder via drag-and-drop. The state is the pure, unit-tested **`shared/pdf-ops.js`** page-list model (`test/pdf-ops.test.mjs`): an array of `{ src, rotation }` descriptors with immutable `makePages`/`rotatePage`/`deletePage`/`movePage` ops and `isIdentity` (drives the dirty flag/Save enablement); undo/redo (Ctrl+Z/Y) is a plain snapshot stack of those arrays. **Save** (button or Ctrl+S) is when the document is first touched: **[pdf-lib](https://pdf-lib.js.org)** loads the original bytes, `copyPages` in the edited order into a fresh `PDFDocument`, applies each rotation delta on top of the page's existing rotation, and the result goes base64 (`shared/base64.js`) → `write-asset` → `refreshGit()`; the saved bytes become the coordinator's baseline so **Done** returns to a preview of what's on disk. Delete/arrows work on the selected thumbnail from the keyboard; the keydown listener is torn down via the sub-view cleanup.

**Limitations**: editing is page-level only (no text/image/annotation editing); pdf-lib rebuilds the file, so document-level metadata not carried by the copied pages may be dropped; an encrypted PDF fails with the error shown in the pane.

## Spreadsheet viewer

Tabular files open in an editable grid (`#sheet-view`) instead of the text editor. The routed extensions are `SHEET_EXT` in `shared/ext.js` (`csv`/`tsv` + the Excel workbook formats `xlsx`/`xls`/`xlsm`/`xlsb`/`ods`), checked by `viewer/center.js`'s `openFromTree` and disjoint from the asset/text routes. File I/O is **[SheetJS](https://sheetjs.com) (`xlsx`)**, resolved from `node_modules` via the same `index.html` import map as three.js (`"xlsx"` → `xlsx.mjs`) and **dynamically imported** the first time a sheet opens, so the (large) library never costs app startup. CSV is read as text over `read-text`; the binary workbook formats are read as base64 over `read-asset` — the same handlers the file editor / asset viewer already use, so no new IPC was added.

**Parsing & the model (`viewer/sheet/workbook.js`).** `toModel(XLSX, data, type)` parses with `cellStyles`/`cellFormula`/`cellNF` on and flattens each worksheet into our editing model: `sheet = { name, cells: Map<"c,r", cell>, nCols, nRows, cols, merges }`, where `cell = { v, f, z, s }` — literal/cached value, formula text (no leading `=`), number format, and a display-only style (`bold`/`italic`/`underline`/`color`/`bg`/`align`, read from the file). `serialize()` rebuilds a SheetJS workbook and writes base64 (xlsx) or a CSV string. The SheetJS **community build writes values, formulas, and number formats but not fonts/fills/borders**, so applied text/cell styling is a session-level display feature that does not persist to `.xlsx` on save — values, formulas, and number formats round-trip faithfully (verified). Number-format display goes through SheetJS's `SSF.format` (`formatValue()`), so currency/percent/date formats render live, including on recomputed cells.

**Formulas — the engine is ours, no extra dependency.** Live recalculation is the dependency-free `shared/sheet-formula.js` (`evaluateFormula(src, ctx)`), a tokenizer → recursive-descent parser (Excel operator precedence: comparison, `&`, `+-`, `*/`, `^`, unary, `%`) → evaluator covering A1 refs/ranges and the common worksheet functions (SUM/AVERAGE/COUNT/MIN/MAX/IF/IFERROR/AND/OR, ROUND family, ABS/SQRT/POWER/MOD, text fns, COUNTIF/SUMIF, VLOOKUP). Excel errors are strings (`#DIV/0!`, `#NAME?`, …) thrown as `FormulaError` and surfaced as the cell result. The engine does **not** own the dependency graph — the grid passes a `ctx.getCell(col,row)` that returns another cell's computed value with memoization + cycle detection (`computeSheet()` in `viewer/sheet/index.js`; a cell referenced mid-evaluation resolves to `#REF!`). Address math (col-letter ↔ index, A1/range parsing) is the shared `shared/sheet-model.js`. Both pure modules are unit-tested (`test/sheet-formula.test.mjs`, `test/sheet-model.test.mjs`).

**The grid & editing (`viewer/sheet/index.js`).** The active sheet renders as a `<table>` (sticky header row + row-number column via CSS, `table-layout: fixed` so `<col>` widths hold and big sheets stay fast) inside the scrollable `#sheet-body`, with the used range plus a blank margin (`PAD_ROWS`/`PAD_COLS`) for new data and a render cap (`MAX_ROWS`/`MAX_COLS`) that shows a truncation note rather than freezing on a huge sheet. Merged cells display via `colSpan`/`rowSpan` on the anchor with covered cells skipped. A cell→`<td>` map lets edits repaint just the touched cells (after recompute, a full repaint ripples formula results). Selection is a rectangle (click, drag, shift-click, header click for a whole row/column); the **formula bar** + **name box** mirror the active cell (the name box jumps to a typed address). Editing is inline (double-click / Enter / F2 / typing) or via the formula bar; an entry starting with `=` becomes a formula, otherwise it's coerced to number/boolean/string. Keyboard: arrows/Tab/Enter navigate, Delete clears, Ctrl+B/I toggle bold/italic, Ctrl+C/V copy-paste a TSV block (through the main clipboard handlers). A right-click context menu inserts/deletes rows & columns (shifting stored coordinates and merges; **a known ceiling — formula references are not re-pointed across an insert/delete**). Multiple worksheets show as Excel-style tabs along the bottom (`#sheet-tabs`). **Save** (button or Ctrl+S) serializes the whole model and writes back over `write-text` (csv) / `write-asset` (xlsx), then `refreshGit()` — edits live only in memory until then, mirroring the text editor. CSP-safe: `xlsx.mjs` uses no `eval`/`Function`, and it loads under the existing import-map nonce. All `sheet.*` UI strings are in the locales.

## Database viewer

Single-file database files open in an editable database browser (`#db-view`) instead of the binary diff/text view: a sidebar of tables & views, a paged grid with inline cell editing, add/delete row, and a SQL console. The routed extensions are `DB_EXT` in `shared/ext.js`, checked by `viewer/center.js` (`openFromTree` **and** `openGitFile`, so a clicked git row opens the working copy too) and disjoint from the asset/sheet/text routes. `.sql` is deliberately **not** routed — it's DDL text, handled by the code editor.

**The engine — SQLite via WASM, on a worker thread.** The actual database work runs in a **`worker_threads` worker** (`src/main/db-worker.js`), not the renderer and not the main thread, using **[sql.js](https://sql.js.org)** — SQLite compiled to WebAssembly, pure JS with **no native build step** (matching this app's no-compile constraint). Keeping it out of the renderer sidesteps the renderer CSP entirely (no `wasm-unsafe-eval`, no import-map entry, no `.wasm` fetch over `file://`); keeping it off the **main thread** means a slow user query or a multi-megabyte database load never stalls PTY output or other IPC. `src/main/db.js` is the thin main-side proxy: it lazily spawns the worker on the first `db-*` call (startup never pays for it), forwards each call as a `{ id, method, repo, args }` message, resolves the matching reply, and restarts a crashed worker on the next call (in-flight calls fail cleanly). Inside the worker, sql.js is `require`d **lazily on first open** (it's large and pulls in a WASM module), with `locateFile` pointing emscripten at the `.wasm` next to it in `node_modules` (resolves transparently inside the packaged asar). sql.js loads the whole file into memory, so each open database is kept in an `open` Map keyed by absolute path — unsaved edits survive switching files (mirroring the text/sheet editors), and **Save** `export()`s the in-memory database and writes the bytes back. The cache is freed on folder change (`onRepoChange` → a `close-all` message).

**Which files open.** The pure, Electron-free `src/main/db-sql.js` (unit-tested in `test/db-sql.test.js`) holds the catalog of industry-standard single-file DB formats (`DB_ENGINES`) and the SQLite header sniff (`isSqliteBuffer` — the 16-byte `SQLite format 3\0` magic). **The header, not the extension, decides**: a `.db` that is really SQLite opens; a `.db` that isn't (Thumbs.db, a Berkeley DB) is rejected cleanly. The SQLite-family engines (`sqlite`/`sqlite3`/`db`/`db3`/`s3db`/`sl3`, plus the SQLite-based GeoPackage `.gpkg` and MBTiles `.mbtiles`) are `editable`; the rest — DuckDB, Microsoft Access (`.mdb`/`.accdb`), SQL Server (`.mdf`/`.ndf`/`.ldf`), MySQL/MariaDB data files, Firebird, dBASE (`.dbf`), Realm, Berkeley DB, HCL Notes, LibreOffice Base — are **recognized but not editable in-process**, so `db-open` returns the engine label and the renderer shows an informational panel (with an "open externally" escape hatch) rather than a broken grid. Server engines whose data is a directory rather than one openable document (PostgreSQL, MongoDB) are intentionally absent.

**Reading & safe writes (`db-sql.js`).** Each statement is built by a pure helper and run with **bound parameters** (never string-interpolated row data); identifiers are double-quote-escaped (`quoteIdent`). The crux is **row identity** for UPDATE/DELETE: `rowIdentity(columns, hasRowid)` prefers the table's **rowid** pseudo-column (ordinary tables), falls back to the declared **PRIMARY KEY** columns (WITHOUT ROWID tables), and last-resorts to **all columns** matched NULL-safely with `IS` — a target with only the last (a view, or a table with no identity) is flagged **read-only**, so the grid never offers an edit it can't safely scope to one row. `buildSelect`/`buildCount`/`buildUpdate`/`buildInsert`/`buildDelete` assemble the SQL (LIMIT/OFFSET are coerced to integers, so paging can't inject). `coerceInput(raw, declType)` converts a grid string to the bound value by **SQLite type affinity** (`affinity()`): a numeric-affinity column parses a numeric string to a Number, an **empty string clears to NULL** (documented — it can't store a literal empty string), everything else stays text. BLOBs come back as a labeled placeholder (`[BLOB n B]`) and are read-only (not editable inline). The **SQL console** runs arbitrary SQL via `db.exec`; `isReadOnlySql()` (a leading-keyword heuristic) decides whether the query dirties the file and invalidates the cached schema — a SELECT/PRAGMA reads, anything with a DML/DDL keyword marks the database dirty and refreshes the table list.

**The grid (`viewer/db/index.js`).** Pure UI over the `db-*` IPC: the sidebar lists tables then views; clicking one loads page 0. A table renders as a `<table>` (sticky header with each column's name + type and a 🔑 on primary keys, sticky row numbers) with **offset paging** (`PAGE_SIZE` 200; a footer shows the row range + prev/next and an **Add row** button, or a read-only note for views). Editing is inline (double-click → input, Enter/blur commits, Esc cancels), persisting each change immediately via `db-update-cell` with the row's identity values; an INTEGER-PK-alias edit updates the row's tracked rowid in place. Delete is a right-click context menu; Add row inserts an all-default row and jumps to the page holding it. NULLs render dimmed/italic. Edits live in the in-memory database until **Save** (button or Ctrl+S) → `db-save` → `refreshGit()`. All `db.*` UI strings are in the locales. The IPC surface (`db-open`/`db-table`/`db-update-cell`/`db-insert-row`/`db-delete-row`/`db-query`/`db-save`) is in `preload/db.js`.

## Run toolbar

The native Electron application menu is removed (`Menu.setApplicationMenu(null)` in `index.js`); a full-width **run toolbar** (`#toolbar`) sits at the very top of the window instead (`body` is a flex column: toolbar then the `#app` grid). It surfaces VS Code run configs as buttons: **one per launch config** (`.vscode/launch.json`), a separator, then **one per task** (`.vscode/tasks.json`). Clicking a button runs that config/task in a **git-pane terminal tab** (see [Git-pane consoles](#git-pane-consoles)) and focuses it — there is no in-process debugger.

`run-configs.js` parses both files as **JSONC** (`parseJsonc()` strips `//` and `/* */` comments outside strings, then trailing commas, then `JSON.parse`). `get-run-configs` returns the launch names (configs + compounds) and task labels for the toolbar — entries hidden in VS Code stay hidden here (task `hide`, launch/compound `presentation.hidden`), and `presentation.order` sorts the visible ones. `run-config({kind, name, inputs?, activeFile?})` **re-reads** the file each time (so edits are picked up), finds the entry by name, and returns one or more **run specs** `{command, cwd, env, name}` (a compound yields one per referenced configuration). It builds each command line via:

- **Launch** (`buildLaunchCommand`): `node`/`python` types map to `node`/`python <program> <args>` (honouring `runtimeExecutable`/`runtimeArgs`). On **Windows** an extensionless build-tool wrapper used as `runtimeExecutable` is rewritten to its Windows variant (`winExe`: `mvnw`→`mvnw.cmd`, `gradlew`→`gradlew.bat`) — PowerShell can't run the bare Unix script inline, so without this Windows shell-executes it by file association into an **external** window instead of the in-app terminal (the Spring-Boot-via-`mvnw` case). A `godot` type (which carries `project`/`scene` instead of a `program`) maps to `godot --path <project> [<scene>]` — the engine binary defaults to `godot` on PATH (overridable via `runtimeExecutable`/`editor_path`, since the Godot Tools extension keeps it in a VS Code setting, not launch.json), and a `scene` of `main`/`current`/absent runs the project's main scene while an explicit `res://` path runs that scene; other interpreter types in `TYPE_RUNTIME` (`go`→`go run`, `php`, `ruby`, `perl`, `lua`, …) map to their runtime when a `program` is present; anything with a `runtimeExecutable` or bare `program` falls back to running that. A **browser** type (`BROWSER_TYPES`: `chrome`/`msedge`/`edge`/`firefox` and their `pwa-` variants) carries no `program` — its intent is to open a `url` (or local `file`), so it maps to the OS opener (`OPEN_CMD`: `Start-Process` on Windows, `open` on macOS, `xdg-open` on Linux) for the default browser, honouring an explicit `runtimeExecutable`/`runtimeArgs` (a specific browser binary) when present; the dev server it points at is expected to be started separately (a compound/preLaunchTask), exactly as in VS Code. A config with none of these (e.g. an attach config, or a browser type with no `url`/`file`) can't be translated, so `run-config` returns `{ok:false, error}` naming the config and its type, and the renderer shows it in the `#run-error-dialog` modal rather than failing silently in the dev console.
- **Task** (`buildTaskCommand`): the `command` (verbatim for `shell` tasks, which may be a whole command line) followed by its quoted `args` (an `{value, quoting}` arg with `"strong"`/`"weak"` quoting is always quoted). **Contributed task types** are expanded to their tool's CLI: `npm` → `npm run <script>` (its `path` becomes the spec's cwd), `typescript` → `npx tsc [-p <tsconfig>] [--watch]`, and `gulp`/`grunt`/`jake` → `npx <tool> <task>`. A task's `options.shell` (`{executable, args}`) wraps the built line in that shell (`wrapShell`; without explicit args the shell's run-one-command flag is inferred — `-Command` for PowerShell, `/d /c` for cmd, `-c` otherwise). tasks.json's **global scope** (top-level `options` + platform blocks) is folded onto every task by `normalizeTasks` (task values win; env/cwd shallow-merged), which both IPC handlers use. **Compound tasks** are supported via `resolveTask`: a task's `dependsOn` (a label or array of labels) is resolved recursively (cycles broken by a `seen` set) and combined per `dependsOrder` — `"sequence"` collapses the steps into **one** terminal whose commands are chained so each runs only if the previous succeeded (bash/zsh `&&`; Windows PowerShell 5.1 has no `&&`, so `chainCommands` gates each later step on the `$?` success variable instead, and `stepCommand` prepends a `cd`/`Set-Location` for any step with its own `options.cwd`), while `"parallel"`/`"any"` (the default) yields **one terminal per** referenced task. A compound may also carry its own `command`, which runs after its dependencies. A sequence shares a single terminal, so the steps' `options.env` are merged onto it (last wins) rather than applied per step.

**Platform overrides** (`mergePlatform`): a task or launch config may carry a per-OS override block — `windows` on win32, `osx` on macOS, `linux` elsewhere — whose fields (`command`, `args`, `options`, `program`, `runtimeExecutable`, `env`, …) win over the base per property (`options` is shallow-merged so overriding only `env` keeps the base `cwd`). This is folded onto the base **before** any translation, so every builder (`buildTaskCommand`, `buildLaunchCommand`, `resolveTask`'s leaf `options`, `launchSpec`) sees resolved fields. It's what lets one entry run `cmd.exe /c foo.bat` on Windows but `bash foo.sh` on macOS/Linux (the co-op-LAN-test pattern) — without it Windows would try the base `bash …sh` and fail when no Unix bash is on PATH.

`substVars()` resolves the VS Code variables: the workspace family (`${workspaceFolder}`, `${workspaceFolderBasename}`, `${workspaceRoot}`, `${cwd}`, `${pathSeparator}`/`${/}`), `${env:NAME}`, and — via the `ctx` argument to `makeRunConfigLib(repo, platform, ctx)` that `run-configs.js` fills in — `${userHome}` (os.homedir), `${config:dotted.key}` (`.vscode/settings.json`), `${defaultBuildTask}` (the `group: "build"` task with `isDefault`, via the pure `defaultBuildTaskName`), `${input:id}` (collected answers, see below), and the `${file}` family (`${file}`, `${fileBasename}`, `${fileBasenameNoExtension}`, `${fileExtname}`, `${fileDirname}`, `${fileDirnameBasename}`, `${relativeFile}`, `${relativeFileDirname}`, `${fileWorkspaceFolder}`) from the renderer's active editor file, which the toolbar passes as `activeFile` (exported by `viewer/file.js`'s `getActiveFile()`). Other `${…}` placeholders (e.g. `${command:…}`, or `${file}` with no open file) are left as-is (best effort).

**Inputs.** When a resolved spec still contains `${input:id}` (the pure `findInputIds` scans command/cwd/env), `run-config` looks the ids up in the file's `inputs` array and returns `{ok:false, needsInputs:[{id,type,description,default,options}]}` instead of running; the toolbar collects answers (`promptString` via the shared `promptText` modal, `pickString` via a small option-list modal in `toolbar.js`) and re-invokes `run-config` with the `{id: value}` map (a `command`-type input is an extension command we can't run, so it errors). **preLaunchTask.** A launch config's `preLaunchTask` (a task label or `${defaultBuildTask}`) is resolved through `resolveTask` and chained in front of the launch command in **one** terminal via the pure `prependTasks` (same `&&`/`$?` chaining and cd-prefixing as sequence compounds), so the launch waits for the task to succeed; a compound-level `preLaunchTask` runs as its own spec before the members (best effort). `postDebugTask`, `problemMatcher`, `isBackground`, and the rest of `presentation` have no runtime meaning without a debugger/terminal UI and are ignored. `cwd`/`env` come from the launch config or the task's `options`. A launch config's **`envFile`** (a `.env` path, e.g. Spring Boot's `dev.env`) is read in `run-configs.js` (file IO, kept out of the pure lib) and parsed by `parseEnvFile` (Electron-free, unit-tested: `KEY=VALUE` lines, `#` comments and blanks skipped, `export ` prefix and matching surrounding quotes stripped); those vars seed the spec's env with the config's inline `env` taking precedence, and a missing file is ignored (the config still launches).

The renderer's `runSpecInConsole(spec)` opens (or **reuses** — matched by `spec.kind` + name) a terminal tab for each spec, spawning its shell in the spec's `cwd`/`env` and running its `command`; relaunching restarts its existing tab with a fresh shell rather than piling up duplicates. `kind` is `'config'` for a `.vscode` run (the default) and `'run'` for an editor "run this file" (see [Run a file](#run-a-file)), so the two never collide on a tab. The renderer rebuilds the toolbar (`loadToolbar()`) on startup, after **Open folder**, and whenever the `.vscode` files change underneath it: `run-configs.js` keeps an `fs.watchFile` (2s poll) on `.vscode/launch.json` and `.vscode/tasks.json` — robust even before the files exist, firing on create/edit/delete — and pushes a `run-configs-changed` event that the renderer answers with `loadToolbar()`. The watch re-points to the new folder on **Open folder** via `repo.js`'s `onRepoChange` listener registry (fired from `setRepoPath`).

**Running indicator.** A launch button's icon reflects whether the config is running, where *running* means **the terminal it started is still alive** (the program inside is typed into a long-lived interactive shell, so there's no process exit to read — the shell, hence the tab, is the liveness signal). Idle shows the play arrow (▶); while live it shows a bold restart glyph (a heavy-stroke inline SVG circular arrow — `RESTART_SVG` in `toolbar.js` — tinted yellow via `.tool-btn.launch.running`), and clicking it restarts the config (`runSpecInConsole` already reuses + restarts a same-named tab). This is **event-driven**: `consoles.js` fires an `onConsolesChanged` listener registry on every console open/close (`createConsole`/`closeConsole`, the latter reached by `term-exit` when a shell dies or the user closes the tab), and `toolbar.js` answers by re-running `refreshRunStates()`, which reads the live config-terminal names from `runningConfigNames()` (consoles of `kind: 'config'`). A 10s `setInterval` re-runs the same refresh as a safety net. A compound launch button is "running" when **any** of its referenced configs' terminals is alive — `get-run-configs` returns each compound's member names via the pure, unit-tested `compoundMembers()`, since a compound's terminal tabs are named after the *referenced* configs, not the compound itself.

**Stop button.** Each launch config is rendered as a `.launch-group` pairing a small red **Stop** rectangle (`.launch-stop`) to the **left** of the launch button. Its slot is always reserved (fixed width even when hidden) so it can animate in/out (opacity + scale) without shifting the launch button or anything after it; it's `visible` only while the config is running (toggled by the same `refreshRunStates()` that flips the icon). Clicking it calls `consoles.js`'s `stopConfig(name)`, which `closeConsole`s every still-open `kind: 'config'` terminal of that name — killing the shell is what ends the running state, so the icon flips back to ▶ and the Stop button hides again via the `onConsolesChanged` refresh. A compound's Stop calls `stopConfig` for each of its member names.

## Run a file

The file editor's header carries a **Run split-button** (`#run-split`: a square `#run-btn` Play button joined to a narrower `#run-menu-btn` caret) shown only while a **runnable** source file is mounted. It runs a single source file with a language interpreter — no `.vscode` config needed — reusing the same terminal-tab plumbing as the run toolbar (`runSpecInConsole`, `kind: 'run'`).

The pure language table lives in `runners-lib.js` (Electron-free, unit-tested): `LANGUAGES` maps file extensions to a language (`python`, `node`, `typescript`, `ruby`, `go`, `php`, `perl`, `lua`, `shell`, `powershell`), each carrying the ordered **candidate interpreter binaries** to look for (`python3`/`python`/`py`, …) and a `build(bin, file)` that yields the command parts (most are `[bin, file]`; `go` → `go run`, `powershell` → `-File`, `deno` → `deno run`). `langForFile()` resolves a file's language; `buildRunCommand(lang, bin, file, args)` quotes the parts (shared `quoteArg`) and appends any extra args. Add a language by adding a `LANGUAGES` entry (and an `EXT_LANG` mapping for highlighting).

`runners.js` owns the OS side and IPC. It **probes PATH without executing anything** (`existsOnPath`: walks `PATH`, honouring `PATHEXT` on Windows so a bare `python` matches `python.exe`/`.cmd`/`.bat`) and keeps a **registered-interpreter store** — `interpreters.json` in `instance.js`'s `sharedDataDir` (machine-wide, survives restarts, like `recent-folders.json`), a `langId → absolute exe path` map. `resolveBinary(lang)` prefers a registered path that still exists, else the first candidate found on PATH. IPC: `get-runner-langs` (the languages + each one's registered interpreter, so the renderer knows which extensions show the button), `resolve-runner({file, args})` (→ `{ok, run:{command, cwd, name, kind:'run'}}`, or `{unsupported}` for a non-runnable type, or `{needsInterpreter, lang, bins}` when no interpreter is found), `pick-interpreter({langId})` (a native `showOpenDialog` to register an executable — the "we couldn't find the binaries" recovery path — persisted), and `clear-interpreter({langId})` (forget a registration, falling back to PATH detection).

The renderer's `viewer/run.js` drives the button. `showRunFor(file)` (called by `file.js` after a runnable file mounts) reveals it; `code-render.js` hides `#run-split` on every view show/hide, so switching to a git diff or a binary notice drops it. The **Play** button calls `resolve-runner` and either runs the spec (`runSpecInConsole`) or, on `needsInterpreter`, opens a confirm dialog offering to **Locate…** the interpreter (`pick-interpreter`) then retries. The **caret** opens a small popover (same chrome as the shell-picker menu) showing the current interpreter (registered path, or "auto-detected from PATH") plus **Run with arguments…** (prompts, then runs with the extra args), **Select interpreter…** (register/override), and, when one is registered, **Reset to auto-detected**.

## Resizable panes

Four drag-gutters (`.gutter` divs in `index.html`) let the user resize, **not reorder**, the panes. The column layout is a CSS grid whose left/right tracks are the vars `--left`/`--right` on `#app` (`grid-template-columns: var(--left) 5px minmax(0,1fr) 5px var(--right)`); the center is the remaining `1fr`. The sidebar split is the var `--sess-h` on `#sidebar` (`#sessions-pane`'s flex-basis; `#files-pane` takes the rest). The git pane is itself split vertically: `#git-main` (status/commit) fills the top and `#git-console` takes a `--console-h` flex-basis at the bottom (default `33%`); see [Git-pane console](#git-pane-console).

`panes.js`'s `resizer(gutter, axis, sign, read, write, min, max)` is one generic pointer-drag handler wired four times — left column, right column, sidebar split, git-pane console split. It pointer-captures the gutter, then on move clamps `base + sign*delta` to `[min, max()]` and writes the var. Clamps keep every pane within bounds: columns can't shrink the center below `CENTER_MIN` (200px) or themselves below their mins; the sessions pane stays ≥80px and leaves ≥140px for the explorer; the console stays ≥80px and leaves ≥160px for the git content. `sign` is `-1` for the right and console gutters (their pane is on the far side, so dragging toward it shrinks it). Each move calls `fit()` so the active terminal reflows live, plus `fitConsole()` for the console terminal. Sizes are session-only — not persisted across restarts.

## Git-pane consoles

The bottom of the git pane (`#git-console`) holds a **multi-terminal panel**: a tab strip (`#term-tabs`) over a stack of terminal hosts (`#console-hosts`), each tab owning its own xterm.js `Terminal` + shell PTY. These are independent of the per-session Claude PTYs. In the main process the PTYs live in a `consoles` `Map<id, pty>` (`spawnConsole(id, opts)`); IPC is keyed by id — `term-create` (returns `{id}`), `term-restart` (kill + respawn under the **same** id), `term-input`, `term-resize`, `term-kill`, and the `term-data`/`term-exit` events (both carry `{id}`). `term-shells` lists the shells the user can pick (`availableShells()`: `ps`/`cmd` on Windows, `$SHELL`'s basename otherwise). A terminal can run a command two ways. The run toolbar **types** it — `spawnConsole` writes `command + '\r'` into the shell, deferred until the shell's **first output** (its prompt): writing the instant the PTY spawns races shell startup — zsh in particular hasn't initialised its line editor (ZLE) until it has sourced its rc files, so an early write lands the text but drops the submitting `\r`, leaving the command typed but never run; a timeout backstops a shell that emits nothing. The [setup wizard](#claude-code-setup-gate)'s embedded terminal instead passes `args` — the shell **argv** (`zsh -ilc '<cmd>'`) — so the command runs as a shell argument with no prompt-typing and no ZLE race at all (the more robust path; the typed path remains for the run toolbar, which reuses one interactive shell across runs).

In `consoles.js` each tab is a record in the `consoles` `Map` (`{term, fit, host, tab, name, kind}`). `createConsole(opts)` builds the xterm/host/tab and calls `term-create`; `selectConsole(id)` toggles which host is visible (and the active tab) and focuses it; `closeConsole(id)` kills the PTY, disposes the term, drops the tab, and selects a sibling — recreating a default shell if none remain, so a terminal is always open. `term-exit` (e.g. the user typed `exit`) routes to `closeConsole`. Both `closeConsole` and `resetConsoles` tear a terminal down through the shared `disposeConsole(id, c)` helper. A PTY's cwd is fixed at spawn (`getRepoPath()` in `spawnConsole`) and never follows a folder switch, so `applyRepoChange` calls **`resetConsoles()`**: it disposes every terminal and opens one fresh shell, which spawns in the now-current repo. Reused id is what lets `term-restart` swap in a fresh shell while `term-data` keeps routing to the same tab; the matching guard in `spawnConsole`'s `onExit` keeps a replaced PTY from reporting its tab as closed.

**A broken terminal must never crash the IDE.** Like the session PTYs, every console PTY operation that can throw when the shell dies under us is contained: `spawnConsole` wraps `pty.spawn` (a bad shell path / cwd / winpty hiccup reports the tab as exited — the renderer reopens a default shell — instead of throwing), and every `p.write` is guarded — the `term-input` handler swallows a write into a closed PTY, and the deferred command write (`sendCommand`, fired from a `setTimeout`/`onData` callback where a throw would have no caller to catch it and would surface as an uncaught main-process exception) is wrapped too. `resize`/`kill` were already guarded for the close race. So a single hung or dead terminal degrades to just that tab closing, never the whole app.

Tabs are **named**: a manually-created one (the **+** button, `#term-add` — a shell-picker menu when more than one shell is available) takes its shell's name (`cmd`/`ps`); a toolbar-launched one takes the config/task name. Labels are truncated (`truncName`, full name in the tab `title`). The header also carries a **Clear** button (`#term-clear`) that calls `term.clear()` on the active terminal (renderer-only; doesn't touch the shell).

## Terminal links & inline browser

Ctrl+click (Cmd on mac) in a terminal opens what's under the cursor — a web URL in an inline browser, a file path in the explorer's viewer (VS Code's gesture). `registerTerminalLinks(term)` wires each session's xterm via `registerLinkProvider`. The provider **always provides links** (decorated only while the modifier is held) rather than gating `provideLinks` on the key state — xterm only re-queries the provider on mouse movement, so gating there would force the user to press Ctrl *before* hovering and keep the terminal focused. Instead a window-level **capture-phase** `keydown`/`keyup` pair tracks `linkModDown` (capture so a focused panel that stops propagation can't hide the key), the link's `hover`/`leave` callbacks record the currently hovered link, and on any modifier change we flip that link's **live `decorations.underline`/`pointerCursor` setters** (xterm swaps these in once a link is hovered) — so a link lights up the instant Ctrl is held over it, regardless of focus or which happened first. Activation is gated on the modifier (`event.ctrlKey`/`metaKey`) so plain clicks pass through and normal hover/drag-to-select are untouched. `findTerminalLinks(lineText)` scans a row for `http(s)://` URLs first, then path-ish tokens (`PATH_RE`), claiming character ranges so the two never overlap. A path token only counts if it has a separator or a known extension (`looksLikePath` against `PATH_EXT`, built from `EXT_LANG`/`IMG_EXT`/`AUDIO_EXT`); a trailing `:line[:col]` is split off on activation and passed to the viewer's jump.

`openTerminalLink` routes by kind: a URL → `showWeb()`; a path → `resolve-link-path` (main resolves an absolute path as-is, else against `baseDir` — the *originating* terminal's own directory, not necessarily whichever folder is currently open — falling back to the open folder when none is given, and reports `isFile`/`isDir`/`inRepo`/`rel`), then a directory opens in the OS file browser, an in-repo file opens via `openFromTree(rel, {line})`, and anything else goes to the OS through `open-external`. `registerTerminalLinks(term, baseDir)` takes that directory per-terminal — a session passes its own `repo` (which can differ from the open folder if the session was created under a since-switched-away project) and a git-pane console passes its `cwd` (a launch config or task can run outside the repo root); it's threaded through to `resolveLinkPath` on every click rather than read from a single global, since two terminals open at once can each need a different base. `showWeb()` is a center overlay (`#web-view`, peer of the diff/asset overlays, hidden by the shared `closeOverlay()` and on session select) holding a back/forward/reload bar, an **editable address bar**, an open-externally button, a **⋮ menu**, a **Terminate** button, and an Electron **`<webview>`** — a separate guest process, so the host-page CSP doesn't restrict the loaded site. It requires `webviewTag: true` in the window's `webPreferences`.

**One persistent instance.** There is exactly **one** `<webview>`, reused for every open — the browser is never re-instantiated. Besides Ctrl+click, it's opened from the **browser button** in the top toolbar (`#browser-btn`, left of the settings gear, both pinned to the toolbar's right edge), which calls the center coordinator's `openWeb()` → `web.js`'s `openWeb()`: it just reveals the overlay over the *already-loaded* page (focusing the address bar only when nothing is loaded yet). Crucially, hiding the overlay (session switch, `closeOverlay()`, opening another center view) **no longer unloads the page** — `hideWeb()` only sets `display:none`, so the site keeps running in the background and is shown again as-is on the next open. The toolbar button carries an **`.active`** accent tint whenever a real (non-`about:blank`) page is loaded, so it doubles as a browser-is-alive indicator; `web.js` toggles it from the `did-navigate(-in-page)` events and on terminate. The only thing that tears the page down is the **Terminate** button (`#web-terminate`, ⏻): a destructive action, so it uses the same **two-click arm** confirm as the git pane's discard/revert (armed red on the first click, kills the browser on the second). Navigating to `about:blank` would leave the webview's guest process alive in the background (media still playing, timers still firing), so `terminateWeb()` instead **destroys the live `<webview>` element** — removing it from the DOM kills its renderer process — and swaps in a fresh blank one (same id/partition, listeners re-attached via `createFrame()`) for the next open, clearing the active tint. It then returns to the active session via `closeOverlay()`. The terminate/close buttons and the browser button are all wired in the center coordinator (`center.js`), which owns the overlay hand-off.

**Inspect element** (`#web-inspect`, a Lucide pipette/color-picker icon left of the open-externally button) lets the user pick any element in the loaded page and copies a full report of it to the clipboard: page URL, XPath, CSS selector, tag/id/classes, pixel size, the key **computed styles** (margins, padding, border, colors, font, layout) and the element's **outerHTML** (trimmed past ~1.5 KB). The picker must run *inside* the guest process, so `web.js` injects it with `webFrame.executeJavaScript('(' + guestInspector.toString() + ')()', true)`: the injected IIFE draws a fixed-position highlight box + selector label that follow the cursor (capture-phase `mousemove`) and returns a **Promise that stays pending until the user clicks an element** — `executeJavaScript` then resolves the host-side `await` with the picked element's (structured-cloneable) data object, which `formatInspect()` turns into the clipboard text (written via `window.api.clipboardWrite`, reliable under `file://`). The button tints accent (`.armed`) while picking and flashes green (`.copied`) after a copy; Escape, re-clicking the button, or closing the overlay (`hideWeb()`) cancel the pick through a `window.__ideInspectCancel` the guest parks for the host to call (`stopInspect()`).

The webview uses a **persistent partition** (`partition="persist:browser"`), so its cookies and site data live on disk under the instance's `userData` and survive restarts (the `persist:` prefix is what makes a partition on-disk rather than in-memory). The **⋮ menu** is the "clean cookies control": its single **Reset cookies** item confirms (danger `confirmDialog`) then calls `clear-web-data` → `session.fromPartition('persist:browser').clearStorageData({ storages: ['cookies'] })` in main, wiping just that partition's cookies and reloading the page; the app's own session is untouched.

The **address bar** (`#web-url`, an `<input>`) is editable: Enter navigates (`normalizeUrl` keeps a real scheme, else prepends `https://`). It offers **suggestions** from a most-recent-first visited-URL list (`web.history` in `localStorage`, capped at 30): focusing the empty bar shows the latest addresses, typing substring-filters them; a suggestion is chosen on `mousedown` (before the input's blur hides the list). `did-navigate`/`did-navigate-in-page` keep the bar in sync (without clobbering mid-typing) and push each address into the history. The ↗ button hands the current URL to `open-external` (→ `shell.openExternal`).

## Error handling & crash logs

Two layers keep a failure from taking down the app.

**Session errors are non-fatal and visible.** Anything that can go wrong around a session — saving/loading the on-disk snapshot, creating/resuming/closing a PTY, committing/reverting/diffing, or tracking hook activity — must never crash the IDE or vanish into the console. In `sessions.js`, `reportSessionError(context, err)` logs the error and pushes a `session-error` event (`{ context, message }`) to the renderer; errors raised before the renderer exists (e.g. a corrupt session file read during the startup load) are **queued** and flushed when the renderer first calls `get-sessions`. Every session IPC handler is wrapped by `guard(context, fn, fallback)` (invoke handlers) or `guardOn(context, fn)` (fire-and-forget `.on` handlers): a throw becomes a reported warning plus a safe return value (`fallback`), instead of a rejected invoke (an unhandled renderer rejection) or an uncaught main-process exception. `persistSession()`/`persistSessions()` are wrapped whole (they run from a timer and on quit, where a throw would be uncaught), `loadPersistedSessions()` distinguishes a missing store (normal first run) from a corrupt file (reported) and never lets one bad file abort the rest, and the hook server reports a failed `recordSessionActivity()` (but still silently ignores a malformed payload). The renderer listens for `session-error` and shows the shared **warning dialog** — `renderer/shared/warn.js`'s `showWarning(message, title)`, the `#warn-dialog` element: the error text in **red** (`#warn-msg`), an **OK** to dismiss and a **Copy** that writes the full text via `clipboardWrite`. It reuses the error/confirm dialog chrome and is safe to call repeatedly (a new message replaces the open one rather than stacking). All its strings are i18n'd (`warn.*`).

**Real crashes are logged.** `crashlog.js`'s `installCrashLogging()` (wired first in `index.js`) records every `uncaughtException`, `unhandledRejection`, and `render-process-gone`/`child-process-gone` to its own timestamped file under a **`crashlogs/`** folder — next to the executable when packaged, else the project root in dev. It does **not** exit (the app stays usable; `window.js` already reloads a dead renderer); the log is a forensic trail. The filename + body formatting is the pure, unit-tested `crashlog-lib.js` (`crashLogName` makes a sortable, filesystem-safe `crash-<iso>.log`; `formatCrash` writes the kind + time + stack). Writes are synchronous so a log survives even a fatal exception. `crashlogs/` is gitignored.

## preload/

The entire IPC surface (`contextBridge.exposeInMainWorld('api', …)`), split into per-subsystem modules that `preload/index.js` composes. Add a channel to the matching module when wiring any new main↔renderer call. `resolve-link-path` and `open-external` back the terminal link feature above.

## Files

Vanilla JS, no bundler, no test framework. The code is split into per-subsystem modules, not three monolithic files; `index.html` + `styles.css` are the only non-JS UI assets. Within `src/`:

- **`main/`** — the main process, one module per concern, each registering its own `ipcMain` handlers as a load side-effect (`index.js` just requires them in order):
  - `index.js` — app bootstrap: command-line switches, require order, `app` lifecycle.
  - `crashlog.js` + `crashlog-lib.js` — crash logging to `crashlogs/` (see [Error handling & crash logs](#error-handling--crash-logs)). `crashlog-lib.js` is the Electron-free filename/body formatting, unit-tested.
  - `instance.js` — per-instance `userData` profile redirect + startup sweep of stale leftover dirs (see [platform-notes.md](platform-notes.md)); must load first. Also mints this window's `instanceId` + `startedAt`, its identity on the wire to a phone. Pure sweep/roster logic in `instance-lib.js` (`staleInstanceDirs`, `liveInstances`, `upsertInstance`).
  - `instance-registry.js` — the machine-wide list of running windows (`remote-instances.json` in `sharedDataDir`), so a phone can see every window and pick which to drive. See [remote-access.md](remote-access.md) — "A room is a machine, not a window".
  - `window.js` — the `BrowserWindow` + window title. Owns `sendToRenderer(channel, payload)` — the one path every main→renderer push must use: it no-ops when the renderer is gone/crashed and swallows the dispose race, so a dead renderer can't turn a `webContents.send` into an unhandled main-process crash. On `render-process-gone` it reloads the window (throttled) so the UI recovers instead of freezing.
