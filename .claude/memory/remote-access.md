# Remote access (mobile companion)

Developers can drive the IDE from a phone: QR pairing, project switching, Claude sessions, git, file editing, terminals, and a **remote browser** (a page rendered on the desktop, streamed to the phone — see **Remote browser**). Dev-server port forwarding exists but is **parked** — see **Port forwarding is parked**. Three parts: the Electron-free `server/` socket bridge, thin glue in `src/main/remote*.js`, and the Expo app in `mobile/`. Design rule (user requirement): **the backend only bridges sockets/communications** — all IDE logic and auth verification live in the desktop app; `server/` never inspects business payloads.

**One transport: the cloud relay.** A phone reaches the desktop only through the relay (`https://ide-yj3x.onrender.com` in a build, `http://localhost:8080` in a dev run). There is no LAN ws server and no "try the LAN first, fall back to the relay" — that dual-transport was removed. A build uses the hosted relay; a dev run uses the relay on your machine (see **Which relay a run talks to**).

## server/ — the socket bridge (plain Node, CommonJS, linted like src/main)

- `protocol.js` — pure: message parse/validate, `REMOTE_CHANNELS` **allowlist** (`req` mirrors `ipcMain.handle`, `send` mirrors `ipcMain.on`), `REMOTE_EVENTS` (renderer pushes forwarded to phones), `fwd-open`/`fwd-close` shapes. To expose a channel remotely you must add it here — an existing handler is not enough. Excluded on purpose: `open-folder` (native dialog), clipboard, `db-*`, runners, assets. **`STREAM_EVENTS` (`pty-data`/`term-data`/`transcript-data`) are subscription-only**: they are the relay socket's bandwidth hogs (a Claude TUI repaints constantly), and broadcasting them to every phone made small pushes like a chat's `transcript-data` queue behind every other session's terminal bytes. A `{t:'watch', ch, id, on}` frame opts a client in per id; the hub sends a stream event only to clients watching that payload's `id`. A client that never sends `watch` (an app predating this) stays unfiltered and receives everything, so old phones keep working.
- `auth-lib.js` — pure: single-use 5-min pairing tokens (3 wrong guesses invalidates), device credentials (32-byte tokens, only sha256 hashes persisted), injected-store CRUD.
- `http-proxy.js` + `http-proxy-lib.js` — dev-server port forwarding: one listener per forwarded port piping to `127.0.0.1:<target>`; auth = one-time `?_ideauth=` URL token → 302 that sets an HttpOnly cookie (pure decisions in the lib); raw-socket Upgrade passthrough keeps Vite/webpack HMR websockets working. A tunnelled browser is piped into this same listener; the listener never knows it is being tunnelled, which is why the tunnel moves bytes and not HTTP. What it forwards is a **site, not a page** — see **A forward is the whole site**.
- `hub.js` — the client state machine with **no socket under it**: pairing, hello → pair/auth → ready, the channel allowlist, the authed-client set and the broadcast fan-out. `connect(send)` returns a client you feed raw frames; `relay-client.js` feeds it and the hub tests drive it directly. One `broadcast()` reaches every phone in the room.
- `relay.js` + `relay-route.js` + `relay-frames.js` + `index.js` + its own `package.json` — standalone cloud relay (deploy: root dir `server`, build `npm install`, start `node index.js`, listens on `process.env.PORT`; live at `https://ide-yj3x.onrender.com`). Frame forwarding between the desktop sockets and the mobile sockets of a room using `{c:<clientId>, d:<frame>}` envelopes; never parses `d`. A room holds **many** desktops, keyed by instance id — see **A room is a machine, not a window**. It also carries the phone's *browser* to a forwarded dev server — see **Port forwarding over the relay**.
- `relay-client.js` — the desktop **dialling out** to the relay, which is the point: a desktop behind NAT can't be dialled, but it can hold one outbound socket, and every phone bound to that window rides it. `?role=desktop&room=<id>&instance=<id>`. Demuxes `{c,…}` into hub clients and `{h,…}` into byte streams. Reconnects with backoff; a dead relay socket drops all its clients so the hub releases whatever they held.
- `debug.js` — the backend's operation trace, off unless `IDE_DEBUG` is set (see **Debug tracing**).
- `keepalive.js` — keeps the relay from being idled out by free hosting. `resolveKeepAlive(env)` reads Render's `RENDER_EXTERNAL_URL` (or `RENDER_EXTERNAL_HOSTNAME`) to learn its own public URL — no URL configured anywhere by hand — and `startSelfPing` GETs `<url>/healthz` every 10 min (Render's idle window is ~15). Off entirely when those env vars are absent (LAN/dev), so only the deployed relay pings. Overrides: `KEEPALIVE_URL`, `KEEPALIVE_INTERVAL_MS` (`0` disables). The ping must be a *real inbound HTTP request to the public URL* — hitting `127.0.0.1` would not reset the host's idle timer.

## Debug tracing (`IDE_DEBUG`)

Nothing in `server/` prints on a normal run. Set `IDE_DEBUG` and `debug.js` traces **every operation the backend performs** — one line each, `12:30:45.123 [hub] req ok ch=git-status ms=12`:

| Namespace | Traces |
|---|---|
| `relay` | the cloud relay's sockets: desktop/mobile joins, replaces and evictions, room lifecycle, every frame routed in either direction (`t`/`ch`/`id` only — `d` stays opaque, as the relay's whole design says) |
| `route` | its per-connection head routing decision (`ws` / `entry` / `tunnel` / `health` / `deny`) |
| `tunnel` | forwarded-port byte streams on both sides — **a line per chunk**, so this is the loud one |
| `client` | `relay-client.js`: dialing, connect, backoff, disconnects, and the frames it demuxes |
| `hub` | the protocol state machine: every frame in, pair/auth outcomes, each `req`/`send` with its allowlist verdict and duration, `fwd-*`, broadcasts and their fan-out size |
| `proxy` | the port-forward listener: auth verdict, upstream status, HMR upgrades |

Value: `1`/`true`/`*` = everything; a comma list picks namespaces (`hub,relay`); a `-` prefix subtracts (`*,-tunnel` — everything but the per-chunk byte traffic, which is what you usually want). The two launch configs set it: **Start backend (relay)** runs `*`, **Start IDE** runs `*,-tunnel` (the desktop embeds `hub.js`/`relay-client.js`/`http-proxy.js`, so their traces land in that terminal).

**Credentials never reach the log.** `redact` blanks any field whose *name* looks like one (`token`, `cookie`, `password`, …) at any depth, caps strings at 120 chars and prints buffers as `<N b>` — the traced frames are exactly the ones carrying pair/device tokens.

## The relay transport

While remote access is on, the desktop holds one outbound socket to the relay (`https://ide-yj3x.onrender.com`, baked into `DEFAULT_RELAY_URL` in `remote-config-lib.js` and `mobile/src/api/config.ts` — keep the two in step). The desktop **dials out** because a machine behind NAT cannot be dialled *in* to; every phone in its room rides that one socket.

The QR carries the relay origin and the room (`ide://pair?v=1&relay=…&room=…&tk=…`). The phone dials that one URL and reconnects to it with backoff (`connection.ts`) — the relay is free-hosted and may cold-start, so the first attempts can fail before it answers.

**Adding a room id to the config was not optional.** A phone holds only the relay URL and cannot rediscover it — so the room is the desktop's address on the relay, and it is persisted, and never silently regenerated (see the room note below). It is minted on first run and **written to disk before the first QR is shown**, not just held in memory.

## Which relay a run talks to (dev vs build)

**A dev run talks to the relay on your machine; a build talks to the deployed one.** Debugging the relay against Render would be debugging the *deployed* code, not the code you are running, so this is not a setting you remember to flip — `resolveRelayUrl` (`remote-config-lib.js`) decides it:

| | relay |
| --- | --- |
| unpackaged (`!app.isPackaged`) | `DEV_RELAY_URL` = `http://localhost:8080` (server/index.js's own default port) |
| packaged | stored `relayUrl` (self-hosted), else `DEFAULT_RELAY_URL` (Render) |
| `IDE_RELAY_URL` set | that, either way — a staging relay, or a dev run aimed at the deployed one |

Two things this gets right that the obvious version doesn't:

- **The dev URL is resolved, never persisted.** `remote-config.json` lives in `sharedDataDir` and is shared with the *installed* app, so writing `localhost` into it would strand a real build. `remote.js` therefore resolves `relayUrl` at startup from the **raw** stored value and never writes it back; a stored URL is honoured only by a packaged app, so it can't drag a debug session onto a remote relay either.
- **A phone can't dial your `localhost`** — that word names the phone. The desktop keeps dialling the relay at `localhost` (same machine), but everything *handed to a phone* — the QR's `relay=`, a forwarded dev-server link, the status the settings pane shows — goes through `relayUrlForPhone`, which swaps in the desktop's LAN address. So a phone on the network follows a dev desktop to the local relay with nothing to configure.

The mobile app's `DEFAULT_RELAY_URL` (`mobile/src/api/config.ts`) is only a **fallback for a QR that carries no relay** — the desktop's QR always carries one, so an Expo run paired with a dev desktop is already on the local relay. A build always falls back to the hosted URL; an Expo run (`__DEV__`) can be pointed elsewhere with `EXPO_PUBLIC_RELAY_URL` (a LAN address — a phone cannot dial your machine's localhost).

## A room is a machine, not a window

The IDE is deliberately multi-instance, and `remote-config.json` lives in `sharedDataDir` — so every window shares one room. That made siblings *collide*: the relay allowed a single desktop socket per room and closed the previous one with `4001 replaced`, so opening the IDE a second time silently knocked the first off the relay.

**The roster is per-relay, not per-machine.** `remote-instances.json` is machine-wide, but a dev run dials the local relay while an installed build dials the hosted one — two different relays, so two different rooms, and a phone can only reach windows on *its* relay. Each entry therefore records the `relay` its window dials, and `list-instances` filters to entries matching the serving window's own relay (entries without the field — pre-relay-only builds — are excluded). Without this, running a dev window beside the installed app offered the phone windows it could never dial: picking one hung, and the project you opened in it never appeared on the phone.

So a window is addressed **inside** the room by an `instanceId` (a uuid per process, `instance.js`, alongside `startedAt`). The relay keys `room.desktops` by it: siblings coexist, a reconnecting window replaces only *its own* stale socket, and closing one window drops only the phones bound to it. Pairing stays **per machine** — `remote-devices.json` is machine-wide, so one QR gets a phone into any window, and there is nothing to re-scan.

**Which window a phone drives is chosen after connecting, not before.** The roster can't be a relay-level thing: the relay is deliberately logic-free, and it must never learn what project a window has open. So `instance-registry.js` has every remote-enabled window keep an entry (`{id, pid, startedAt, project}`) in `remote-instances.json` in `sharedDataDir` — the only thing separate processes share, there being no IPC between them — and the authed `list-instances` channel serves the live ones, **oldest first**. Whichever window the phone's socket lands on answers for its siblings out of that file. A window killed or crashed never removes its entry, so the reader prunes by pid liveness (`liveInstances` in the pure `instance-lib.js`), exactly as the stale-dir sweep does.

The phone's **first** dial can't name a window — the roster arrives *over* the connection, so before it has one there is nothing to name. It gets the room's newest desktop (`r.latest`) and lists from there; picking a specific window re-dials with `&instance=` on the relay URL.

**When the desktop shuts down, the phone is told, not left guessing.** The relay evicts a dead window's phones with close code `4002 desktop gone` (`relay.js`). The mobile `Connection` surfaces that code via `onDesktopGone` (`connection.ts`); `App.tsx` then keeps the pairing, resets `chosen`/`instance` so navigation falls back to the Welcome waiting screen, shows a `showNotice` dialog (`ErrorDialog.tsx` — same card as errors, plain text, no Copy), and re-dials the **machine-level** endpoint: an `&instance=` id died with the process, so keeping it would wait on a window that can never return. On the relay side, a phone that dials an *empty* room falls back to `DEFAULT_INSTANCE`, and a restarted desktop mints a fresh uuid — so an arriving desktop **adopts** DEFAULT-bound waiters (rebinds them and announces `clientJoined`); a phone that *named* a window is never adopted by a sibling. The phone doesn't rely on adoption alone: a dial that reaches the relay but isn't `ready` within `READY_TIMEOUT_MS` (10s — the room has no desktop, so the `hello` never comes) is torn down and re-dialled every `REDIAL_MS` (3s); each fresh dial binds to the room's newest desktop, so a reopened desktop is picked up within seconds even against an old relay. Exponential backoff applies only when the dial itself never opened (relay cold-start, no network).

## Port forwarding is parked (not removed)

The whole port-forwarding feature below still exists and still passes its tests, but it is **switched off** while the remote browser covers mobile testing. Two flags, one per side:

- Desktop: `PORT_FORWARDING_ENABLED = false` in `src/main/remote.js` — `enable()` then injects neither `forward` into `createHub` (the hub answers `fwd-open` with `fwd-err 'forwarding-disabled'`, already its no-forward behavior) nor `tunnel` into `startRelayClient` (which now refuses a tunnel `open` with `tunnelClose 'forwarding-disabled'` instead of crashing on a stale `/p/` URL).
- Mobile: `SHOW_PORTS_TAB = false` in `mobile/App.tsx` — the Ports tab is unregistered; `PortsScreen.tsx` and `forwardPort`/`closeForward` in `connection.ts` are untouched.

To re-enable: flip both flags. Nothing else changed.

## Remote browser (the Browser tab)

The phone browses through the **desktop's** renderer instead of forwarding a port to its own browser: an offscreen `BrowserWindow` in the main process (`src/main/remote-browser.js`) renders the page, JPEG frames stream to the phone, and taps/scrolls/typing come back as input events the desktop injects. Near-VPS interactivity over the existing relay socket, with zero relay changes — frames are ordinary `ev` frames riding the `{c,d}` envelope.

- **Channels** (`server/protocol.js`): req `browser-open {url?, width, height, quality?, maxFps?, mode?}` → `{id:'main', url, canGoBack, canGoForward}`; sends `browser-navigate {url}`, `browser-input {events}`, `browser-resize {width,height,quality?}`, `browser-nav {action: back|forward|reload|stop}`, `browser-close`. Events: `browser-frame {id:'main', seq, w, h, b64}` (a **watched STREAM_EVENT** — frames flow only to a phone that watches id `'main'`) and `browser-state {url, title, loading, canGoBack, canGoForward}` (small, broadcast).
- **Rendering is push-based, not polled**: `webPreferences.offscreen` gives a `paint` event only on dirty frames, `setFrameRate` caps it (≤15fps, default 8), and a frame gate (`createFrameGate` in the pure `remote-browser-lib.js`) collapses paint bursts with a trailing-edge send so the last frame always lands. `paint` goes quiet on an idle page — `invalidate()` after `did-stop-loading`/resize forces the first frame. Windows renders offscreen with software compositing; fine at phone sizes. **This is the "software output device" OSR mode (CPU `image.toJPEG()`), and it requires hardware acceleration to be OFF** — so `src/main/index.js` calls `app.disableHardwareAcceleration()` at startup. With HW accel on, creating the offscreen window crashes the GPU process on Windows (crashpad "not connected", the whole app exits); that's the documented requirement, not a workaround. Consequence: the `enable-gpu-rasterization`/`enable-zero-copy` switches are now inert. Re-enabling app-wide GPU would mean moving OSR to `webPreferences.offscreen.useSharedTexture` (GPU shared-texture, `paint` delivers a texture handle not a NativeImage).
- **Input coords are normalized 0..1 on the wire** — only the desktop knows the viewport size, so a resize can never race a tap. `toInputEvents` (pure) maps tap → mouseDown/mouseUp, drag → mouseWheel deltas (content follows the finger), text/key → char / keyDown+keyUp, all via `sendInputEvent`. Mouse emulation, deliberately: Electron touch injection is unreliable, so hover menus and pinch-zoom don't work (accepted).
- **URL hygiene** (`normalizeBrowserUrl`, pure): bare hosts get `https://`, `localhost:3000` is a host:port not a scheme, and `file:`/`javascript:`/`about:`/`chrome:` are refused — only the web is on offer. `setWindowOpenHandler` denies popups and navigates the same view instead.
- **Lifecycle**: one window, id `'main'`, `partition: 'persist:remote-browser'`. Created on `browser-open`, destroyed on `browser-close`, on remote `disable()` (`setBroadcast(null)`), when the authed-client count hits zero (`onClientCount`, wired through `onClientsChanged` in `remote.js`), and on `before-quit`. The phone re-issues `browser-open` on every focus/ready, so a torn-down window heals itself.
- **Mobile ↔ desktop mode**: a toolbar toggle (sticky in SecureStore, `browserMode`) switches the viewport. Mobile mode captures at ≤480 CSS px (`MOBILE_CAPTURE_W`) **with a phone user agent** (`uaForMode` in the lib — set before `loadURL`, since a UA only takes effect on the next navigation) so sites serve their mobile layout; desktop mode captures a 1024-px-wide viewport (`DESKTOP_CAPTURE_W`) with Electron's own desktop UA and letterboxes it. A mode flip re-issues `browser-open` (fresh window at the new size+UA). **Mobile mode is size+UA only — no device-metrics emulation of any kind**: on an *offscreen* window, BOTH `webContents.debugger`/CDP (electron#27768, #14759) AND the public `enableDeviceEmulation` crash the main process instantly (verified empirically on Electron 31/Windows: crashpad "not connected", app exits on the call — this is what killed the desktop when the Browser tab opened, twice, once per API). So meta-viewport/`window.screen`/dPR extras are dropped; the phone UA string (`uaForMode`) plus the phone-sized viewport still get sites to serve their mobile layout. A `render-process-gone` handler logs and tears the window down if the offscreen renderer ever dies, so a crash can't be silent again.
- **The soft keyboard must not resize the capture.** Android resizes the app window when the keyboard opens, so the frame area's layout shrinks — resizing the desktop viewport to that transient shape reflowed the page and broke the aspect on every open/close. `BrowserScreen` derives the capture size only from the last **keyboard-free** layout (`Keyboard` listeners gate `onFrameLayout`; a size equal to the last one sent is never re-sent), and the frame simply letterboxes while the keyboard is up. Because the view's aspect can therefore differ from the frame's, **touch coordinates are normalized against the displayed `contain`-fit image rect** (`toFrameNorm`), not the raw view — taps on the letterbox bars are ignored, scroll positions clamp into the rect.
- **Mobile** (`mobile/src/screens/BrowserScreen.tsx`, the Browser tab): URL bar + back/forward/reload/stop from `browser-state`; frames render as a base64-data-URI `Image` (`fadeDuration 0`, out-of-order frames dropped by `seq`); a `PanResponder` overlay turns a still release into a tap and a drag into batched scroll deltas (50ms flush); a hidden off-screen `TextInput` forwards each soft-keyboard keystroke as it happens (nothing to diff).
- **Bandwidth is the risk to respect**: every frame shares the one relay socket with terminal/transcript streams. The caps (fps ≤15, JPEG quality 20–90 default 55, viewport clamp in `clampViewport`) are what keep a frame from queueing PTY data behind it — don't raise them casually, and keep frames well under the relay's max ws payload.

## Remote desktop control (the Control tab)

The phone drives the **whole desktop machine**, not a page inside it: the selected display streams to the phone as JPEG frames, and touches/keystrokes are injected at OS level so the user can test their own app running anywhere on the desktop. This is the sibling of the remote browser — same watched-frame transport, same normalized-coordinate input — but the picture is the screen and the injection is real mouse/keyboard, not a `BrowserWindow`. Works on macOS and Windows.

- **OS injection is `@nut-tree-fork/nut-js`** — a maintained fork of the abandoned nut-js, N-API so it loads under Electron with **no rebuild** (which `npmRebuild: false` in the builder config requires, the same reason node-pty uses a prebuilt package). Added to `asarUnpack` (`node_modules/@nut-tree-fork/**`) because it ships native `.node` binaries a page inside the asar can't load. The module is **lazily required and degrades**: if it fails to load (missing prebuild) or macOS hasn't granted Accessibility, the screen still streams and `control-open` returns `input:false` with a `warnings[]` the Control tab shows — the feature doesn't die whole.
- **Channels** (`server/protocol.js`): req `control-open {width, height, quality?, maxFps?, display?}` → `{id:'main', platform, screenW, screenH, input, warnings, displays:[{id,label,primary}], display}`; sends `control-input {events}`, `control-close`. `display` selects which monitor to capture — re-issuing `control-open` with a different id is the whole display-switch mechanism (no separate channel). Event `screen-frame {id:'main', seq, w, h, cursor:{cx,cy}|null, b64}` — a **watched STREAM_EVENT** (id `'main'`), so frames flow only to a phone watching it, and its bytes don't queue behind another session's terminal on the one relay socket.
- **Capture is polled, not push-based** (unlike the browser's offscreen `paint`): `desktopCapturer.getSources({types:['screen'], thumbnailSize})` on a `setInterval` at ≤10fps (default 5). A `capturing` flag drops a tick that would stack on a slow `getSources`. **Identical frames are suppressed** (`Buffer.equals`) so a still screen sends nothing — *unless the cursor moved*, since the desktop cursor position (from `screen.getCursorScreenPoint`, normalized to the display in the pure `normalizeCursor`) rides each frame and the phone paints it as an overlay. Screen-recording perms on macOS gate capture; `control-open` warns if not granted.
- **Multi-display**: the switcher list is **source-driven, not display-driven**. `captureTargets()` (async) pairs each `desktopCapturer` screen source with its Electron display (by `source.display_id`, positional fallback) and returns **only capturable displays** — Windows commonly exposes fewer screen sources than `getAllDisplays()` reports (mixed-GPU / off panels), and a display with no source must not appear or selecting it would stream a *different* screen under its coordinates. `selectTarget(id)` points capture + injection at one target (default: primary if capturable, else the first capturable one — never an un-streamable display). `captureFrame` matches the selected `sourceId` **strictly with no wrong-screen fallback**: if the source is momentarily gone it skips the tick rather than stream another monitor. Injection is global across the virtual desktop, so `injectOrigin` (the display's top-left in physical px = `bounds`×`scaleFactor`) is added to every `move`, and `screenPx` (what `toControlOps` normalizes against) is that display's physical size — so streamed pixels and injection space are always the *same* display. The phone shows the switcher chips only when more than one target is capturable. Caveat: mixed per-monitor scale factors are only approximately handled — not hardware-tested across heterogeneous DPI.
- **Input coords are normalized 0..1 of the captured screen** — only the desktop knows real pixels, so a resize can't race a touch. `toControlOps` (pure, in `remote-control-lib.js`) maps one phone item → abstract OS ops: `move`, `tap` (→ move+click, honouring `button`/`double`/`mods`), `down`/`up` (drag), `scroll` (finger-follows-content inversion, applied in the glue via nut's `scrollUp/Down/Left/Right`), `key` (→ a `combo` of held modifiers + key), `text` (typed, but a **single modified char becomes a combo** so Ctrl+C isn't smeared across a string). `remote-control.js` runs the ops through a **serial promise queue** (`queue = queue.then(...)`) — input is order-sensitive, so one op finishes before the next starts.
- **Key names → nut-js `Key` enum** is a candidate list per name (`KEYS` in the lib: `enter`→`['Enter','Return']`, `meta`→`['LeftSuper','LeftWin','LeftCmd']`, letters/digits/F-keys/nav/mods), and the glue picks the first candidate the loaded module actually exports (`resolveKey`) — the enum's spelling varies across versions, so a rename there can't break a key. A combo missing any key **fires nothing** rather than a partial chord.
- **Lifecycle** mirrors the browser: one id `'main'`, capture started on `control-open`, stopped on `control-close`, on remote `disable()` (`setBroadcast(null)`), when the authed-client count hits zero (`onClientCount`, wired through `onClientsChanged` in `remote.js` beside the browser's), and on `before-quit`. The phone re-issues `control-open` on every focus/ready and on a frame-area relayout.
- **Mobile** (`mobile/src/screens/ControlScreen.tsx`, the Control tab, `desktop-outline` icon): the streamed screen with an **absolute** touch model, not a trackpad — a tap clicks where you touched, a single finger dragging moves the cursor there live (throttled `move` sends), two fingers scroll, a long press right-clicks. A **Drag** toggle arms click-and-hold (mouseDown on touch, mouseUp on release) for dragging windows/selections; a **Double-click** button; a horizontally-scrolling **key bar** (arrows, Esc/Tab/Enter/Backspace/Space) with **sticky modifiers** Ctrl/Alt/Shift/Cmd-or-Win that highlight when armed, apply to the next key/tap/text, then clear (a lock isn't offered — one-shot covers combos); and a hidden off-screen `TextInput` (the **Type** toggle) that forwards each soft-keyboard keystroke for filling forms in the app under test. Because the image is `contain`-fit, touch coords are normalized against the **displayed letterboxed rect** (`toNorm`) — touches on the bars are ignored. The modifier badge names Cmd on macOS and Win elsewhere off `control-open`'s `platform`. There is **no title/subtitle header** (dropped to maximize the streamed image); a top **display switcher** row of chips appears only when `displays.length > 1`, and the frame area gets `insets.top` padding since the header no longer supplies it.
- **Bandwidth**: same discipline as the browser — the frame shares the relay socket with terminal/transcript streams. Caps live in the lib (`clampCaptureSize` ≤1440, `clampFps` ≤10, `clampQuality` 20–90 default 50); don't raise them casually.
- **Security**: `control-open`/`control-input`/`control-close` are auth-gated like every channel here — a paired phone already has terminal access to the machine, so full input is the same trust tier, not an escalation. The relay never sees the frames or input as anything but opaque `{c,d}` payloads.

## A forward is the whole site, not the page it opened

Forwarding port 3000 makes the phone a **browser onto that dev server**: `/login`, `/admin`, any path, typed onto the address. Three things make that true, and each one is a thing that would otherwise quietly break it:

- **The proxy is mounted at the root** and the auth cookie is `Path=/`, so the token is spent once and every later path on that origin is already authorised. The address to walk from is therefore the **origin**, never the opened link — that one ends in `?_ideauth=…`, and appending `/login` to it produces nonsense. The Ports tab shows the origin (`base`) under each forward for exactly this reason — here that origin is the relay root, where the `_idetunnel` cookie is what keeps you pointed at this desktop.
- **`Location` is rewritten on the way out** (`rewriteLocation`). The proxy hands the dev server `Host: 127.0.0.1:<target>` because dev servers check it — so a framework answering `/admin` with a redirect built from that Host says `Location: http://127.0.0.1:3000/login`, which would point the *phone's* browser at the phone's own localhost. Reducing it to `/login` lets the browser resolve it against the relay origin it came in on. A localhost URL on a *different* port is left alone: it isn't ours.
- **`Set-Cookie` is made keepable** (`rewriteSetCookie`): `Domain=localhost` names a host the phone is not on, so a session cookie from a login round-trip would be dropped and the login would never stick. Both `Domain` and `Secure` are stripped and the cookie defaults to host-only on the relay origin the browser actually used. `SameSite=None` keeps its `Secure` — browsers reject that pair broken up.

- **`Origin`/`Referer` are made to agree with that Host on the way in** (`rewriteRequestOrigin`). Dev servers (Vite, Next dev, Django, Rails) guard mutating requests with a CSRF check: Origin must match their own Host. The browser's Origin names the relay/proxy address, we hand the server `Host: 127.0.0.1:<target>` — so without the rewrite every login POST is refused while GETs work. Rewritten **only when the header names the proxy's own public host** (compared against the incoming Host); a genuinely cross-origin Origin (an OAuth provider posting back) passes through untouched. The HMR upgrade path rewrites Origin too — Vite runs the same check there. Production builds usually skip this dev-server check, which is why the bug only showed against a dev run.

`forward.open(port, ctx, path)` mints a link that *lands* on a path (`/p/<room>/<inst>/3000/admin?_ideauth=…`); `normalizeForwardPath` keeps that path a path, so a phone can never steer the auth token at another origin.

## Port forwarding over the relay (the Ports tab, off-LAN)

The Ports tab opens a URL in the phone's **browser**, and a browser cannot speak our ws protocol — so off-LAN the relay itself has to be the public HTTP entry point. The design constraint that shapes everything here: **the relay tunnels raw TCP bytes, never HTTP.** It splices the browser's connection down the desktop's relay socket and into the desktop's own `http-proxy` listener on `127.0.0.1`, so `http-proxy.js` handles the request, the `?_ideauth=` → cookie swap and the HMR upgrade exactly as it does for a LAN phone, and none of it was reimplemented. `{h:<streamId>, t:'open'|'data'|'end'|'close'}` frames (`relay-frames.js`) are the whole tunnel vocabulary.

`relay-route.js` (pure) tells the four kinds of traffic on the single Render port apart from the request head alone, and **the order of its checks is load-bearing**:

1. `/p/<room>/<instance>/<port>/…` — the entry URL `forward.open` hands a relayed phone. 302 to the site **root**, parking the target in an `_idetunnel` cookie. Checked first so opening a second port switches the cookie instead of being swallowed by the tunnel the first one installed.
2. `Cookie: _idetunnel=<room>.<instance>.<port>` — everything else below that root: spliced. **Routing by cookie rather than by path is why the tunnel can live at the root**, and that is the point: a dev server emits absolute asset paths (`/assets/x.js`), so serving it under a `/p/…` prefix would mean rewriting every URL it produces. This check must come *before* the websocket one — a dev server's HMR socket is an `Upgrade` too, and is told apart from the IDE's own only by carrying this cookie.
3. `Upgrade: websocket` — `/?role=desktop|mobile`, the IDE protocol.
4. otherwise — the health check the keep-alive ping hits.

**The target is a window, not just a machine.** Two instances can each be proxying the same dev-server port, and the `?_ideauth=` token is only good at the proxy that minted it — so a room id alone would hand the browser to a coin flip. The relay is deployed separately and outlives any one desktop build, so it still meets desktops that name no window: an entry path or cookie with no instance segment resolves to `DEFAULT_INSTANCE` (`'default'`), and the two forms are told apart without ambiguity because an instance id is never a bare number.

**The relay decides where a connection goes once, from its first request head — so every routed HTTP exchange is made single-use.** A non-upgrade tunneled request gets `Connection: close` injected into its head (`singleUseHead` in `relay-route.js`), the health check and entry/deny replies all answer `connection: close`, and only websocket upgrades (HMR, the IDE protocol) stay connection-long. This is load-bearing behind the deployed relay: Render's front proxy pools and reuses upstream connections across unrelated requests, so before this a `/login` click could arrive on a connection routed to the health check ("ide-relay ok" instead of the page) and a fetch could 502 on a connection whose tunnel was already gone. Routing is therefore effectively per-request, driven by the cookie on a fresh connection each time. Tests should still pass `agent: false` so each call is its own connection.

A malformed room, instance or port is denied outright, never routed. The tunnel is **not** an open door: a stranger who guesses the relay URL is carried to the desktop's proxy, which refuses them (403) — the desktop keeps the last word on auth, as everywhere else here. A cookie naming a window that isn't there is a 502, never a fallback to whichever window happens to be in the room.

## Desktop glue (src/main)

- `remote-bridge.js` — subsystems register via its `handle`/`on` (aliased `bridge.handle`/`bridge.on`) instead of `ipcMain`; registers with both Electron and a registry. Remote calls invoke the same fn with a stub event `{sender:null, remote:true, deviceId}`. Bridged: `repo.js`, `sessions.js`, `git.js`, `explorer.js`, `consoles.js`, `session-commit.js`, and *part of* `run-configs.js` (`get-run-configs` / `run-config-start` / `run-config-stop`; the desktop-only `run-config` stays on raw `ipcMain` — see **Run panel**). Kept on raw `ipcMain`: `db.js`, `runners.js`, `onboarding-store.js`, `window.js`, `remote.js` itself.
- `remote.js` — desktop control (never remote-callable **except** `list-instances`, which is bridged precisely because it must be): enable/disable, which starts the relay client over one `createHub(...)`; QR payloads carrying the relay origin and room; device store in `remote-devices.json` under `sharedDataDir`; forwards `sendToRenderer` pushes via the `onBroadcast` seam in `window.js`; owns the port-forward proxy map (proxies die on disable/quit). `forward.open(port)` always mints a relay entry URL. Failing to reach the relay is **not fatal**: it retries in the background.
- `instance-registry.js` — this window's entry in the machine-wide `remote-instances.json`, and the roster `list-instances` serves. Published on `enable()` (which is what makes the window *reachable*, and therefore listable), republished on `onRepoChange` (the project is the only field that changes while a window runs, and it is what the phone picks by), removed on `disable()` and `before-quit`. See **A room is a machine, not a window**.
- **Pairing survives a restart only if the service does.** A paired phone stores the relay URL+room and reconnects on its own, so nothing about it is rediscoverable: the desktop must come back **in the same room**. The service used to start only from the Settings toggle, so every restart left the devices paired but unreachable and the only cure was re-scanning the QR. `remote-config.json` (beside `remote-devices.json` in `sharedDataDir`; shape normalized by the pure `remote-config-lib.js`) persists `{enabled, relayUrl, room}`: `enable()`/`disable()` write it, and `app.whenReady()` re-enables the relay if it was on. The room is **never** silently regenerated — minting a new one would strand every device holding the old one.
- **Every** repo switch pushes `folder-changed` — `setRepoPath` in `repo.js` sends it, so the dialog (Browse…), recent menu, dock menu and remote `open-folder-path` all notify paired phones and the desktop UI alike. It used to be pushed only from the remote/dock paths, so a project opened via the desktop's own dialog left phones showing "no project open". The renderer re-applies its own opens twice (invoke result + push); that duplicate refresh is accepted.
- **The phone is an input device for the desktop: main owns the state, clients are views of it.** Any action that changes the *shape* of the session list (new / suspend / resume / kill / evict) ends with `broadcastSessions()` in `src/main/sessions.js` — a `sessions-changed` push carrying the whole list, so every client reconciles against main instead of trusting its own optimistic edit. The desktop renderer's `syncSessions()` adopts rows created elsewhere (a `live` flag in the payload says whether to attach a real terminal or the "restore me" placeholder), mirrors archive/restore without re-issuing the IPC (main already killed/respawned the PTY), and drops rows deleted elsewhere. Echoes of a client's own change reconcile to a no-op. **A new list-shape mutation must call `broadcastSessions()` or the other client goes stale** — that was the original bug: the mobile list only listened to `session-meta`, which fires for tracked-file churn and never for archive/delete. This includes the paths that *aren't* explicit user actions: a session ending by natural PTY exit (`/exit`, TUI close, CLI death) deletes the row in `spawnPty`'s `onExit`, so it broadcasts there too (and `chat.forget(id)`s the gone session, or its transcript watcher/messages leak). The rows are built from one shared shaper (`baseRow`): `sessionList()` adds the tracked-file array, `sessionRow()` (the phone page) omits it — a new field can't land on one view and desync the other.

  Beyond the identity/state fields, `baseRow` carries what a phone session **card** draws: `startedAt` / `lastActiveAt` (ms epochs — `0` means "unknown", i.e. a snapshot written before these existed, which a row renders as *no* time rather than as 1970); `tool` (`{name, file}` — the in-flight tool, for the mono line on a working card); and `added` / `removed` (the "+124 −38" pill). Three things are load-bearing here:
  - **`lastActiveAt` is stamped in `setSessionState`**, the single funnel every state change goes through, plus on each `PreToolUse`.
  - **`tool` is cleared by `setSessionState` whenever the state isn't `working`** — *not* by `PostToolUse`. Claude runs tools in parallel, so one finishing says nothing about whether the session is still busy, and clearing there would blank the line mid-turn. It is also **never persisted and never sets `changed`**: it's ephemeral (a restored session runs nothing), and `changed` means the tracked-*file* list moved — raising it per tool call would fire a `session-meta` push and a disk write on every single tool.
  - **`added`/`removed` come from the ops, not from `git diff`** (`diffStat()` in `edit-ops.js`, unit-tested). Not just because the ops are already in memory and the phone polls its page — because a working-tree diff is **global and cannot attribute lines to a session**, which is the whole point of the pill. It's a magnitude, not a patch stat: a `write` has no pre-image so its removals are invisible, and `fileOps` files (binary, renamed, deleted) carry no line counts at all.
- **A session restored from disk has no PTY until someone opens it — and a phone counts as someone.** The Claude process can't outlive the desktop, so every session reloaded on launch comes back `live: false`; the desktop spawns one lazily when you click the row (`selectSession` → `resumeSessionUI`, for non-archived rows — archived ones wait for an explicit Restore). A phone opening such a session must resume it too, or it stares at a dead terminal: no `pty-data` will ever arrive and its keystrokes go nowhere. So `SessionsScreen.open()` keys the resume off **`live`, not `archived`** — the bug was that it resumed only archived rows, leaving every session from before the last desktop launch unusable from the phone. Main's `resume-session` spawns the PTY headlessly (no desktop UI involved) and `broadcastSessions()` tells the desktop, so `syncSessions()` must key its terminal rebuild off **`meta.live && s.suspended`** rather than an archive→active edge: in this case `archived` was false all along and never changes, and keying off it left the desktop showing "select to resume" over an already-running session. `resume-session` is **idempotent**, and the `s.pty` early-return alone is not enough to make it so: `spawnPty` awaits (resolving the claude binary), so two concurrent resumes both pass the guard before either assigns `s.pty`. It dedupes on an in-flight promise set **synchronously** (`s.spawning`) — the second caller joins the first spawn — because two clients can ask at once, a client's `live` flag is always a moment stale, and a second spawn would strand the first PTY as an orphaned `claude` process nothing can reach or kill.
- **The desktop gets that push with the whole list; a phone gets it as a bare signal.** The phone *pages* its list (`query-sessions`, below) instead of holding the set, so shipping every session on every list change would defeat the paging — `remote.js`'s `onBroadcast` hook sends `sessions-changed` to remote clients with a `null` payload, meaning only "refetch your page". The renderer's own `sendToRenderer` payload is untouched; `syncSessions()` still reconciles against the full list.
- `query-sessions {tab, query, offset, limit}` → `{items, total, counts}` (`src/main/sessions.js`, pure logic in `session-query-lib.js`) — one page of the list, tab-filtered, searched and sorted newest-first in main, scoped to the open project. Rows are slimmer than `sessionList()`'s: no tracked-file array, which is the bulk of a row and which a phone's list doesn't draw. `limit` is clamped (default 30, max 200) so one request can never pull a whole archive. Search matching mirrors the desktop's Archived-tab search exactly (fold + whitespace-split ANDed terms, `.ext` matches by suffix) — the same semantics as `shared/name-match.js` and explorer's `search-names`, reimplemented CommonJS-side because main can't import the renderer's ESM.
- UI: **Settings dialog → "Remote access" group** (`src/renderer/remote-pane.js`): enable toggle, QR (`qrcode-generator` ESM via import map), paired-device list with revoke. It refreshes on the settings-btn click (additive `addEventListener`, since settings.js owns `.onclick`).
- **Toolbar phone indicator** (`#phone-indicator`, left of the usage meter): a lucide smartphone icon shown while at least one paired device holds a live socket to this window. The hub's `onClientsChanged(count)` (fires on authed-client add/remove, not on unauthed sockets) is wired in `remote.js` to `sendToRenderer('remote-clients-changed', count)` — a desktop-only channel, not in `REMOTE_EVENTS` — and `disable()` sends an explicit `0`. `remote-pane.js` toggles the icon's `hidden` off the count.

## Push notifications (completed sessions reach a closed app)

The phone holds **no background socket** (battery — see the suspend/resume gotcha), so a session finishing while the phone sleeps can only reach it over the platform push channel: **Expo's push service** (FCM/APNs underneath). The desktop is the sender; there is no third backend and the relay is not involved.

- **Registration**: on every `ready`, the phone fetches its Expo push token (`mobile/src/api/push.ts` — permission prompt, Android `sessions` channel, cached per launch) and calls the bridged `register-push {token}` (in `REMOTE_CHANNELS.req`). `remote.js` binds it to **the caller's own** device record (`event.deviceId` — one phone can never register for another) in `remote-devices.json` as `pushToken`. Revoking a device deletes the record, token included. `token: null` clears it.
- **Sending**: `setSessionState` (`sessions.js`) fires on the **`working` → `completed`** transition only — `pushed` after a commit or a restored session settling is not a run finishing. It calls `push.notifySessionCompleted` (`src/main/push.js`, thin) which POSTs to `https://exp.host/--/api/v2/push/send` for every device with a token; all decisions (token shape, message build, which tokens a `DeviceNotRegistered` ticket says to drop, the store update) are pure in `push-lib.js` (`test/push-lib.test.js`). Sending is gated on the remote service being **enabled** (`push.setEnabled` mirrors enable/disable) and is fire-and-forget — a push failure never touches the session.
- **Tap → chat**: the message's `data` carries `{sessionId}`. `onNotificationTap` (push.ts) handles both live taps and the tap that cold-started the app (`getLastNotificationResponseAsync`, deduped by response identifier); `App.tsx` parks the id in `pendingSession` and navigates `Chat {id}` once `chosen && ready && navReady` — a cold start pairs and connects first. Foreground banners are suppressed (`setNotificationHandler`): the ws events already update the UI live.
- **Prereqs/limits**: needs an EAS project id (`eas init` → `extra.eas.projectId` in app.json) — without it, on a simulator, or with permission denied, push silently stays off and nothing else is affected. A standalone Android build needs FCM credentials configured in EAS. **Multi-window caveat**: the tap opens whatever window the phone connects to by default; a session from a sibling window won't be found there (accepted — single window is the common case).

## The session as a chat (the phone has no terminal)

**A Claude session is a conversation, and the phone renders it as one** — messages, a composer, image attachments, a `/` command menu. There is no xterm on the session screen and no way to reach one: a terminal on a phone is a bad terminal, and the thing the user actually wants from a session is the conversation. (The *run* console — a dev server's log — is still a terminal, because that genuinely is one. It keeps `XtermWebView`.)

Nothing about this is scraped from the TUI. `src/main/chat.js` owns the three pieces a chat needs that a PTY stream doesn't give you:

- **The messages** come from **Claude Code's own JSONL transcript** — the same data the TUI draws, already structured. Its path is published in exactly one place, `transcript_path` on **every hook payload**, so `sessions.js` hands it over (`noteTranscript`) and persists it on the session record: no hook fires for an *archived* session, so without that field a restart would leave its conversation unreadable. `chat.js` tails the file while the PTY is live (`fs.watch` + a debounced read from the last byte offset) and `transcript-lib.js` folds the new lines into messages. A session started **from scratch** is handed its path before Claude has written a line to it, so the file **not existing** is the normal first thing that happens, not an error: `chat.js` retries the watch until it appears, and `noteTranscript` re-pokes on every hook payload rather than only on a change of path. Both matter — a resumed session's file is already on disk and works without them, so the failure they prevent (a new session whose chat is empty forever, not even showing the user's own messages) is invisible until you start one. It's a **reducer, not a parser**: a tool's result lands in a *later* line than its call, so a result **patches the message that already carries the call**, which is why `transcript-data` pushes are upserted by uuid rather than appended. Every push carries a `seq` and the `session-transcript` snapshot reports the last one it contains, so a push that raced the fetch can't drop an older copy of a message on top of a newer one (the same scheme as `session-scrollback`). The window is bounded at 400 messages — a transcript runs to tens of MB and is never shipped whole. Sidechain (subagent) entries are skipped: the `Task` call that spawned them is already in the chat. An **Edit/MultiEdit/Write call** carries a `diff` on its tool block (`{added, removed, lines: [{n, sign, text}]}`, capped at 40 lines): built from the *input*'s `old_string`/`new_string` the moment the call appears (`n: 0` — no line number yet), then **upgraded by the result's** `toolUseResult.structuredPatch` (`diffFromPatch`) to the CLI's own hunks with real line numbers. The phone's `ToolCard` renders it to the design mock's 4a: a green `+N −N` badge in the header, rows of `<n>  <sign> <code>` on 10%-alpha red/green washes with soft red/green code (`redSoft`/`greenSoft`). **Only the Edit tools' diff is open by default** (the diff is the payload, so those cards aren't expandable); a Write's diff — the whole file — waits behind the chevron like any other tool's output.
- **The question the session is blocked on** — one of Claude's own multiple-choice questions (the `AskUserQuestion` tool) or a permission prompt. Either is drawn *in the terminal*, and the transcript won't have it until it's answered (Claude records a tool call only once it has run), so a chat that ignored it would sit there looking idle while the session waited forever. It is pushed as `session-ask` and rendered as a card above the composer. `ask-lib.js` is the whole model: what the card shows, and the keystrokes that answer it.

  **The question is read off the hook, never off the terminal — the terminal cannot be read.** Claude Code's TUI is an **Ink** app: it paints by *moving the cursor*, not by emitting rows and spaces. Strip the escapes and a whole screen collapses into one glued line with the gaps gone (`…❯1.Postgres` running straight into the next option) and characters *missing* wherever a repaint overwrote a cell — a captured box really does read back `Embedded file-based daabase`. No regex survives that, and the first version of this feature (which scraped the PTY tail for a numbered menu) never once produced a correct card: what reached the phone was rows stitched out of stale frames, showing a question that had already been answered and offering the question text itself as an option. **Do not try to parse the PTY.** It doesn't need parsing: every question is *announced* by a hook that carries it whole —
    - `PreToolUse` for `AskUserQuestion` carries `tool_input.questions`: **every** question (Claude can ask up to four in one call), every option, each option's description, and `multiSelect`. It fires *before* the box is even painted.
    - `PermissionRequest` carries the tool it wants to run, which is enough to say what is being allowed.

  **A question is only `needs-input` because we made it one.** Claude Code fires a `Notification` when it wants *permission*, but a question is an ordinary **tool call** — its only hook is a plain `PreToolUse`, which would leave the session yellow with a menu on screen and nothing saying so. So `hook-events.js` maps `PreToolUse` for an ask tool to `needs-input`. `Claude Code also fires a `PermissionRequest` for `AskUserQuestion` itself`; `ask-lib.js` ignores it, or a bare "Allow AskUserQuestion?" card would replace the actual question.

  **Answering is `answer-ask {id, answers}` — the whole box at once.** The card is built from the hook, but the thing on screen is still a menu, and keystrokes are the only way to work one. Those keystrokes were established against a live CLI (not guessed), and `keystrokes()` replays what a person would have pressed:
    - an option is its **1-based number**, and picking one **auto-advances to the next question**; after the last one the box lands on its own **Submit tab, where `1` submits**. So two questions answer as `2`, `1`, `1` — and an answer missing for *any* question produces **no keystrokes at all**, because the submit key would otherwise land on a menu that is still asking and pick whatever sits under the cursor.
    - the row after the last real option is **"Type something."** — it opens a text field, so a custom answer is that number, then the words, then Enter. That is the point of a chat over a menu: the user is never stuck choosing between someone else's answers.
    - a **`multiSelect`** question does *not* auto-advance: its numbers **toggle** rows on and off and **Enter commits** the set, so an answer carries `keys` (an array) rather than `key`. The card mirrors that — its options are checkboxes that tint as you toggle them and a "Choose N selected" button commits, because a tap that answered immediately could only ever produce one choice. Single-select still tints the *first* option (the one Claude leads with) and commits on the tap; on a multi-select card that standing tint would be indistinguishable from a row you had already picked, so there it follows your toggles instead.
    - a **permission** box answers yes with `1` (always its first option) and no with **Esc** (it prints `Esc to cancel`). The card offers only those two. The "yes, and stop asking this session" row in between is numbered *differently depending on what Claude Code suggests for that tool*, so its number cannot be derived from the hook — and a guessed number there would **grant** what the user declined. It is also a standing grant, which a phone is the wrong place to hand out.

  `sessions.js` waits for the paint to land and go quiet before typing (the hook beats the box onto the screen, and keys pressed into a box that isn't there yet are simply lost), and leaves a beat between keystrokes so a key can't arrive mid-redraw and land on the question on its way out. The card is cleared the moment the keys go out — not when the confirming hook lands — so it can't invite a second answer into a menu that has moved on. A prompt answered at the *desktop's* terminal clears it too: `PostToolUse` says the tool ran, and `Stop` covers the rejected case (a rejected tool never runs, so it never fires `PostToolUse`).
- **The composer's inputs.** `list-slash-commands` merges Claude Code's own commands with the markdown ones in the project's and the user's `.claude/commands` (project shadows user shadows builtin, as in the CLI). `save-attachment` writes a phone's photo to a temp dir and returns its path — Claude reads an image file exactly as it reads a source file, so a path is all a prompt needs, and nothing lands in the user's working tree.
- **What the session is running: the model, and how hard it thinks.** The chat's top bar carries **both as tappable badges** under the title ("Working · [Opus ⌄] · [High ⌄]") — the same pair, in the same place, that the desktop
  keeps in its session bar — `src/components/chat/BadgeMenu.tsx` is one pill plus the menu that changes it, and the chat mounts one per setting. The menu is anchored **under the badge it belongs to** (measured at the tap with `measureInWindow`, clamped inside the screen): with two badges side by side, a menu floating in the middle of the screen would leave you guessing which one you were about to change. It grows out of its top edge like the desktop's `.effort-menu` — RN scales about the centre, so the translate offsets half the collapsed height back to pin that edge, the same trick (mirrored) as the Sessions screen's model menu. Both are switched on the **running** session over `set-session-model` / `set-session-effort` (main types the CLI's own `/model`/`/effort` into the session and remembers the choice on the record; a Codex session respawns instead — see [settings.md](settings.md#reasoning-effort-per-session)), so there is no Apply button: a pick applies as it is tapped, and the badge moves with the tap rather than with the round trip — a picker that sat unchanged until the desktop answered would read as one that didn't work. Note the badge can therefore run **ahead of the session**: a typed command queues behind the current turn, so a switch made mid-run shows on the badge before the CLI acts on it. The `session-model` / `session-effort` pushes correct it, and are also how the phone follows a change made **at the machine** (its badge menu, or a slash command typed into its terminal). The model menu drops the `default` inherit sentinel, as the desktop's does — it isn't a model you pick — while the effort badge names "Auto" even though that is the default: a badge you can tap has to say what it is.

  **Effort is the one of the two a phone can't pick at creation**, and deliberately so: it is a *spawn flag* (`--effort`), so offering it on the Sessions screen's New-session menu would mean a second sticky default to keep — while a session created at its model's own effort and raised to `xhigh` from the chat, once, covers what anyone actually does. The model list on that menu stays the one thing picked up front (`mobile/src/api/models.ts`, sticky in SecureStore); `EFFORTS` lives beside it and is remembered nowhere.

**How the chat stays in step with the desktop (the sync contract).** Three layers, each catching what the one above misses: (1) *live pushes* — `transcript-data` is a **watched stream** (see `STREAM_EVENTS` in `protocol.js`): ChatScreen calls `conn.watch('transcript-data', id)` **before** fetching the snapshot (frame ordering on one socket makes the subscribe land first, so a message written between the two isn't missed), and `Connection` ref-counts watches and **replays them after every re-auth**, since the hub forgets a dead socket's subscriptions. The same applies to `term-data` (ConsoleTerminal watches while mounted) and `pty-data` (`primePrompt` watches just long enough to see the TUI's first paint). (2) *seq-gap resync* — every push carries the stream's counter; ChatScreen tracks the last one applied, and a push that isn't its successor (a frame lost in the relay, or the stream restarted under a `--resume`) applies what arrived and then re-runs the snapshot fetch (`load()`) to fill the hole — during which pushes buffer and reconcile against the new snapshot's seq exactly as at mount, so a resync can't re-enter itself. (3) *reconnect* — a new `conn` re-runs the whole effect from scratch. The point of the watch layer is **latency**: before it, every phone received every live session's `pty-data` (a Claude TUI repaints constantly) through the single relay socket, and the chat's small pushes queued behind that firehose.

Sending is `send-prompt {id, text, images}` (`sessions.js`), not raw `pty-input`: the TUI ingests a multi-line paste over several ticks and an Enter bundled with it fires the prompt half-typed, so the Enter is a **separate write a beat later** — the desktop's `newSessionWithPrompt` timings, for the desktop's reasons. It also waits for the session's first PTY output before typing: a session opened the instant it was created hasn't painted its input box, and keystrokes that arrive first are dropped.

Mobile side: `src/screens/ChatScreen.tsx` (messages + the ask card + the commit bar). **The list follows the tail unless the reader leaves it**: an `atBottom` ref is recomputed on every scroll (`scrollEventThrottle` 16 — read the flag a fifth of a second late and a reader who just scrolled up gets yanked back down by the next message) and the list re-pins on `onContentSizeChange` *and* `onLayout`, the second being the keyboard taking the bottom of the screen, which shrinks the viewport without adding a message. The ask card is the list's **footer**, not a pinned overlay: it takes whatever height its question needs (no `maxHeight`, no inner ScrollView) and scrolls with the conversation, so a long question never walls off earlier messages — `onContentSizeChange` re-pins when it appears. Scrolling back down within `BOTTOM_SLACK` of the end resumes following. Its parts live in `src/components/chat/` (`BadgeMenu` — the model/effort pill + its anchored menu, above; `MessageView` — user bubbles vs. assistant prose vs. expandable tool/thinking cards; `Composer` — draft, attachments, the `/` menu, which opens only while the **first** word is being typed as a command; `Markdown` — a small hand-written renderer, since Claude answers in markdown and a chat showing `**` at the reader is broken. **Every text style must carry the line height for its own font size.** RN draws lines exactly `lineHeight` apart with no minimum, so a style that raises `fontSize` while inheriting someone else's `lineHeight` doesn't get a cramped line — it gets one line drawn *on top of* the next. Two ways in, and the chat had both: a heading composed as `[body, h1]` kept `body`'s 22 under a 21px font, and an inline `` `code` `` span swapped in the mono family with no line height of its own, which Android measures with the taller metrics and overlaps the paragraph around it — and Claude writes `code` in half its sentences, so that alone made most of the chat unreadable. Headings size their own line box; `Inline` gives a nested span the line box of whatever it lands in (`StyleSheet.flatten(style).lineHeight`), which is the body's in a paragraph and a bigger one in a heading — a constant would just move the collision into headings), and `src/api/chat.ts` (the wire types + `upsert`). `src/theme.ts` holds the design tokens the chat is built from — the desktop's own palette, so the phone and the machine look like one product.

## Shared desktop assets on the phone (generated)

The Files screen shows the same icons and the same colours as the desktop explorer, and neither is maintained twice. `mobile/` is a separate package with its own Metro root, so it cannot import `src/renderer/` — instead `scripts/gen-mobile-assets.mjs` (`npm run gen:mobile`) bakes the desktop's own tables and artwork into `mobile/src/generated/desktop-assets.ts`: `EXT_ICON`/`NAME_ICON` + every SVG in `src/renderer/file-icons/` (inlined, self-colored, painted by `react-native-svg`'s `SvgXml` in `src/components/FileIcon.tsx`), and `EXT_LANG` from `shared/highlight.js`. **`src/renderer/shared/file-icons.js` stays the single source** — after adding an icon or a language mapping, re-run the generator; `test/mobile-assets.test.mjs` fails the build if the checked-in output has drifted.

`CodeView` renders in a WebView because highlight.js emits HTML (XtermWebView's precedent: library from a CDN — the phone has its own internet). It pins the *same* hljs 11.11.1 + `vs2015` theme `index.html` pins, side-loads the same `dos`/`powershell` grammars missing from the common bundle, and ports `hlLines()` from `shared/highlight.js` so each line is a balanced row against a sticky gutter. A file over 400k chars renders as plain text — hljs is linear but the DOM it builds is not. If the CDN is unreachable, highlighting degrades to plain text rather than failing.

## Sessions are shared (no handover)

A session is **not** locked to whichever client opened it: the phone and the desktop drive the same session at the same time, like two panes onto one tmux window. There is no claim channel (`session-control` was removed from the protocol along with `controlledBy`/`heldByAnotherDevice` in main and the desktop's "Controlled by mobile" cover), no `controlled` flag on session rows, and hub `onDisconnect` is no longer wired (the hook still exists in `server/hub.js` for tests). The last resize from either end wins the PTY's geometry — accepted trade-off for simultaneous access.

## Run panel (launch configs, tasks, and their terminals)

The phone's header-right button opens `src/components/RunDrawer.tsx` — a right slide-in mirroring `ProjectDrawer` — listing the `.vscode` launch configs, the tasks, and the terminals they opened. Tapping a terminal navigates to Console **without** closing the panel: the drawer is a Modal (floats above pushed screens), so `App.tsx` gates its `visible` on the Main screen's focus (`useIsFocused`) — hidden while Console is up, back on screen when you back out.

**Nothing runs on the phone.** A terminal tab is a renderer thing (xterm + tab UI), so `run-config-start {kind, name}` resolves the specs in main and pushes them to the *desktop* renderer as `run-specs`; `toolbar.js` opens them through `runSpecInConsole` — the exact path a toolbar click takes. One code path, so the two ends can't drift, and tab reuse means a second tap is a rerun (the play icon flips to a restart glyph, as on the desktop). `run-config-stop {name}` kills the config's terminal(s), resolving a compound to its members like the toolbar's Stop. `${input:...}` configs are the one thing that can't run headlessly — the answers come from a modal the desktop renderer owns — so they come back as a plain error telling you to start them from the desktop.

**Main is the source of truth for what's open.** It used to store a bare pty per id and let the renderer keep the name/kind privately, which is why nothing else could see what was running. Each console entry now carries `{name, kind}` (the renderer sends both — naming still has one owner) and every open/close/restart pushes `terminals-changed`. That single event does double duty: it's the terminal list *and* the run state, since a launch config "runs" for exactly as long as its terminal is open (kind `config`, matched by name — the definition the desktop toolbar already used). `killConsole()` always pushes `term-exit`, unlike the pty's own `onExit` (which stays quiet for a terminal we deliberately replaced), so a stop from the phone closes the desktop tab too.

The run terminal itself (`src/screens/ConsoleTerminal.tsx`) attaches over `term-list` + `term-scrollback` (same bounded-tail + `seq` dedupe scheme as session scrollback, below) and types over `term-input`. Two deliberate non-features:

- **It does not resize the PTY — it mirrors the PTY's grid instead.** That terminal is also open in a desktop tab sized for the desktop; a phone-width resize would reflow the view under whoever is sitting at the machine. But the byte stream is painted for exactly the PTY's cols×rows — every cursor move, `ESC[K` rewrite and progress-bar repaint targets those cells — so *fitting* the phone's xterm to phone width scrambled it: output landed mid-screen, on top of earlier text. `term-scrollback` therefore carries the PTY's `cols`/`rows`, the screen pins its xterm to them **before** replaying the snapshot (`setDims` → `window.__setDims`, which shrinks the font until that many columns fit the phone's width — small text beats scrambled text), and a `term-resized` push (`REMOTE_EVENTS`; sent on desktop `term-resize` and on a restart's respawn under the same id) keeps it following. While pinned, a viewport change (keyboard, rotation) re-derives only the font size, never the cols/rows.
- **It does not claim the terminal.** A run terminal is a shared shell — both ends stay live, exactly as sessions themselves now are (see **Sessions are shared**).

**Typing goes through a native input bar, not xterm's hidden textarea.** Mobile IMEs (Android especially) feed the WebView composition events xterm mangles or drops, so tapping the terminal and typing never reliably reached the PTY. The bar under the terminal is a native `TextInput` whose send writes the line + `\r` in one `term-input`, plus a `^C` key (`\x03`) — stopping the running command without killing the tab, the one control key a shared shell needs from a phone. It hides once the terminal has exited. Tap-to-focus typing into xterm still exists for platforms where it works; the bar is the path that always does.

**You cannot select xterm's own output on touch, and lifting its `user-select:none` does not fix it** — xterm's pointer handlers `preventDefault` in order to run its *own* mouse selection, so the platform's long-press gesture never fires. (That was the first attempt; the Select button did nothing.) Select mode instead lays `#sel` — a plain `<pre>` holding `bufferText()` — over the terminal: ordinary DOM text, so native handles, the magnifier, the system Copy menu and native scrolling all behave. It's a **snapshot**; re-rendering it as live output arrived would wipe a selection mid-drag, so the terminal keeps receiving underneath and the overlay stays still until you leave the mode. `__copy()` takes the selection, else the whole buffer.

## Terminal scrollback (mobile)

Both apply to the **run console** — the one terminal left on the phone. (A Claude session is a chat now, and its history comes from the transcript, not from retained PTY bytes.)

- **The terminal is streamed as text, never as pixels.** Main forwards the PTY's own bytes and xterm renders them on the phone, so the output is real scrollable text (selectable, copyable — see **Run panel**) rather than a picture of a terminal. Nothing here is a video stream, and nothing should become one. (The two **screen-streaming** features are the deliberate exceptions: the **remote browser** streams a web page as watched `browser-frame` JPEGs, and **remote desktop control** streams the whole screen as `screen-frame` JPEGs — a page and a live desktop have no text form. See **Remote browser** and **Remote desktop control**.)
- **Nothing to scroll back to.** Leaving the screen destroys the WebView and its whole buffer. So `sessions.js` keeps a bounded tail of each live session's PTY output (`s.scroll`, 200k chars, whole chunks dropped off the front, reset on every `spawnPty` since a fresh PTY redraws anyway) and serves it over `session-scrollback`, and `consoles.js` does the same for a terminal (`term-scrollback`, `SCROLL_MAX`, **1M chars**). The screen replays that snapshot before going live.
- **The retention cap is two numbers, and they only work as a pair.** `SCROLL_MAX` in `consoles.js` is a budget in **characters**; xterm's `scrollback` in `XtermWebView.tsx` is a budget in **rows**. Main's cap is the real ceiling — what it drops off the front, no client can ever ask for again — but the phone is narrow and soft-wraps one desktop-width line into two or three rows, so a row budget sized to match main's characters *at desktop width* would silently discard the oldest output as it arrived, and raising main's cap alone would look like it did nothing. xterm's is therefore held well above what 1M chars can produce at phone width (100k rows), leaving main's cap the only thing that truncates. Raise them together or not at all. Every `pty-data`/`term-data` carries a `seq`, and the snapshot reports the last `seq` it contains, so chunks that raced the fetch are dropped instead of printed twice — ordering on the single socket guarantees nothing older is missed. A session's tail outlived the chat rewrite for a second reason: it is **the only copy of the box the TUI is drawing**, and `chat.js` reads it to lift out a permission prompt (see **The session as a chat**).
- **Drags don't scroll.** xterm stacks `.xterm-screen` **over** the scrollable `.xterm-viewport`, so a touch on the terminal body never reaches the scroll container. The page drives `viewport.scrollTop` from the touch itself. Do **not** go back to `term.scrollLines()`: it snaps the drag to whole rows and re-renders synchronously inside the touch handler, which is what made it stutter. Pixel-accurate scrollTop writes, coalesced to one per frame, let xterm render off its own scroll event (the desktop scrollbar's path); a velocity-decay fling on touchend supplies the momentum a native list would have.

## Gotchas

- `git-log` takes `{limit, skip, query}` and returns `{commits, incoming, hasMore}`. **The desktop calls it with no args**, so the defaults must keep reproducing its old behavior (newest 100 + incoming) — don't make an option required. Browsing pages straight out of git (`-n`/`--skip`), but a *search* can't: the query matches author and hash as well as the subject, which `git log --grep` can't express, so main scans the log in 2000-commit chunks and filters each with `filterCommits` — the same matcher the desktop History tab uses — stopping as soon as the page is full. Both paths over-fetch one commit past the page so `hasMore` is free (`pageCommits`). `incoming` rides along only on an unfiltered first page: repeating it per page would duplicate rows, and a search asks about local history, not about what a pull would bring in.
- **The relay link is `wss://`** (TLS terminated at Render), so it is encrypted in transit; but the relay operator sits in the middle of it. Frames are opaque to the relay and auth is verified end-to-end by the desktop, yet nothing is end-to-end *encrypted*. In a dev run the relay is plain `ws://localhost:8080` (handed to a phone as `ws://<lan>:8080`) — trusted-network only. End-to-end encryption + cert pinning in the QR remains the known fast-follow.
- Revoking a device does not terminate its already-open socket; it only blocks the next auth.
- **One socket carries one identity for its whole life.** The hub refuses a `pair` on an already-identified client and an `auth` that would resolve to a *different* device (an identity swap) — otherwise the swap would leak the original identity, since `close()` releases only the current `deviceId`. Re-auth to the *same* device is allowed, because a desktop restart re-announces its phones and they re-auth on the socket they already hold.
- **The relay bounds what it accepts, because it is logic-free but not liveness-free.** The ws accept path validates `room` with the same `isRoom` the HTTP-routing path uses (the two diverging was an unbounded room-creation hole); rooms and per-room clients are capped (`maxRooms`/`maxClientsPerRoom`, options on `startRelay` so tests can shrink them); a ws heartbeat (`ping` every 30s, terminate on a missed pong) reaps half-open desktops/phones a locked screen or dropped Wi-Fi left behind; a raw demuxer socket has a head-delivery deadline (slow-loris) cleared the moment it's routed; `WebSocketServer` sets `maxPayload`; and `relay-client` caps a stream's pre-connect queue. The desktop keeps the last word on auth regardless — these only stop resource exhaustion, not access.
- **The port-forward proxy is hardened against a hung/misbehaving dev server:** the upstream request has a timeout, an upstream error *after* headers were sent resets the connection instead of appending an error string into the live body, and the session cookie is compared in constant time (`safeEqual`), same as the URL token.
- **The mobile `req()` times out — and a timeout kills the socket.** A response that never arrives (hung handler, dropped frame) would otherwise leave its promise pending forever and any `busy`/`loading` guard keyed off it stuck. A timeout also `close()`s the socket it was sent on (if it's still the live one and state is `ready`): a 20s silence almost always means a half-open socket (Wi-Fi↔cell switch, NAT reset) the OS won't report for minutes, and closing it hands recovery to the normal reconnect path — event-driven healing with no keepalive pings. The `Connection` also `close()`s the previous socket when it re-dials (a replaced connection reconnects itself forever otherwise), ignores a replaced socket's late `onclose`, guards `JSON.parse` of each frame, clears stored credentials on `auth-err` (so a revoked token doesn't loop every launch), and can `queue` a fire-and-forget send to flush on the next `ready`.
- **Background = suspend, foreground = instant reconnect (the battery rule).** There is deliberately **no background keepalive**: the OS freezes the app's JS in background and the relay's 30s heartbeat reaps the silent socket regardless, so holding it open only costs radio wakeups. `App.tsx` listens to `AppState`: on `background` it calls `conn.suspend()` (close the socket on purpose, cancel any pending backoff timer — clean known-dead state instead of a half-open socket that still claims `ready`); on `active` it calls `conn.resume()` (backoff reset to 0, reconnect immediately — the user is looking at the screen, so the first attempt must not inherit a delay accumulated while asleep). Screens already refetch on reconnect-to-`ready`, so the UI heals itself. Do not add interval pings to the mobile app; the req-timeout rule above covers foreground staleness.
