# Remote access (mobile companion)

Developers can drive the IDE from a phone: QR pairing, project switching, Claude sessions, git, file editing, terminals, and dev-server port forwarding. Three parts: the Electron-free `server/` socket bridge, thin glue in `src/main/remote*.js`, and the Expo app in `mobile/`. Design rule (user requirement): **the backend only bridges sockets/communications** — all IDE logic and auth verification live in the desktop app; `server/` never inspects business payloads.

## server/ — the socket bridge (plain Node, CommonJS, linted like src/main)

- `protocol.js` — pure: message parse/validate, `REMOTE_CHANNELS` **allowlist** (`req` mirrors `ipcMain.handle`, `send` mirrors `ipcMain.on`), `REMOTE_EVENTS` (renderer pushes forwarded to phones), `fwd-open`/`fwd-close` shapes. To expose a channel remotely you must add it here — an existing handler is not enough. Excluded on purpose: `open-folder` (native dialog), clipboard, `db-*`, runners, assets.
- `auth-lib.js` — pure: single-use 5-min pairing tokens (3 wrong guesses invalidates), device credentials (32-byte tokens, only sha256 hashes persisted), injected-store CRUD.
- `ws-server.js` — `startRemoteServer({port, host, hub})` on `0.0.0.0:<port>` (`0` = ephemeral; the desktop passes a fixed one — see below) (`ws` dep); 15s heartbeat. It owns **sockets and nothing else** — every frame goes to the hub, so this file has no protocol in it. Given the hub's opts instead of a `hub` it builds one, which is what the tests do. It rejects on a bind error, so the caller can catch `EADDRINUSE`.
- `http-proxy.js` + `http-proxy-lib.js` — dev-server port forwarding: one listener per forwarded port piping to `127.0.0.1:<target>`; auth = one-time `?_ideauth=` URL token → 302 that sets an HttpOnly cookie (pure decisions in the lib); raw-socket Upgrade passthrough keeps Vite/webpack HMR websockets working. **Untouched by the relay work** — a tunnelled browser is piped into this same listener and it cannot tell one from a LAN browser, which is exactly why the tunnel moves bytes and not HTTP.
- `hub.js` — the client state machine with **no socket under it**: pairing, hello → pair/auth → ready, the channel allowlist, the authed-client set and the broadcast fan-out. `connect(send, via)` returns a client you feed raw frames. It exists so a phone on the LAN and a phone down the relay run the *same* machine against the same device store, and one `broadcast()` reaches both. `via` (`'lan' | 'relay'`) rides on every `invoke` ctx and on `forward.open` — the only thing the two transports disagree about is which address a phone can actually reach (see **Port forwarding** below).
- `relay.js` + `relay-route.js` + `relay-frames.js` + `index.js` + its own `package.json` — standalone cloud relay (deploy: root dir `server`, build `npm install`, start `node index.js`, listens on `process.env.PORT`; live at `https://ide-yj3x.onrender.com`). Frame forwarding between the desktop sockets and the mobile sockets of a room using `{c:<clientId>, d:<frame>}` envelopes; never parses `d`. A room holds **many** desktops, keyed by instance id — see **A room is a machine, not a window**. It also carries the phone's *browser* to a forwarded dev server — see **Port forwarding over the relay**.
- `relay-client.js` — the desktop **dialling out** to the relay, which is the point: a desktop behind NAT can't be dialled, but it can hold one outbound socket, and every phone bound to that window rides it. `?role=desktop&room=<id>&instance=<id>`. Demuxes `{c,…}` into hub clients (`via: 'relay'`) and `{h,…}` into byte streams. Reconnects with backoff; a dead relay socket drops all its clients so the hub releases whatever they held.
- `keepalive.js` — keeps the relay from being idled out by free hosting. `resolveKeepAlive(env)` reads Render's `RENDER_EXTERNAL_URL` (or `RENDER_EXTERNAL_HOSTNAME`) to learn its own public URL — no URL configured anywhere by hand — and `startSelfPing` GETs `<url>/healthz` every 10 min (Render's idle window is ~15). Off entirely when those env vars are absent (LAN/dev), so only the deployed relay pings. Overrides: `KEEPALIVE_URL`, `KEEPALIVE_INTERVAL_MS` (`0` disables). The ping must be a *real inbound HTTP request to the public URL* — hitting `127.0.0.1` would not reset the host's idle timer.

## Two transports: LAN and relay (they run together, not instead of each other)

While remote access is on, the desktop is reachable **both** ways at once, off one hub: the LAN ws server, and an outbound socket to the relay (`https://ide-yj3x.onrender.com`, baked into `DEFAULT_RELAY_URL` in `remote-config-lib.js` and `mobile/src/api/config.ts` — keep the two in step). The desktop **dials out** to the relay because a machine behind NAT cannot be dialled *in* to; every phone in its room rides that one socket.

The QR carries both (`ide://pair?v=1&host=…&port=…&relay=…&room=…&tk=…`) because at scan time neither end knows which will work, and the phone keeps both. It dials **LAN first, relay second** (`dialOrder`): the LAN link is faster and survives the relay being down or cold-starting, and the relay is the only route once the phone leaves the network. A `live` flag in `connection.ts` separates "that endpoint never answered, try the next" (advance immediately) from "a working link dropped" (start the sweep again from the top — the phone may have changed networks). Each endpoint gets a bounded turn (`DIAL_TIMEOUT_MS`): an unreachable LAN address usually fails fast, but a network that black-holes the packets would otherwise hang the app forever.

**Adding a room id to the config was not optional.** A phone holds `{lan, relay}` and can rediscover neither — so the room is the desktop's address on the relay in exactly the way the port is its address on the LAN, and it is persisted, and never silently regenerated, for exactly the same reason (see the port note below). It is minted on first run and **written to disk before the first QR is shown**, not just held in memory.

## A room is a machine, not a window

The IDE is deliberately multi-instance, and `remote-config.json` lives in `sharedDataDir` — so every window shares one room and one configured port. That made siblings *collide*: the relay allowed a single desktop socket per room and closed the previous one with `4001 replaced`, so opening the IDE a second time silently knocked the first off the relay; on the LAN, the second window lost the race for the fixed port, took an ephemeral one, and no phone could find it.

So a window is addressed **inside** the room by an `instanceId` (a uuid per process, `instance.js`, alongside `startedAt`). The relay keys `room.desktops` by it: siblings coexist, a reconnecting window replaces only *its own* stale socket, and closing one window drops only the phones bound to it. Pairing stays **per machine** — `remote-devices.json` is machine-wide, so one QR gets a phone into any window, and there is nothing to re-scan.

**Which window a phone drives is chosen after connecting, not before.** The roster can't be a relay-level thing: the relay is deliberately logic-free, and it must never learn what project a window has open. So `instance-registry.js` has every remote-enabled window keep an entry (`{id, pid, startedAt, lanPort, project}`) in `remote-instances.json` in `sharedDataDir` — the only thing separate processes share, there being no IPC between them — and the authed `list-instances` channel serves the live ones, **oldest first**. Whichever window the phone's socket lands on answers for its siblings out of that file. A window killed or crashed never removes its entry, so the reader prunes by pid liveness (`liveInstances` in the pure `instance-lib.js`), exactly as the stale-dir sweep does.

The phone's **first** dial can't name a window — the roster arrives *over* the connection, so before it has one there is nothing to name. It gets the room's newest desktop (`r.latest`) and lists from there. `lanPort` is in the roster because that is the half a phone cannot derive: the host is the machine's, but the port is the window's.

## Port forwarding over the relay (the Ports tab, off-LAN)

The Ports tab opens a URL in the phone's **browser**, and a browser cannot speak our ws protocol — so off-LAN the relay itself has to be the public HTTP entry point. The design constraint that shapes everything here: **the relay tunnels raw TCP bytes, never HTTP.** It splices the browser's connection down the desktop's relay socket and into the desktop's own `http-proxy` listener on `127.0.0.1`, so `http-proxy.js` handles the request, the `?_ideauth=` → cookie swap and the HMR upgrade exactly as it does for a LAN phone, and none of it was reimplemented. `{h:<streamId>, t:'open'|'data'|'end'|'close'}` frames (`relay-frames.js`) are the whole tunnel vocabulary.

`relay-route.js` (pure) tells the four kinds of traffic on the single Render port apart from the request head alone, and **the order of its checks is load-bearing**:

1. `/p/<room>/<instance>/<port>/…` — the entry URL `forward.open` hands a relayed phone. 302 to the site **root**, parking the target in an `_idetunnel` cookie. Checked first so opening a second port switches the cookie instead of being swallowed by the tunnel the first one installed.
2. `Cookie: _idetunnel=<room>.<instance>.<port>` — everything else below that root: spliced. **Routing by cookie rather than by path is why the tunnel can live at the root**, and that is the point: a dev server emits absolute asset paths (`/assets/x.js`), so serving it under a `/p/…` prefix would mean rewriting every URL it produces. This check must come *before* the websocket one — a dev server's HMR socket is an `Upgrade` too, and is told apart from the IDE's own only by carrying this cookie.
3. `Upgrade: websocket` — `/?role=desktop|mobile`, the IDE protocol.
4. otherwise — the health check the keep-alive ping hits.

**The target is a window, not just a machine.** Two instances can each be proxying the same dev-server port, and the `?_ideauth=` token is only good at the proxy that minted it — so a room id alone would hand the browser to a coin flip. The relay is deployed separately and outlives any one desktop build, so it still meets desktops that name no window: an entry path or cookie with no instance segment resolves to `DEFAULT_INSTANCE` (`'default'`), and the two forms are told apart without ambiguity because an instance id is never a bare number.

**The relay decides where a connection goes once, from its first request head, and then splices it.** A *reused* keep-alive socket therefore keeps going wherever its first request went, whatever cookie later ones carry. A browser never notices, because the entry redirect that changes the cookie is answered with `connection: close` — but a test using Node's global agent will silently assert against the previous request's tunnel unless it passes `agent: false`.

A malformed room, instance or port is denied outright, never routed. The tunnel is **not** an open door: a stranger who guesses the relay URL is carried to the desktop's proxy, which refuses them (403) — the desktop keeps the last word on auth, as everywhere else here. A cookie naming a window that isn't there is a 502, never a fallback to whichever window happens to be in the room.

## Desktop glue (src/main)

- `remote-bridge.js` — subsystems register via its `handle`/`on` (aliased `bridge.handle`/`bridge.on`) instead of `ipcMain`; registers with both Electron and a registry. Remote calls invoke the same fn with a stub event `{sender:null, remote:true, deviceId}`. Bridged: `repo.js`, `sessions.js`, `git.js`, `explorer.js`, `consoles.js`, `session-commit.js`, and *part of* `run-configs.js` (`get-run-configs` / `run-config-start` / `run-config-stop`; the desktop-only `run-config` stays on raw `ipcMain` — see **Run panel**). Kept on raw `ipcMain`: `db.js`, `runners.js`, `onboarding-store.js`, `window.js`, `remote.js` itself.
- `remote.js` — desktop control (never remote-callable **except** `list-instances`, which is bridged precisely because it must be): enable/disable, which starts **both** the LAN ws server and the relay client over one `createHub(...)`; QR payloads carrying both transports (prefers private-range IPv4 for the LAN half); device store in `remote-devices.json` under `sharedDataDir`; forwards `sendToRenderer` pushes via the `onBroadcast` seam in `window.js`; owns the port-forward proxy map (proxies die on disable/quit). `forward.open(port, ctx)` picks the URL off `ctx.via` — the proxy is the same either way, only the address the phone can reach it at differs. Failing to reach the relay is **not fatal**: it retries in the background and same-network use is unaffected.
- `instance-registry.js` — this window's entry in the machine-wide `remote-instances.json`, and the roster `list-instances` serves. Published on `enable()` (which is what makes the window *reachable*, and therefore listable), republished on `onRepoChange` (the project is the only field that changes while a window runs, and it is what the phone picks by), removed on `disable()` and `before-quit`. See **A room is a machine, not a window**.
- **Pairing survives a restart only if the service does.** A paired phone stores its endpoints and reconnects on its own, so nothing about them is rediscoverable: the desktop must come back **listening, on the same port, in the same room**. Both halves used to be missing — the server only ever started from the Settings toggle, and it bound an *ephemeral* port — so every restart left the devices paired but unreachable, and the only cure was re-scanning the QR. `remote-config.json` (beside `remote-devices.json` in `sharedDataDir`; shape normalized by the pure `remote-config-lib.js`) persists `{enabled, port, relayUrl, room}`: `enable()`/`disable()` write it, and `app.whenReady()` re-enables the server if it was on. The port defaults to a fixed `DEFAULT_PORT` and is **never** silently migrated — if it's taken (a sibling instance, which serves the same devices out of the same store, so the phones keep working against it) this instance falls back to an ephemeral port for its own QR and leaves the config alone. Moving the stored port, or minting a new room, would strand every device holding the old one.
- A remote `open-folder-path` also pushes `folder-changed` so the desktop UI follows the switch (repo path is global — remote switch switches the desktop too, by design).
- **The phone is an input device for the desktop: main owns the state, clients are views of it.** Any action that changes the *shape* of the session list (new / suspend / resume / kill / evict) ends with `broadcastSessions()` in `src/main/sessions.js` — a `sessions-changed` push carrying the whole list, so every client reconciles against main instead of trusting its own optimistic edit. The desktop renderer's `syncSessions()` adopts rows created elsewhere (a `live` flag in the payload says whether to attach a real terminal or the "restore me" placeholder), mirrors archive/restore without re-issuing the IPC (main already killed/respawned the PTY), and drops rows deleted elsewhere. Echoes of a client's own change reconcile to a no-op. **A new list-shape mutation must call `broadcastSessions()` or the other client goes stale** — that was the original bug: the mobile list only listened to `session-meta`, which fires for tracked-file churn and never for archive/delete.
- **A session restored from disk has no PTY until someone opens it — and a phone counts as someone.** The Claude process can't outlive the desktop, so every session reloaded on launch comes back `live: false`; the desktop spawns one lazily when you click the row (`selectSession` → `resumeSessionUI`, for non-archived rows — archived ones wait for an explicit Restore). A phone opening such a session must resume it too, or it stares at a dead terminal: no `pty-data` will ever arrive and its keystrokes go nowhere. So `SessionsScreen.open()` keys the resume off **`live`, not `archived`** — the bug was that it resumed only archived rows, leaving every session from before the last desktop launch unusable from the phone. Main's `resume-session` spawns the PTY headlessly (no desktop UI involved) and `broadcastSessions()` tells the desktop, so `syncSessions()` must key its terminal rebuild off **`meta.live && s.suspended`** rather than an archive→active edge: in this case `archived` was false all along and never changes, and keying off it left the desktop showing "select to resume" over an already-running session. `resume-session` is **idempotent** (a live `s.pty` returns early) — two clients can ask at once and a client's `live` flag is always a moment stale, and a second spawn would strand the first PTY as an orphaned `claude` process nothing can reach or kill.
- **The desktop gets that push with the whole list; a phone gets it as a bare signal.** The phone *pages* its list (`query-sessions`, below) instead of holding the set, so shipping every session on every list change would defeat the paging — `remote.js`'s `onBroadcast` hook sends `sessions-changed` to remote clients with a `null` payload, meaning only "refetch your page". The renderer's own `sendToRenderer` payload is untouched; `syncSessions()` still reconciles against the full list.
- `query-sessions {tab, query, offset, limit}` → `{items, total, counts}` (`src/main/sessions.js`, pure logic in `session-query-lib.js`) — one page of the list, tab-filtered, searched and sorted newest-first in main, scoped to the open project. Rows are slimmer than `sessionList()`'s: no tracked-file array, which is the bulk of a row and which a phone's list doesn't draw. `limit` is clamped (default 30, max 200) so one request can never pull a whole archive. Search matching mirrors the desktop's Archived-tab search exactly (fold + whitespace-split ANDed terms, `.ext` matches by suffix) — the same semantics as `shared/name-match.js` and explorer's `search-names`, reimplemented CommonJS-side because main can't import the renderer's ESM.
- UI: **Settings dialog → "Remote access" group** (`src/renderer/remote-pane.js`): enable toggle, QR (`qrcode-generator` ESM via import map), paired-device list with revoke. It refreshes on the settings-btn click (additive `addEventListener`, since settings.js owns `.onclick`).

## mobile/ — Expo app (TypeScript, own package; NOT in electron-builder files)

`src/api/connection.ts` (ws client: req/res correlation, reconnect+backoff, ev emitter, `forwardPort()`), `src/api/pairing.ts` (QR parse + SecureStore). Navigation: after pairing, a bottom tab bar (`@react-navigation/bottom-tabs` + Ionicons, styled in App.tsx's `MainTabs`) hosts Sessions/Git/Files/Ports; the two full-screen terminals — `Terminal` (a Claude session) and `Console` (a run terminal) — sit above it in the root stack. Both render `src/components/XtermWebView.tsx`, the one xterm-in-a-WebView implementation (touch scrolling, fling, select/copy); a screen supplies the wire and the top bar. Project switching and running are deliberately **not** tabs: they're the two side actions bracketing the header — a square icon button at header-left opens `src/components/ProjectDrawer.tsx` (left slide-in: recent folders, **Switch window** when the machine has more than one open, and Unpair), and one at header-right opens `src/components/RunDrawer.tsx` (right slide-in: launch configs, tasks, terminals — see **Run panel**). The header *title* shows the active project's name (not the screen name — the tab bar already names the screen). Screens: **Welcome** (`src/screens/WelcomeScreen.tsx` — the landing page, in the **three states a phone can be in before it is driving a window**: *unpaired* (the desktop welcome page's dot mark, tagline and action list reshaped for touch, a primary **Scan QR code** button, the three pairing steps — it exists because the camera used to open itself on launch, which asks for a permission before saying what the app is; Pair sits behind that button as the second screen of the unpaired stack, with a back button); *connecting*; and *choosing a window* — the machine's open IDE windows, oldest first, each showing the project it has open and how long it has been up. They are one screen because they are one moment to the user, and because a machine only says how many windows it has **after** you have connected to it, so the pair prompt and the chooser cannot be separate destinations you navigate between. `App.tsx` orchestrates: connect → `list-instances` → **skip straight in when there is only one** (the common case keeps today's zero-tap flow; an empty list means a desktop too old to know the channel) → otherwise render the chooser. Picking a window re-dials it (`instanceEndpoints` in `src/api/instances.ts` narrows the paired endpoints to one window: the LAN port is the window's, the relay URL gains `&instance=`), except for the window that served the roster (`current: true`) — we are already on it. **The choice is never persisted**: an instance id lives only as long as its process, so every launch lists afresh. `ProjectDrawer` carries a **Switch window** row when more than one is open, which re-lists rather than reusing the launch roster), Pair (expo-camera scan), Sessions (Active/Archived/All tabs, newest-first, state-dot + name only; archive = `suspend-session`, restore = `resume-session`, delete = `kill-session` — the same channels the desktop panel uses. **The list is paged, never held whole**: each tab fetches a screenful at a time over `query-sessions` and asks for the next page on `onEndReached`, so a long archive is neither shipped nor rendered in full. The Archived tab carries a search bar (debounced 200ms, cleared when you leave the tab, as on the desktop) whose query is applied *in main* — only matching rows come over the wire. `sessions-changed` / `folder-changed` / reconnect-to-`ready` all refetch the currently-open window (`max(PAGE, loaded)`, so scrolling back up hits no hole), and a single `busy` ref drops a request that would race one in flight. `status` and `session-name` patch loaded rows in place. It also refetches on tab focus via `useFocusEffect` — the tab navigator keeps screens mounted, so mount effects never re-run — and polls every 5s while focused, as a safety net for pushes missed while the socket was down), Terminal (xterm.js in a WebView via CDN, base64-injected `pty-data`; own top bar with back/name/**Stop** (`\x1b` down `pty-input` — the desktop's interrupt key; enabled only while `state === 'working'`, since `interruptState` ignores the ESC in every other state) over a **"Commit N files"** bar backed by `session-diff-stat` + `commit-session`, refreshed off `session-meta`, with the desktop's commit-while-running confirm. Two things the desktop gets for free and this doesn't — see **Terminal scrollback** below), Git (at parity with the desktop pane: per-file stage/unstage/discard, stage-all + discard-all on the Changes header, and a branch sheet off the header button — `git-branches` fetched fresh on open, search doubles as the create field like the desktop, remote-only branches checked out by short name so git's DWIM makes a local tracking branch, delete offered only on non-current locals. Pull/push carry Ionicons arrows and the desktop's ahead/behind count badge — a green circle pinned to the button's top-right corner, ringed in the background colour (`.git-ahead-badge` in `src/styles/git.css`); it's a *sibling* of the button rather than a child, since Android clips children overflowing a radiused parent. Every destructive action — discard, discard-all, delete-branch — uses a destructive `Alert` in place of the desktop's two-click arm, there being no hover on touch. Discard-all loops `git-revert` over the *unstaged* files only, leaving staged ones alone, exactly as the desktop's Changes header does. A changed-file row is laid out like a **Files-tab row** so the two lists read as the same app — the desktop file icon, the filename, and the folder above it in muted text beneath — with the porcelain code as a tinted letter chip on the right (`?` renders as `U`, as in VS Code: a bare question mark beside a filename reads as a missing icon, not a status) and the stage/discard glyphs sized to a real 32px tap target. Sections with nothing in them are dropped rather than headed with a `(0)`, so a clean tree is one empty state instead of two barren headers. A pull that can't fast-forward or a push the remote rejects is handed to Claude, exactly as on the desktop: main already flags those with `needsMerge`, so the screen confirms and then calls `newSessionWithPrompt` (`src/api/session-prompt.ts` — the mobile twin of the renderer's `newSessionWithPrompt` + `pendingPrompts` drain) with the same merge prompt, operation phrase, and raw git error the desktop sends, then navigates to the new session's terminal. The Conflicts header carries the desktop's "Let Claude resolve" button into the same flow. **The prompt cannot be sent when `new-session` resolves** — Claude's TUI hasn't painted its input box, so the keystrokes land nowhere. `primePrompt` waits for the session's first `pty-data`, types after 1s, and sends Enter as a *separate* write 400ms later (bundling `\r` with the text submits before the TUI finishes ingesting a multi-line paste, firing the prompt half-typed) — the desktop's timings, for the desktop's reasons. Unlike the renderer, it can only subscribe *after* it learns the id, so a first chunk in that window would be missed; a 3s fallback types anyway, and a `sent` flag keeps the two triggers from double-typing. Commit message is optional: an empty one makes main stage-all and author it via Haiku — the `git-commit` contract the desktop pane relies on too — so the button stays enabled whenever the tree is dirty and spins while Haiku runs. Refetches on tab focus, since the tab navigator keeps the screen mounted. A **Changes / History** switch under the header swaps the body: History is `src/components/CommitHistory.tsx`, the desktop History tab on a phone — incoming commits previewed above the local log, an unpushed/pushed split driving the action (unpushed → `git-undo-commit`, which rewrites history and is offered with **no confirm**, nothing having left the machine and its changes coming back staged; pushed → `git-revert-commit` behind a destructive `Alert`, since it lands a new commit everyone will see), and tapping a row fetches that one patch on demand via `git-commit-diff`. **It is paged and searched in main, never held whole** — see the `git-log` note below), Files (browser + viewer/editor: a back button and tappable breadcrumbs over `list-dir`, rows carrying the **desktop's own file icons** and a chevron on folders, pull-to-refresh, and a live re-list on `tree-changed` / `folder-changed` / tab focus. Opening a file shows it **syntax-highlighted** (`src/components/CodeView.tsx`) with a pencil that flips to a plain monospace `TextInput`; Save (`write-text`) is enabled only while dirty, and leaving with unsaved edits prompts. Android's back button climbs a folder, then leaves the file, before backing out of the tab. It must **never** call `navigation.setOptions({title})` — in a bottom-tab navigator `title` feeds the *tab bar label* as well as the header, so the old code renamed the "Files" tab button to whatever folder you were in), Ports (fwd-open → `Linking.openURL`). See `mobile/README.md`.

## Shared desktop assets on the phone (generated)

The Files screen shows the same icons and the same colours as the desktop explorer, and neither is maintained twice. `mobile/` is a separate package with its own Metro root, so it cannot import `src/renderer/` — instead `scripts/gen-mobile-assets.mjs` (`npm run gen:mobile`) bakes the desktop's own tables and artwork into `mobile/src/generated/desktop-assets.ts`: `EXT_ICON`/`NAME_ICON` + every SVG in `src/renderer/file-icons/` (inlined, self-colored, painted by `react-native-svg`'s `SvgXml` in `src/components/FileIcon.tsx`), and `EXT_LANG` from `shared/highlight.js`. **`src/renderer/shared/file-icons.js` stays the single source** — after adding an icon or a language mapping, re-run the generator; `test/mobile-assets.test.mjs` fails the build if the checked-in output has drifted.

`CodeView` renders in a WebView because highlight.js emits HTML (XtermWebView's precedent: library from a CDN, since the phone has internet even when the desktop link is LAN-only). It pins the *same* hljs 11.11.1 + `vs2015` theme `index.html` pins, side-loads the same `dos`/`powershell` grammars missing from the common bundle, and ports `hlLines()` from `shared/highlight.js` so each line is a balanced row against a sticky gutter. A file over 400k chars renders as plain text — hljs is linear but the DOM it builds is not. If the CDN is unreachable, highlighting degrades to plain text rather than failing.

## Session handover (who's driving)

Two clients typing into one PTY is a mess, so a session has at most one holder. The phone claims it (`session-control {id, on:true}`, a `send` channel) for as long as its terminal screen is mounted; main records `s.controlledBy = deviceId` and pushes `session-control {id, controlled}` to the desktop, which **covers that session's terminal** — the xterm keeps consuming `pty-data` underneath, so the cover only hides it, and taking control back shows the current screen with no rebuild. While covered, `disableStdin` plus the cover element keep the desktop from typing into it. The authoritative gate is in main: while `controlledBy` is set, `pty-input` and `pty-resize` from anyone but the holder are dropped (`heldByAnotherDevice`), so a desktop merely *opening* the covered session can't reflow the PTY under the phone. The renderer's `fit()` also no-ops on a controlled session (the covered xterm keeps mirroring phone-sized output for instant takeover), and `setControlled(off)` re-fits the active session to snap the PTY back to desktop dimensions.

Releasing is deliberately looser than claiming — only a phone can claim, but anyone can release:

- the phone, on leaving the screen (`on:false`);
- the desktop, via **Take control** on the cover (a local `session-control` with no `deviceId` wins over whoever holds it);
- main, when the session is archived (no PTY left to drive);
- main, when the phone's **socket dies** — a locked phone or dropped Wi-Fi never sends `on:false`, so `ws-server` calls the injected `onDisconnect(deviceId)` → `releaseDeviceControl`, and the desktop uncovers. This is the one that matters: without it a vanished phone strands the session covered forever. It fires only when the device has no *other* live socket, so a reconnect doesn't yank the session away mid-use.

`sessionList()` carries `controlled`, so a reloaded renderer comes up covered correctly. Anything that wipes a container's children (`suspendSessionUI`, `rebuildTerminalUI`) must `clearControlled(s)` — the cover node goes with them, and a stale flag would suppress rebuilding it.

## Run panel (launch configs, tasks, and their terminals)

The phone's header-right button opens `src/components/RunDrawer.tsx` — a right slide-in mirroring `ProjectDrawer` — listing the `.vscode` launch configs, the tasks, and the terminals they opened.

**Nothing runs on the phone.** A terminal tab is a renderer thing (xterm + tab UI), so `run-config-start {kind, name}` resolves the specs in main and pushes them to the *desktop* renderer as `run-specs`; `toolbar.js` opens them through `runSpecInConsole` — the exact path a toolbar click takes. One code path, so the two ends can't drift, and tab reuse means a second tap is a rerun (the play icon flips to a restart glyph, as on the desktop). `run-config-stop {name}` kills the config's terminal(s), resolving a compound to its members like the toolbar's Stop. `${input:...}` configs are the one thing that can't run headlessly — the answers come from a modal the desktop renderer owns — so they come back as a plain error telling you to start them from the desktop.

**Main is the source of truth for what's open.** It used to store a bare pty per id and let the renderer keep the name/kind privately, which is why nothing else could see what was running. Each console entry now carries `{name, kind}` (the renderer sends both — naming still has one owner) and every open/close/restart pushes `terminals-changed`. That single event does double duty: it's the terminal list *and* the run state, since a launch config "runs" for exactly as long as its terminal is open (kind `config`, matched by name — the definition the desktop toolbar already used). `killConsole()` always pushes `term-exit`, unlike the pty's own `onExit` (which stays quiet for a terminal we deliberately replaced), so a stop from the phone closes the desktop tab too.

The run terminal itself (`src/screens/ConsoleTerminal.tsx`) attaches over `term-list` + `term-scrollback` (same bounded-tail + `seq` dedupe scheme as session scrollback, below) and types over `term-input`. Two deliberate non-features:

- **It does not resize the PTY.** That terminal is also open in a desktop tab sized for the desktop; a phone-width resize would reflow the view under whoever is sitting at the machine. The PTY keeps its dimensions and xterm soft-wraps to the phone. Log output — what a launch config produces — reads fine; a full-screen TUI would look loose.
- **It does not claim the terminal.** Unlike a Claude session (see **Session handover**), a run terminal is a shared shell: both ends stay live, like two panes onto one tmux window.

**You cannot select xterm's own output on touch, and lifting its `user-select:none` does not fix it** — xterm's pointer handlers `preventDefault` in order to run its *own* mouse selection, so the platform's long-press gesture never fires. (That was the first attempt; the Select button did nothing.) Select mode instead lays `#sel` — a plain `<pre>` holding `bufferText()` — over the terminal: ordinary DOM text, so native handles, the magnifier, the system Copy menu and native scrolling all behave. It's a **snapshot**; re-rendering it as live output arrived would wipe a selection mid-drag, so the terminal keeps receiving underneath and the overlay stays still until you leave the mode. `__copy()` takes the selection, else the whole buffer.

## Terminal scrollback (mobile)

The desktop's xterm instance lives as long as the session, which hides two problems the phone has to solve explicitly:

- **Nothing to scroll back to.** Leaving the session screen destroys the WebView and its whole buffer. So `sessions.js` keeps a bounded tail of each live session's PTY output (`s.scroll`, 200k chars, whole chunks dropped off the front, reset on every `spawnPty` since a fresh PTY redraws anyway) and serves it over `session-scrollback`. The screen replays that snapshot before going live. Every `pty-data` carries a `seq`, and the snapshot reports the last `seq` it contains, so chunks that raced the fetch are dropped instead of printed twice — ordering on the single socket guarantees nothing older is missed.
- **Drags don't scroll.** xterm stacks `.xterm-screen` **over** the scrollable `.xterm-viewport`, so a touch on the terminal body never reaches the scroll container. The page drives `viewport.scrollTop` from the touch itself. Do **not** go back to `term.scrollLines()`: it snaps the drag to whole rows and re-renders synchronously inside the touch handler, which is what made it stutter. Pixel-accurate scrollTop writes, coalesced to one per frame, let xterm render off its own scroll event (the desktop scrollbar's path); a velocity-decay fling on touchend supplies the momentum a native list would have.

## Gotchas

- `git-log` takes `{limit, skip, query}` and returns `{commits, incoming, hasMore}`. **The desktop calls it with no args**, so the defaults must keep reproducing its old behavior (newest 100 + incoming) — don't make an option required. Browsing pages straight out of git (`-n`/`--skip`), but a *search* can't: the query matches author and hash as well as the subject, which `git log --grep` can't express, so main scans the log in 2000-commit chunks and filters each with `filterCommits` — the same matcher the desktop History tab uses — stopping as soon as the page is full. Both paths over-fetch one commit past the page so `hasMore` is free (`pageCommits`). `incoming` rides along only on an unfiltered first page: repeating it per page would duplicate rows, and a search asks about local history, not about what a pull would bring in.
- **The LAN link is still plain `ws://`** — UI copy says trusted-network only. The relay link is `wss://` (TLS terminated at Render), so that half is encrypted in transit; but the relay operator sits in the middle of it. Frames are opaque to the relay and auth is verified end-to-end by the desktop, yet nothing is end-to-end *encrypted*. `wss://` on the LAN + cert pinning in the QR remains the known fast-follow.
- Revoking a device does not terminate its already-open socket; it only blocks the next auth.
- Tests: `test/remote-{protocol,auth-lib,ws-server,relay,relay-route,relay-client,http-proxy,keepalive}.test.js` — real ws clients/http servers in-process, no Electron. `remote-relay-client.test.js` is the end-to-end one: a desktop dialling out, a phone dialling in, and a browser fetching a **loopback-only** dev server through the tunnel. `remote-relay.test.js` covers the multi-window room (siblings coexisting, a phone naming a window, a phone naming none, one window leaving without dropping its siblings' phones); the roster's pure half is `test/instance-lib.test.js` (`liveInstances`, `upsertInstance`). Locale parity test requires every `remote.*` string in all 5 locales.
- **Never bulk-rewrite locale files (or any non-ASCII file) with PowerShell Get-Content/Set-Content** — PS 5.1 reads ANSI by default and mojibakes UTF-8; it also BOM-stamps. Use the Edit tool.
