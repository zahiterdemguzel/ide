# Remote access (mobile companion)

Developers can drive the IDE from a phone: QR pairing, project switching, Claude sessions, git, file editing, terminals, and dev-server port forwarding. Three parts: the Electron-free `server/` socket bridge, thin glue in `src/main/remote*.js`, and the Expo app in `mobile/`. Design rule (user requirement): **the backend only bridges sockets/communications** — all IDE logic and auth verification live in the desktop app; `server/` never inspects business payloads.

## server/ — the socket bridge (plain Node, CommonJS, linted like src/main)

- `protocol.js` — pure: message parse/validate, `REMOTE_CHANNELS` **allowlist** (`req` mirrors `ipcMain.handle`, `send` mirrors `ipcMain.on`), `REMOTE_EVENTS` (renderer pushes forwarded to phones), `fwd-open`/`fwd-close` shapes. To expose a channel remotely you must add it here — an existing handler is not enough. Excluded on purpose: `open-folder` (native dialog), clipboard, `db-*`, runners, assets.
- `auth-lib.js` — pure: single-use 5-min pairing tokens (3 wrong guesses invalidates), device credentials (32-byte tokens, only sha256 hashes persisted), injected-store CRUD.
- `ws-server.js` — `startRemoteServer({invoke, deviceStore, forward, ...})` on `0.0.0.0:<ephemeral>` (`ws` dep); hello → pair/auth → ready state machine; 15s heartbeat; routes `fwd-open/close` to the injected `forward` hook (absent → `forwarding-disabled`).
- `http-proxy.js` + `http-proxy-lib.js` — dev-server port forwarding: one LAN listener per forwarded port piping to `127.0.0.1:<target>`; auth = one-time `?_ideauth=` URL token → 302 that sets an HttpOnly cookie (pure decisions in the lib); raw-socket Upgrade passthrough keeps Vite/webpack HMR websockets working.
- `relay.js` + `index.js` + its own `package.json` — standalone cloud relay (deploy: root dir `server`, build `npm install`, start `node index.js`, listens on `process.env.PORT`). Pure frame forwarding between one desktop socket and N mobile sockets per room using `{c:<clientId>, d:<frame>}` envelopes; never parses `d`. The desktop-side relay client (connecting out to a relay) is **not wired yet**.

## Desktop glue (src/main)

- `remote-bridge.js` — subsystems register via its `handle`/`on` (aliased `bridge.handle`/`bridge.on`) instead of `ipcMain`; registers with both Electron and a registry. Remote calls invoke the same fn with a stub event `{sender:null, remote:true, deviceId}`. Bridged: `repo.js`, `sessions.js`, `git.js`, `explorer.js`, `consoles.js`, `session-commit.js`, and *part of* `run-configs.js` (`get-run-configs` / `run-config-start` / `run-config-stop`; the desktop-only `run-config` stays on raw `ipcMain` — see **Run panel**). Kept on raw `ipcMain`: `db.js`, `runners.js`, `onboarding-store.js`, `window.js`, `remote.js` itself.
- `remote.js` — desktop control (never remote-callable): enable/disable the embedded server, `ide://pair?v=1&host=…&port=…&tk=…` QR payloads (prefers private-range IPv4), device store in `remote-devices.json` under `sharedDataDir`, forwards `sendToRenderer` pushes via the `onBroadcast` seam in `window.js`, owns the port-forward proxy map (proxies die on disable/quit).
- A remote `open-folder-path` also pushes `folder-changed` so the desktop UI follows the switch (repo path is global — remote switch switches the desktop too, by design).
- **The phone is an input device for the desktop: main owns the state, clients are views of it.** Any action that changes the *shape* of the session list (new / suspend / resume / kill / evict) ends with `broadcastSessions()` in `src/main/sessions.js` — a `sessions-changed` push carrying the whole list, so every client reconciles against main instead of trusting its own optimistic edit. The desktop renderer's `syncSessions()` adopts rows created elsewhere (a `live` flag in the payload says whether to attach a real terminal or the "restore me" placeholder), mirrors archive/restore without re-issuing the IPC (main already killed/respawned the PTY), and drops rows deleted elsewhere. Echoes of a client's own change reconcile to a no-op. **A new list-shape mutation must call `broadcastSessions()` or the other client goes stale** — that was the original bug: the mobile list only listened to `session-meta`, which fires for tracked-file churn and never for archive/delete.
- **The desktop gets that push with the whole list; a phone gets it as a bare signal.** The phone *pages* its list (`query-sessions`, below) instead of holding the set, so shipping every session on every list change would defeat the paging — `remote.js`'s `onBroadcast` hook sends `sessions-changed` to remote clients with a `null` payload, meaning only "refetch your page". The renderer's own `sendToRenderer` payload is untouched; `syncSessions()` still reconciles against the full list.
- `query-sessions {tab, query, offset, limit}` → `{items, total, counts}` (`src/main/sessions.js`, pure logic in `session-query-lib.js`) — one page of the list, tab-filtered, searched and sorted newest-first in main, scoped to the open project. Rows are slimmer than `sessionList()`'s: no tracked-file array, which is the bulk of a row and which a phone's list doesn't draw. `limit` is clamped (default 30, max 200) so one request can never pull a whole archive. Search matching mirrors the desktop's Archived-tab search exactly (fold + whitespace-split ANDed terms, `.ext` matches by suffix) — the same semantics as `shared/name-match.js` and explorer's `search-names`, reimplemented CommonJS-side because main can't import the renderer's ESM.
- UI: **Settings dialog → "Remote access" group** (`src/renderer/remote-pane.js`): enable toggle, QR (`qrcode-generator` ESM via import map), paired-device list with revoke. It refreshes on the settings-btn click (additive `addEventListener`, since settings.js owns `.onclick`).

## mobile/ — Expo app (TypeScript, own package; NOT in electron-builder files)

`src/api/connection.ts` (ws client: req/res correlation, reconnect+backoff, ev emitter, `forwardPort()`), `src/api/pairing.ts` (QR parse + SecureStore). Navigation: after pairing, a bottom tab bar (`@react-navigation/bottom-tabs` + Ionicons, styled in App.tsx's `MainTabs`) hosts Sessions/Git/Files/Ports; the two full-screen terminals — `Terminal` (a Claude session) and `Console` (a run terminal) — sit above it in the root stack. Both render `src/components/XtermWebView.tsx`, the one xterm-in-a-WebView implementation (touch scrolling, fling, select/copy); a screen supplies the wire and the top bar. Project switching and running are deliberately **not** tabs: they're the two side actions bracketing the header — a square icon button at header-left opens `src/components/ProjectDrawer.tsx` (left slide-in: recent folders + Unpair), and one at header-right opens `src/components/RunDrawer.tsx` (right slide-in: launch configs, tasks, terminals — see **Run panel**). The header *title* shows the active project's name (not the screen name — the tab bar already names the screen). Screens: **Welcome** (`src/screens/WelcomeScreen.tsx` — the landing page an unpaired phone opens on: the desktop welcome page's dot mark, tagline and action list reshaped for touch, a primary **Scan QR code** button, the three pairing steps, and the trusted-network note. It exists because the camera used to open itself on launch, which asks for a permission before saying what the app is; Pair now sits behind that button as the second screen of the unpaired stack, with a back button), Pair (expo-camera scan), Sessions (Active/Archived/All tabs, newest-first, state-dot + name only; archive = `suspend-session`, restore = `resume-session`, delete = `kill-session` — the same channels the desktop panel uses. **The list is paged, never held whole**: each tab fetches a screenful at a time over `query-sessions` and asks for the next page on `onEndReached`, so a long archive is neither shipped nor rendered in full. The Archived tab carries a search bar (debounced 200ms, cleared when you leave the tab, as on the desktop) whose query is applied *in main* — only matching rows come over the wire. `sessions-changed` / `folder-changed` / reconnect-to-`ready` all refetch the currently-open window (`max(PAGE, loaded)`, so scrolling back up hits no hole), and a single `busy` ref drops a request that would race one in flight. `status` and `session-name` patch loaded rows in place. It also refetches on tab focus via `useFocusEffect` — the tab navigator keeps screens mounted, so mount effects never re-run — and polls every 5s while focused, as a safety net for pushes missed while the socket was down), Terminal (xterm.js in a WebView via CDN, base64-injected `pty-data`; own top bar with back/name/**Stop** (`\x1b` down `pty-input` — the desktop's interrupt key; enabled only while `state === 'working'`, since `interruptState` ignores the ESC in every other state) over a **"Commit N files"** bar backed by `session-diff-stat` + `commit-session`, refreshed off `session-meta`, with the desktop's commit-while-running confirm. Two things the desktop gets for free and this doesn't — see **Terminal scrollback** below), Git (at parity with the desktop pane: per-file stage/unstage/discard, stage-all + discard-all on the Changes header, and a branch sheet off the header button — `git-branches` fetched fresh on open, search doubles as the create field like the desktop, remote-only branches checked out by short name so git's DWIM makes a local tracking branch, delete offered only on non-current locals. Pull/push carry Ionicons arrows and the desktop's ahead/behind count badge — a green circle pinned to the button's top-right corner, ringed in the background colour (`.git-ahead-badge` in `src/styles/git.css`); it's a *sibling* of the button rather than a child, since Android clips children overflowing a radiused parent. Every destructive action — discard, discard-all, delete-branch — uses a destructive `Alert` in place of the desktop's two-click arm, there being no hover on touch. Discard-all loops `git-revert` over the *unstaged* files only, leaving staged ones alone, exactly as the desktop's Changes header does. A changed-file row is laid out like a **Files-tab row** so the two lists read as the same app — the desktop file icon, the filename, and the folder above it in muted text beneath — with the porcelain code as a tinted letter chip on the right (`?` renders as `U`, as in VS Code: a bare question mark beside a filename reads as a missing icon, not a status) and the stage/discard glyphs sized to a real 32px tap target. Sections with nothing in them are dropped rather than headed with a `(0)`, so a clean tree is one empty state instead of two barren headers. A pull that can't fast-forward or a push the remote rejects is handed to Claude, exactly as on the desktop: main already flags those with `needsMerge`, so the screen confirms and then calls `newSessionWithPrompt` (`src/api/session-prompt.ts` — the mobile twin of the renderer's `newSessionWithPrompt` + `pendingPrompts` drain) with the same merge prompt, operation phrase, and raw git error the desktop sends, then navigates to the new session's terminal. The Conflicts header carries the desktop's "Let Claude resolve" button into the same flow. **The prompt cannot be sent when `new-session` resolves** — Claude's TUI hasn't painted its input box, so the keystrokes land nowhere. `primePrompt` waits for the session's first `pty-data`, types after 1s, and sends Enter as a *separate* write 400ms later (bundling `\r` with the text submits before the TUI finishes ingesting a multi-line paste, firing the prompt half-typed) — the desktop's timings, for the desktop's reasons. Unlike the renderer, it can only subscribe *after* it learns the id, so a first chunk in that window would be missed; a 3s fallback types anyway, and a `sent` flag keeps the two triggers from double-typing. Commit message is optional: an empty one makes main stage-all and author it via Haiku — the `git-commit` contract the desktop pane relies on too — so the button stays enabled whenever the tree is dirty and spins while Haiku runs. Refetches on tab focus, since the tab navigator keeps the screen mounted. A **Changes / History** switch under the header swaps the body: History is `src/components/CommitHistory.tsx`, the desktop History tab on a phone — incoming commits previewed above the local log, an unpushed/pushed split driving the action (unpushed → `git-undo-commit`, which rewrites history and is offered with **no confirm**, nothing having left the machine and its changes coming back staged; pushed → `git-revert-commit` behind a destructive `Alert`, since it lands a new commit everyone will see), and tapping a row fetches that one patch on demand via `git-commit-diff`. **It is paged and searched in main, never held whole** — see the `git-log` note below), Files (browser + viewer/editor: a back button and tappable breadcrumbs over `list-dir`, rows carrying the **desktop's own file icons** and a chevron on folders, pull-to-refresh, and a live re-list on `tree-changed` / `folder-changed` / tab focus. Opening a file shows it **syntax-highlighted** (`src/components/CodeView.tsx`) with a pencil that flips to a plain monospace `TextInput`; Save (`write-text`) is enabled only while dirty, and leaving with unsaved edits prompts. Android's back button climbs a folder, then leaves the file, before backing out of the tab. It must **never** call `navigation.setOptions({title})` — in a bottom-tab navigator `title` feeds the *tab bar label* as well as the header, so the old code renamed the "Files" tab button to whatever folder you were in), Ports (fwd-open → `Linking.openURL`). See `mobile/README.md`.

## Shared desktop assets on the phone (generated)

The Files screen shows the same icons and the same colours as the desktop explorer, and neither is maintained twice. `mobile/` is a separate package with its own Metro root, so it cannot import `src/renderer/` — instead `scripts/gen-mobile-assets.mjs` (`npm run gen:mobile`) bakes the desktop's own tables and artwork into `mobile/src/generated/desktop-assets.ts`: `EXT_ICON`/`NAME_ICON` + every SVG in `src/renderer/file-icons/` (inlined, self-colored, painted by `react-native-svg`'s `SvgXml` in `src/components/FileIcon.tsx`), and `EXT_LANG` from `shared/highlight.js`. **`src/renderer/shared/file-icons.js` stays the single source** — after adding an icon or a language mapping, re-run the generator; `test/mobile-assets.test.mjs` fails the build if the checked-in output has drifted.

`CodeView` renders in a WebView because highlight.js emits HTML (XtermWebView's precedent: library from a CDN, since the phone has internet even when the desktop link is LAN-only). It pins the *same* hljs 11.11.1 + `vs2015` theme `index.html` pins, side-loads the same `dos`/`powershell` grammars missing from the common bundle, and ports `hlLines()` from `shared/highlight.js` so each line is a balanced row against a sticky gutter. A file over 400k chars renders as plain text — hljs is linear but the DOM it builds is not. If the CDN is unreachable, highlighting degrades to plain text rather than failing.

## Session handover (who's driving)

Two clients typing into one PTY is a mess, so a session has at most one holder. The phone claims it (`session-control {id, on:true}`, a `send` channel) for as long as its terminal screen is mounted; main records `s.controlledBy = deviceId` and pushes `session-control {id, controlled}` to the desktop, which **covers that session's terminal** — the xterm keeps consuming `pty-data` underneath, so the cover only hides it, and taking control back shows the current screen with no rebuild. While covered, `disableStdin` plus the cover element keep the desktop from typing into it.

Releasing is deliberately looser than claiming — only a phone can claim, but anyone can release:

- the phone, on leaving the screen (`on:false`);
- the desktop, via **Take control** on the cover (a local `session-control` with no `deviceId` wins over whoever holds it);
- main, when the session is archived (no PTY left to drive);
- main, when the phone's **socket dies** — a locked phone or dropped Wi-Fi never sends `on:false`, so `ws-server` calls the injected `onDisconnect(deviceId)` → `releaseDeviceControl`, and the desktop uncovers. This is the one that matters: without it a vanished phone strands the session covered forever. It fires only when the device has no *other* live socket, so a reconnect doesn't yank the session away mid-use.

`sessionList()` carries `controlled`, so a reloaded renderer comes up covered correctly. Anything that wipes a container's children (`suspendSessionUI`, `rebuildTerminalUI`) must `clearControlled(s)` — the cover node goes with them, and a stale flag would suppress rebuilding it.

## Run panel (launch configs, tasks, and their terminals)

The phone's header-right button opens `src/components/RunDrawer.tsx` — a right slide-in mirroring `ProjectDrawer` — listing the `.vscode` launch configs, the tasks, and the terminals they opened.

**Nothing runs on the phone.** A terminal tab is a renderer thing (xterm + tab UI), so `run-config-start {kind, name}` resolves the specs in main and pushes them to the *desktop* renderer as `run-specs`; `toolbar.js` opens them through `runSpecInConsole` — the exact path a toolbar click takes. One code path, so the two ends can't drift, and tab reuse means a second tap is a rerun (the play icon flips to a restart glyph, as on the desktop). `run-config-stop {name}` kills the config's terminal(s), resolving a compound to its members like the toolbar's Stop. `${input:...}` configs are the one thing that can't run headlessly — the answers come from a modal the desktop renderer owns — so they come back as a plain error telling you to start them from the desktop.

**Main is the source of truth for what's open.** It used to store a bare pty per id and let the renderer keep the name/kind privately, which is why nothing else could see what was running. Each console entry now carries `{name, kind}` (the renderer sends both — naming still has one owner) and every open/close/restart pushes `terminals-changed`. That single event does double duty: it's the terminal list *and* the run state, since a launch config "runs" for exactly as long as its terminal is open (kind `config`, matched by name — the definition the desktop toolbar already used). `killConsole()` always pushes `term-exit`, unlike the pty's own `onExit` (which stays quiet for a terminal we deliberately replaced), so a stop from the phone closes the desktop tab too.

The run terminal itself (`src/screens/ConsoleTerminal.tsx`) attaches over `term-list` + `term-scrollback` (same bounded-tail + `seq` dedupe scheme as session scrollback, below) and types over `term-input`. Two deliberate non-features:

- **It does not resize the PTY.** That terminal is also open in a desktop tab sized for the desktop; a phone-width resize would reflow the view under whoever is sitting at the machine. The PTY keeps its dimensions and xterm soft-wraps to the phone. Log output — what a launch config produces — reads fine; a full-screen TUI would look loose.
- **It does not claim the terminal.** Unlike a Claude session (see **Session handover**), a run terminal is a shared shell: both ends stay live, like two panes onto one tmux window.

**You cannot select xterm's own output on touch, and lifting its `user-select:none` does not fix it** — xterm's pointer handlers `preventDefault` in order to run its *own* mouse selection, so the platform's long-press gesture never fires. (That was the first attempt; the Select button did nothing.) Select mode instead lays `#sel` — a plain `<pre>` holding `bufferText()` — over the terminal: ordinary DOM text, so native handles, the magnifier, the system Copy menu and native scrolling all behave. It's a **snapshot**; re-rendering it as live output arrived would wipe a selection mid-drag, so the terminal keeps receiving underneath and the overlay stays still until you leave the mode. `__copy()` takes the selection, else the whole buffer.

## Terminal scrollback (mobile)

The desktop's xterm instance lives as long as the session, which hides two problems the phone has to solve explicitly:

- **Nothing to scroll back to.** Leaving the session screen destroys the WebView and its whole buffer. So `sessions.js` keeps a bounded tail of each live session's PTY output (`s.scroll`, 200k chars, whole chunks dropped off the front, reset on every `spawnPty` since a fresh PTY redraws anyway) and serves it over `session-scrollback`. The screen replays that snapshot before going live. Every `pty-data` carries a `seq`, and the snapshot reports the last `seq` it contains, so chunks that raced the fetch are dropped instead of printed twice — ordering on the single socket guarantees nothing older is missed.
- **Drags don't scroll.** xterm stacks `.xterm-screen` **over** the scrollable `.xterm-viewport`, so a touch on the terminal body never reaches the scroll container. The page drives `viewport.scrollTop` from the touch itself. Do **not** go back to `term.scrollLines()`: it snaps the drag to whole rows and re-renders synchronously inside the touch handler, which is what made it stutter. Pixel-accurate scrollTop writes, coalesced to one per frame, let xterm render off its own scroll event (the desktop scrollbar's path); a velocity-decay fling on touchend supplies the momentum a native list would have.

## Gotchas

- `git-log` takes `{limit, skip, query}` and returns `{commits, incoming, hasMore}`. **The desktop calls it with no args**, so the defaults must keep reproducing its old behavior (newest 100 + incoming) — don't make an option required. Browsing pages straight out of git (`-n`/`--skip`), but a *search* can't: the query matches author and hash as well as the subject, which `git log --grep` can't express, so main scans the log in 2000-commit chunks and filters each with `filterCommits` — the same matcher the desktop History tab uses — stopping as soon as the page is full. Both paths over-fetch one commit past the page so `hasMore` is free (`pageCommits`). `incoming` rides along only on an unfiltered first page: repeating it per page would duplicate rows, and a search asks about local history, not about what a pull would bring in.
- Connection is plain `ws://` — UI copy says trusted-network only. `wss://` + cert pinning in the QR is the known fast-follow.
- Revoking a device does not terminate its already-open socket; it only blocks the next auth.
- Tests: `test/remote-{protocol,auth-lib,ws-server,relay,http-proxy}.test.js` — real ws clients/http servers in-process, no Electron. Locale parity test requires every `remote.*` string in all 5 locales.
- **Never bulk-rewrite locale files (or any non-ASCII file) with PowerShell Get-Content/Set-Content** — PS 5.1 reads ANSI by default and mojibakes UTF-8; it also BOM-stamps. Use the Edit tool.
