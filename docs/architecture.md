# Architecture

Standard Electron main/preload/renderer split. The renderer is pure UI with **no** Node or OS access — everything that touches the OS lives in the main process and is reached over IPC declared in `preload/`.

```
renderer/  ──IPC(preload/)──►  main/  ──►  node-pty / git / http hook server
  (UI)                          (OS)
```

## The main process — three subsystems

1. **PTY manager.** One `pty.spawn(claude, ...)` per session, keyed by a generated UUID. The UUID is passed as `--session-id` so it round-trips inside hook payloads — that is how a hook event is matched back to its sidebar row. Sessions live in the `sessions` Map; `pty-input` / `pty-resize` / `kill-session` operate on it by id. Each record stores the **`repo`** it was created in (`getRepoPath()` at `new-session`), so a session is bound to its project: it spawns/resumes with that folder as `cwd` regardless of which folder is currently open, the snapshot persists it, and the renderer shows only the open folder's sessions — switching folders (`setSessionsRepo()`) re-filters the list (on top of the active/archived tab) without tearing down the other projects' live terminals. **Archiving** a session (the sidebar ×) doesn't kill it — `suspend-session` kills only the Claude PTY to free its resources while keeping the session record (and all its tracked-file state) with `pty: null` and `suspended: true`; the `onExit` handler sees `suspended` and skips the usual teardown. **Restoring** it (`resume-session`) respawns with `claude --resume <id>`, so the conversation continues under the same id and `recordSessionActivity()` keeps accumulating `edits`/`fileOps` against the same record — any uncommitted work tracked before archiving survives and stays committable. The renderer disposes the xterm terminal on archive and rebuilds it on restore (`attachTerminal()`), keeping its `files` list intact throughout. Each session record also accumulates **what the agent did**: `firstPrompt` (its first `UserPromptSubmit`); `edits` — a `Map<absPath, op[]>` recording every `PostToolUse` of a file tool (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) as a replayable op (`editOp()`); and `fileOps` — a `Map<absPath, 'add'|'delete'>` of **path-level changes a text op can't express** (a binary file a `Bash`/MCP tool created, or a file it renamed/moved/deleted), filled by diffing the git working tree across a tool call (see [Tracking filesystem changes](#tracking-filesystem-changes)). `recordSessionActivity()` fills all of these from the hook stream and pushes a `session-meta` event to the renderer (the file list is `trackedFiles()` = the union of `edits` and `fileOps` keys); `commit-session` replays the ops and folds the path-level changes to commit just that session's work (see [Per-session commit](#per-session-commit)). On the first prompt it also fires `generateSessionName()` — a Haiku call via the shared `runHaiku()` helper (prompt capped at 2000 chars) asking for a 2-4 word title, pushed to the renderer as `session-name`. See [Haiku generation](#haiku-generation) for how `runHaiku()` reaches the model.
2. **Hook → status server.** A Node `http` server on a random `127.0.0.1` port. See [status-detection.md](status-detection.md).
3. **Git.** Plain `git` porcelain via `execFile` in `repoPath` (no git library). Every call goes through the shared `git()` helper, which prepends `-c core.quotePath=false` (so non-ASCII paths come back verbatim and round-trip to `add`/`diff` instead of being C-quoted), sets `GIT_TERMINAL_PROMPT=0` (a remote that wants credentials with no helper fails fast instead of hanging on an unanswerable prompt), and applies a 120s `timeout` backstop so a stuck network op can't wedge the UI. `git status --porcelain=v1` is parsed into staged/unstaged lists **and a separate `conflicts` list** — unmerged entries (`DD/AU/UD/UA/DU/AA/UU`, both columns set) are pulled out so they aren't double-listed with wrong-direction stage/unstage actions; the renderer shows them in a dedicated Conflicts section (click to view, ✓ to `git add` once resolved). stage = `git add`, unstage = `git reset HEAD` (falls back to `git rm --cached` for a repo with no commits); also `git commit -m` (called with an empty message, it stages-all-if-needed then asks Haiku to author the message from the staged diff — `generateCommitMessage()`, the shared `runHaiku()` helper in `claude.js` (see [Haiku generation](#haiku-generation)), capped at 12000 chars in / 1000 out; the message used is returned to the renderer), `git-undo` (`git reset --soft HEAD~1` — undo the last commit, keep its changes staged), `git push` (a branch with no upstream fails with "has no upstream branch", so the handler retries once with `push -u origin HEAD` — the common first-push case), `git-fetch`, and `git-pull` (`pull --ff-only`, so a pull that would need a merge fails cleanly instead of opening an editor with no tty or leaving conflicts — the user can then hand the merge to a Claude session). `commit-session` commits **only the hunks a single session edited** — see [Per-session commit](#per-session-commit). `git-diff` returns a file's unified diff (`--cached` for staged, `--no-index /dev/null <file>` for untracked) for the center-pane diff viewer. `git-revert` discards a file's changes — `git clean` for untracked, `git restore --staged --worktree` otherwise. The **History tab** is backed by `git-log` (last 100 commits, unit-separator-delimited `hash/short/subject/author/relDate` fields), `git-commit-diff` (`git show --format=<hash>` — the bare patch for the center diff viewer), and `git-revert-commit` (`git revert --no-edit <hash>` — a non-destructive revert commit, safe on pushed history). `open-folder` runs `git rev-parse --show-toplevel` (`repoRoot()`) so porcelain paths and add/reset line up regardless of which subfolder the user picks. The chosen folder is written to `last-folder.txt` in the shared data dir (`instance.js`'s `sharedDataDir`, not the per-instance profile — so it persists across runs and is shared by all instances) and reloaded on startup (`loadLastFolder()`), so the app reopens the last repo automatically; the renderer picks it up via its initial `refreshGit()`. **Branches**: `gitStatus` also returns the checked-out branch name (`currentBranch()` = `git rev-parse --abbrev-ref HEAD`, `'HEAD'` when detached) for the header indicator; `git-branches` lists local branches (`git for-each-ref --sort=-committerdate refs/heads`, most-recent first) plus the current one for the selector popover, `git-checkout` switches to a branch (`git checkout <branch>`, failing cleanly when the worktree would be clobbered), and `git-create-branch` creates and switches to a new branch off the current HEAD (`git checkout -b <branch>`, surfacing git's rejection of invalid ref names).

## Haiku generation

`runHaiku()` (in `claude.js`) backs every short generation (commit messages, session titles). It has two paths:

1. **Fast path — direct Messages API.** A one-shot `claude -p` cold-starts the whole CLI (Node + plugins + hooks + MCP) and sends its full agentic system prompt + tool definitions on every call — ~5s for a one-line commit message. Instead `runHaiku()` calls `https://api.anthropic.com/v1/messages` directly with the OAuth token Claude Code already stores in `~/.claude/.credentials.json` (`claudeAiOauth.accessToken`), a one-line system prompt, and no tools: **~1s, no process boot, no new dependency or API key.** Subscription (OAuth) tokens are only accepted when the **first system block is the Claude Code identity string** (`"You are Claude Code, …"`) — the same one the CLI sends; a second system block carries the "output only the requested text" rule. Sent with headers `authorization: Bearer <token>`, `anthropic-version: 2023-06-01`, `anthropic-beta: oauth-2025-04-20`.
2. **Fallback — the CLI.** The OAuth token is short-lived; `readOauthToken()` returns null when it's missing or within 60s of `expiresAt`, and any non-200/transport error also yields null. In those cases `runHaiku()` falls back to the original one-shot `claude -p --model haiku` (prompt over stdin). That path is slow but always works **and refreshes the stored token on disk**, so the fast path heals itself on the next call. Both callers already treat a null result as "Haiku unavailable" and fall back further (raw prompts / `firstPrompt`).

The model id is pinned (`claude-haiku-4-5-20251001`) because the raw API needs a full id, not the `haiku` alias; if it ever 404s the call returns null and the CLI fallback (which does accept the alias) takes over.

## Per-session commit

Each session is tracked independently so its work can be committed on its own, **even when two sessions edit the same file** — each commit then contains only the lines that session changed.

**Tracking (record).** Every hook payload carries the `--session-id` we spawned with, so `recordSessionActivity()` can attribute it. Per session it keeps:

- `firstPrompt` — the first `UserPromptSubmit`, used **only** as the seed for the Haiku-generated session title (set once).
- `currentPrompt` / `pendingMsgs` — drive the **commit message** so a session can commit many times. `currentPrompt` is the latest `UserPromptSubmit`; when an edit is then recorded, that prompt is appended (deduped) to `pendingMsgs`. A per-session commit is messaged from `pendingMsgs.join('\n\n')` (so the 2nd, 3rd, … commit each describe their own work, not the first prompt forever), then `pendingMsgs` is cleared on a successful commit — or on a revert that empties `edits`. A prompt that produced no edit (e.g. a question) never enters `pendingMsgs`.
- `edits` — a `Map<absPath, op[]>`. Every `PostToolUse` of a file tool becomes a *replayable op* via `editOp()`: `Write` → `{t:'write', content}`, `Edit` → `{t:'edit', old, new, all}`, `MultiEdit` → `{t:'multi', edits}`, `NotebookEdit` → `{t:'opaque'}` (can't be replayed as text). A `.gitignore`d file is **never** recorded here — `recordSessionActivity()` runs `isIgnored()` (`git check-ignore`) on the edited path first and skips it, so a session editing an ignored file (a log, `.env`, a build artifact) never tracks or commits it. This mirrors the filesystem-diff path, which already excludes ignored files because they don't appear in `git status`.
- `fileOps` — a `Map<absPath, 'add'|'delete'>` of path-level changes no text op can express (a binary a `Bash`/MCP tool produced, or a rename/move/delete), filled by diffing the working tree across each tool call. See [Tracking filesystem changes](#tracking-filesystem-changes).

**Commit (replay).** `commit-session(id)` rebuilds, for each touched file, *only this session's version*:

1. `replayEdits(base, ops)` replays the session's ops onto the file's committed (`HEAD`) contents — `base` comes from `git show HEAD:<path>` (empty for a new file). String replacement, in order. It returns `{content, clean}`; `clean` is false if an `old_string` is no longer present (the other session moved/overwrote it) or the op was opaque.
2. If `clean`, the synthesized `content` is committed. If not, that file falls back to its **whole current working-tree contents** (best effort — overlapping same-line edits are inherently ambiguous).
   Then the session's `fileOps` are folded in as extra commit entries (skipping any path already covered by an edit): an `'add'` reads the file's **current bytes as a `Buffer`** (so binary stays intact) and an `'delete'` becomes a removal entry. This is what gets binary files, renames/moves, and deletions into the per-session commit alongside the text hunks.

   **Commit message.** A session's raw prompts are conversational ("now make the button bigger"), not a commit message, so `generateCommitMessage(prompts)` runs them through a Haiku call via `runHaiku()` (same pattern as `generateSessionName`, prompts capped at 6000 chars) asking for an imperative subject + optional body. The input is `pendingMsgs` (the prompts whose edits this commit contains), falling back to `firstPrompt`. If Haiku is unavailable it falls back to the joined raw prompts, then `firstPrompt`, then `session <id8>`; the final message is capped at 1000 chars. This call adds latency, so the renderer's commit button shows a spinner and blocks re-clicks until it resolves (see [Renderer](#renderer)).
3. `commitBlobs(entries, msg)` commits those entries **without touching the working tree or the real index's other entries**: it seeds a throwaway index (`GIT_INDEX_FILE` in a tmp file) from `HEAD`, `hash-object -w`s each blob entry (`content` may be a string or a `Buffer`, so binary is byte-exact) and `update-index --add`s it — or `update-index --force-remove`s a `delete` entry — into that throwaway index, then `write-tree` → `commit-tree -p HEAD` → `update-ref HEAD`. Finally it syncs the **real** index for *just those paths* (new blobs via `--cacheinfo`, removals via `--force-remove`), so the committed changes read as clean and only the *other* session's edits remain as unstaged changes.

Because step 3 commits a synthesized blob rather than the shared working file, the other session's edits to the same file are never swept in — they stay in the working tree for that session to commit later (replaying onto the now-updated `HEAD`).

On success `commit-session` **forgets the committed paths** from that session's `edits` *and* `fileOps` maps and re-pushes `session-meta` with the shrunk file list (`trackedFiles()` = union of the two). The renderer's per-session commit button is driven entirely by this list (re-evaluated on every `session-meta` and on select): `Commit N files` while files remain, and a disabled **`Nothing to commit`** once the list is empty. So the button doubles as the commit-state indicator — after a commit it goes disabled until the session edits again, and can be re-used many times across a session. There is no separate persistent "Committed" status label; `#session-commit-msg` only carries the transient *revert* result and commit/revert *errors* for the active session.

**Known ceilings.** Two sessions editing the *exact same lines* → the second replays unclean and falls back to a whole-file commit. `NotebookEdit` is always whole-file. A version of a session's file the user manually staged is overwritten in the index by the synthesized blob. This relies on Claude Code's `PostToolUse` payload exposing `tool_input.old_string`/`new_string`/`content`. For `fileOps`, the executable bit isn't preserved (everything commits as mode `100644`), `.gitignore`d files are never tracked (the text-edit path filters them via `git check-ignore`, the filesystem-diff path because they don't appear in `git status` — the intended behaviour in both), and attribution is per-tool-call — two sessions whose `Bash` calls overlap in time can mis-attribute each other's filesystem changes.

## Per-session revert

The symmetric operation: `revert-session(id)` undoes **only what this session did in the working tree**, leaving another session's edits to the same file intact. It does *not* snapshot — it de-applies the session's own ops.

For each touched file, `inverseEdits(working, ops)` is the inverse of `replayEdits`: it walks the session's ops newest-first and backs each substitution out of the *current* working contents (`Edit` `old→new` is reversed to `new→old`). Because each reversal only rewrites this session's own `new` strings, another session's edits (different text regions) are untouched. The reverted contents are written straight to the working file.

`clean=false` when an op can't be inverted — a full `Write` or opaque op (no stored pre-image), a pure deletion (can't relocate the removed text), or a `new` string that's since been overwritten. Then a hard reset to `HEAD` is only done when **no other live session also edited that file** (`git show HEAD:<path>`, or delete if the file was new this session); if another session shares the file it's **skipped** and reported, since clobbering its work is worse than leaving ours. Successfully reverted files are dropped from the session's `edits` map (skips stay tracked so the user can retry after resolving); a `session-meta` event refreshes the bar's file count.

The session's `fileOps` are undone the same way: a path tracked in `HEAD` is restored with `git checkout HEAD -- <path>` (binary-safe — git writes the committed bytes directly), an `'add'` with no `HEAD` version was new this session so it's `unlink`ed, and a path another live session also touched (`edits` **or** `fileOps`) is **skipped**. Reverted `fileOps` are forgotten too; the `trackedFiles()` count covers both maps.

## Tracking filesystem changes

The text-edit tools (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) are the only ones whose `PostToolUse` payload tells us *what* changed in a file. Everything else a session does to the filesystem — a binary a `Bash` command produced (a build artifact, a downloaded asset, an image an MCP tool generated), a `mv`/`git mv` rename, a file moved in or out of the repo, an `rm` — leaves no replayable op. Those are caught instead by **diffing the git working tree around the tool call** (`sessions.js`):

- On **`PreToolUse`** for any file-touching tool — `tracksFs(name)`: not a text-edit tool and not in the read-only set (`Read`/`Grep`/`Glob`/…) — `recordSessionActivity()` snapshots `git status --porcelain=v1 --untracked-files=all --no-renames` into `s.preStatus` as a `Map<relPath, "XY">`. `--no-renames` makes a rename surface as a separate delete + add, so each path is attributed on its own.
- On **`PostToolUse`** for the same tool, it snapshots again and `applyFsDiff()` compares: a path whose status changed and that **now exists on disk** is recorded `fileOps[abs] = 'add'`; one that's **gone** is recorded `'delete'` — unless it was only ever untracked (never reached `HEAD`, so there's nothing to commit as a deletion), in which case it's simply forgotten. A path already covered by an `edits` op is left to that machinery; a path an edit tracked but that a later tool then moved/removed is dropped from `edits` (its text ops are now void) and recorded as a `fileOp`.

The snapshot must be taken **before** the tool runs, so the hook server `await`s `recordSessionActivity()` *before* answering the hook — a command hook blocks its tool until the `curl` returns, so awaiting the `PreToolUse` snapshot guarantees the "before" state predates the tool's writes. This adds one `git status` of latency to each file-touching tool (none to read-only tools or non-tool events), which is negligible for an interactive session. Because the diff is scoped to `git status`, `.gitignore`d paths are invisible (correctly never committed), and the attribution window is a single tool call — see the per-commit *Known ceilings* for the concurrency caveat.

## Renderer

The renderer is split across `renderer/` modules (`sessions.js`, `git-pane.js`, …); this section covers the session bar + git pane. It keeps a `Map` of sessions, each owning its own xterm.js `Terminal` in a hidden container div. Switching sessions just toggles which container is visible, preserving scrollback. `setState()` sets the status-dot CSS class. Clipboard handling for every terminal (session and git-pane console alike) is the shared `attachClipboard()` in `renderer/shared/terminal.js`: Ctrl+C copies the selection (passing SIGINT through to the PTY only when nothing is selected — key matched case-insensitively so it still fires with Shift held), Ctrl+V / right-click paste, and right-click copies when a selection exists. Copies go through `copyToClipboard()` → main's `clipboard-write` handler (`clipboard.writeText`), because the renderer's async Clipboard API is unreliable under `file://` (not a secure context, so `navigator.clipboard` can be `undefined` and throw synchronously — which is why Ctrl+C silently did nothing) and otherwise needs document focus the canvas can lack right after a selection; a hidden-textarea `execCommand('copy')` remains only as a last resort if the IPC bridge is absent. Text paste likewise reads through main's `clipboard-read` handler. Paste checks the clipboard for an **image** before falling back to text: a PTY can't carry image bytes, so main's `paste-image` handler (`clipboard.readImage()`) spills the bitmap to a temp PNG under `os.tmpdir()/claude-ide-pastes` and returns its path, which the renderer pastes formatted by `opts.formatImagePath` — sessions emit an `@<path>` mention (so Claude reads the image), console shells get the raw path quoted-if-spaced. Because the Claude CLI turns on terminal mouse reporting, a plain mouse drag is forwarded to the app instead of selecting; **hold Shift while dragging** to force a local selection (xterm's built-in override). The git pane re-fetches on a 3s focus-gated poll, after stage/unstage, and after open-folder (there is no manual refresh button — `git-fetch`/`git-pull` and the branch selector cover the remaining needs). A top `#session-bar` shows the active session's name on the left and, on the right, a two-click **Revert** button (`revert-session` — de-applies just this session's edits, see [Per-session revert](#per-session-revert)) and a commit button that calls `commit-session` for that session only; `updateSessionBar()` refreshes them on select (also disarming Revert) and on each `session-meta` event. The commit button shows `Commit N files` from the tracked file count, or a disabled `Nothing to commit` when that count is zero (see [Per-session commit](#per-session-commit)). The name prefers the Haiku-generated title from `session-name` (which also updates the sidebar row label), falling back to the first prompt, then `session <id8>`. Clicking a git row calls `openFile()`, which routes by extension: images/audio open `#asset-view` (see Asset viewer), everything else opens `#diff-view` over the center pane — `renderDiff()` parses git's unified diff into coloured rows with old/new line-number gutters. Both overlays share `closeOverlay()`; selecting a session also hides them. Each row's status badge carries a tooltip from `statusLabel()` (`renderer/shared/git-status.js`, unit-tested) that spells out which column it's in, because the same porcelain letter reads differently across the staged/unstaged split — most confusingly `D`: a **staged deletion** vs a **file deleted on disk but not staged**. Unstaging a staged deletion does *not* clear it — `git reset` restores the file's index entry but leaves it gone on disk, so git re-reports it as an unstaged deletion and the same red `D` just hops to the Changes list; the tooltip points an unstaged deletion at Discard, which is what actually restores the file. Each row also has a two-click discard (rotate-ccw icon) button left of stage/unstage — the first click arms it red, the second calls `git-revert`. The "Changes" header carries matching stage-all (`+`) and discard-all (same rotate-ccw icon, same two-click arm) buttons that loop those calls over every unstaged file. Stage/unstage/discard now surface a failed result through the shared git-error dialog instead of silently no-op'ing. When `gitStatus` returns conflicts, a **Conflicts** section (`#conflicts-section`, hidden when empty) renders above Staged: each row opens the file's diff and carries a ✓ button that `git add`s it once the markers are resolved (resolution itself can be handed to a Claude session); it moves out of the section on the next refresh. The commit box has **Commit** (calls `git-commit`: commits the staged changes, but if nothing is staged it first runs `git add -A` so a bare click behaves like "commit all"; clicking with an **empty** message box has main author one from the staged diff via Haiku — the button disables and shows `Writing commit message…` during the call, then reports `Committed: <subject>`), **Undo** (calls `git-undo` → `git reset --soft HEAD~1`, keeping the changes staged), and **Push** (calls `git-push`, then re-refreshes). The Push button carries a green pill **ahead badge** pinned to its top-right corner (`#git-ahead`) showing the number of unpushed commits — `gitStatus` returns this as `ahead` (`git rev-list --count @{u}..HEAD`, `0` when there is no upstream or no commits yet), and `refreshGit()` hides the badge when the count is zero. The **Pull** button (pane header) mirrors this with its own green **behind badge** (`#git-behind`) showing how many commits a pull would bring in — `gitStatus` returns this as `behind` (`git rev-list --count HEAD..@{u}`, reflecting the last fetch's view of the remote, so it populates after a fetch/poll); while pulling, the button shows the same sweeping progress bar as Push (`.loading`) and disables. Toolbar/action icons are inline Lucide SVGs chosen to read distinctly: discard = `rotate-ccw` (single arrow), undo = `undo-2` (hook arrow), branch = `git-branch`.

**Branch selector.** The git pane head carries a `#git-branch` button showing the current branch name (a `git-branch` icon + `#git-branch-name`, refreshed from `gitStatus().branch` on every `refreshGit()`). Clicking it toggles `#branch-menu`, a popover anchored to the head (`#git-main` is the positioning context) holding a `#branch-search` box and a scrolling `#branch-list`. Opening it fetches `git-branches` fresh (so new branches appear), renders local branches most-recent-first with the current one marked, and the search box substring-filters the list live. The list scrolls past ~15 rows (`max-height`), so a repo with many branches stays usable. Clicking a branch calls `git-checkout` and re-refreshes (errors surface through the shared git-error dialog); the popover dismisses on outside click or Escape. The search box doubles as a **create** field: whenever the typed name isn't already an exact branch, a `+`-prefixed "Create branch" row (`.branch-create`) appears at the top of the list — clicking it, or pressing Enter on a non-matching name, calls `git-create-branch` (`git checkout -b <name>` off the current HEAD) and re-refreshes; pressing Enter on a name that matches an existing branch switches to it instead.

The git pane has two tabs (`#git-tabs`, styled like the session filter tabs): **Changes** (the staged/unstaged lists + commit box, all wrapped in `#git-changes-view`) and **History** (`#git-history-view`). Switching to History calls `refreshHistory()` → `git-log`, caching the commits in `allCommits` and rendering one row per commit (subject + `short · author · relDate` meta line) via `renderHistory()`. A `#history-search` box atop the list filters those cached commits live (no re-fetch): every whitespace-split term must appear in a commit's subject, author, or hash (full or short), case-insensitively — the same rule as the unit-tested `filterCommits()` in `git-parse.js`, replicated in the renderer since it can't `require` main. Clicking a row opens that commit's full patch in the diff overlay via `openCommit()` → `showCommit()` (`renderDiff` with its `fileHeaders` flag on, so each `diff --git` emits a file-name header row and re-derives the highlight language from that path). Each row's two-click **revert** button (same rotate-ccw arm-then-confirm as the discard buttons) calls `git-revert-commit`; on failure it surfaces the shared git-error dialog (now retitled per call via `#git-error-title`). `refreshHistory()` also runs after a commit/undo so the log stays current; it is **not** on the 3s poll (which only refreshes the Changes lists).

## File explorer

The left sidebar is split 50/50 by default: the sessions list on top, a file tree (`#file-tree`) below; a drag-gutter between them resizes the split (see [Resizable panes](#resizable-panes)). The tree is **lazy** — `loadDir(rel, container, depth)` fetches one directory level over `list-dir` (main reads `repoPath/<rel>` and returns `{name, dir}` entries, folders first then alphabetical, `.git` hidden) and only loads a folder's children the first time it's expanded (▸/▾ twisty). Clicking a row selects it (`select(row, rel, dir)` records the last-clicked item in `selected`, for files and folders alike); clicking a file also calls `openFromTree()`, which reuses the git pane's viewers: images/audio → `showAsset()`, everything else → `showFile()` — the **file editor** (see [File editor](#file-editor)), which fetches the file's text over `read-text` and mounts an editable view in the `#diff-view` container (binary files, sniffed by a NUL byte, show `(binary file)`). The tree refreshes on startup, on **Open folder** (button at the pane's bottom), and via its header ⟳ button — no auto-poll. `refreshTree()` **preserves expansion**: before rebuilding it snapshots which folders are open (rows whose twisty is `▾`, keyed by the `data-rel` it stamps on every row), reloads the root, and re-expands those paths recursively, then re-applies the selection highlight — so a refresh (including the one after creating/renaming/deleting a file) leaves the tree looking unchanged apart from the edit. Two more header icons sit before ⟳: **+** (`#files-new`) prompts for a filename (with extension, e.g. `run.py`) through the shared `promptText` modal (`renderer/shared/prompt.js`), then creates the file over `create-file` (main joins it under `repoPath`, refuses paths escaping the root, `mkdir -p`s the parent, writes with the `wx` flag so it fails if the file already exists, and re-prompts with the error otherwise). A **folder+** (`#files-new-folder`) icon sits beside it: it prompts for a folder name and creates it over `create-folder` (same path guards; refuses to clobber an existing entry; `mkdir -p`) via the parallel `createFolderIn(dir)` loop, then refreshes the tree. Next comes a `chevrons-up` **Collapse all** button (`collapseAll`, which just flips every open `▾` twisty back to `▸`). The prompt-and-create loop is `createFileIn(dir)` (re-prompts on error, opens the file on success); `dirOf(rel, dir)` computes a tree item's destination dir — inside it when it's a folder, alongside it (its parent) when it's a file — and `targetDir()` applies that to the toolbar button's `selected`, falling back to the repo root when nothing is selected.

**Context menu & file ops.** Right-clicking a tree row (`contextmenu`) selects it and opens a single floating `#tree-ctx-menu` (built once in `tree.js`, positioned at the cursor and nudged back inside the viewport, dismissed by any outside click or Escape). Its items, each with a leading glyph icon: **New file**, **New folder**, a separator, **Rename**, **Add to chat**, **Copy path**, **Show in file browser**, a separator, then a red **Delete** (`.ctx-danger`). The opener is also exported as `showTreeContextMenu(ev, rel, dir)` so the search-results list can raise the same menu on its (always-file) rows. **New file** runs the shared `createFileIn(dirOf(rel, dir))` against the right-clicked row — so it lands inside a folder or alongside a file (the same flow as the toolbar **+** button); **New folder** does the same through `createFolderIn`. Rename prompts for the new name through `promptText` (pre-filled, **Rename** confirm button) and calls `rename-file` (main `renameSync`, both old/new paths guarded against escaping `repoPath`, refuses to clobber an existing name). Delete confirms through `confirmDialog` (danger variant) and calls `delete-file`, which moves the target to the OS Recycle Bin via `shell.trashItem` (recoverable, not a permanent unlink); failures (rename or delete) report through `confirmDialog` as a one-button notice. Both refresh the tree afterward (expansion preserved, per above) and fix up `selected`. **Copy path** writes the repo-relative path with `navigator.clipboard`; **Add to chat** sends a Claude `@<rel>` mention into the active session via `sendToActiveSession()` (exported from `sessions.js`, writes to the session PTY and refocuses it). **Show in file browser** calls `reveal-in-folder`, which guards the path against escaping `repoPath` and opens the OS file browser with the item selected via `shell.showItemInFolder`.

**Drag to terminal.** Tree **file** rows are `draggable` and put `@<rel>` on the drag's `text/plain`. `#terminal-host` accepts the drop (`dragover` sets `dropEffect = 'copy'`; `drop` reads `text/plain` and feeds it to `sendToActiveSession()`), so dragging a file into the terminal inserts its `@<rel>` mention into the active Claude session — the same text the **Add to chat** menu item sends. The same drop target also accepts **OS file drops** (from Explorer/Finder): those carry no `text/plain`, so the handler falls back to `dataTransfer.files`, resolves each to an absolute path via `api.pathForFile()` (preload-exposed `webUtils.getPathForFile`, since Electron 31 dropped `File.path`), and sends them as space-separated `@<abs>` mentions. A `#file-search` box atop the pane (debounced 150ms) replaces the tree with a `#search-results` list while it has text: **Files** come first from `search-names` (an async recursive walk in main, skipping `.git`/`node_modules`, case-insensitive, capped at 500 — the query is split on whitespace into terms that **all** must match the file's full repo-relative path, so `enemy .mp3` finds files whose path contains both `enemy` and the `.mp3` extension (folder names count too); a `*.png`/`.png` term matches by extension, any other term is a substring of the path, so a bare `png`/`js` gathers that file type too), then a **References** section is filled in asynchronously from `search-refs` (`git grep -n -I -F -i --untracked`, a subprocess so it never blocks the UI; capped at 500) — the filename results render immediately and don't wait on it. A run-token guards against a slow refs response landing after the query changed. Clicking a result opens it via `openFromTree()`; a reference result also passes `{ line, term }` so `showFile()` calls `jumpToLine()` to centre that line in the editor and select the matched word in the textarea. Right-clicking a result raises the same context menu as the tree (via `showTreeContextMenu`), treating the hit as a file. The opened folder isn't labelled in the sidebar; instead `setWindowTitle(repoPath)` sets the OS window title to `IDE / <project name>` (the path's basename), updated on startup (via `refreshGit`) and on **Open folder**.

## Quick Open

A keyboard-driven fuzzy file switcher (`quick-open.js`), the IDE staple for jumping to a file without the tree. **Ctrl/Cmd+P** from anywhere opens a centered palette — the listener is **capture-phase** (like [terminal links](#terminal-links--inline-browser)) so a focused xterm terminal can't swallow the key first, and it `preventDefault`s the browser's own print binding. On open it fetches the repo's flat file list **once** over `list-files` (a main-process async walk in `explorer.js`, same `.git`/`node_modules` skip set as `search-names` but unfiltered, capped at 10k) and caches it for that session of the palette, then fuzzy-matches **client-side on every keystroke** — no per-keystroke IPC, so typing is instant.

The matcher is `renderer/shared/fuzzy.js`, kept DOM-free so it's unit-tested the same way as `git-status.js` (see [testing.md](testing.md)). `fuzzyMatch(query, target)` requires the query to be a subsequence of the path (case-insensitive, whitespace ignored) and scores it by what makes a guess feel right: runs of consecutive characters, matches on a word/path boundary (a separator, `.`/`_`/`-`, or a camelCase hump), and matches inside the **basename** rather than a parent dir; ties break toward the shorter path and earlier first match. It returns the matched indices so the palette can **bold** them. `fuzzyFilter(query, items, limit)` ranks the whole list and keeps the top 100. ↑/↓ move the highlight (wrapping, `scrollIntoView`), Enter opens the highlighted file through the same `openFromTree()` the tree and search use, and Esc / backdrop click dismiss. The list shows `quickOpen.noMatches` when nothing matches and `quickOpen.empty` when the repo has no files.

## File editor

Opening any text file (`showFile()` in `viewer/file.js`) mounts an **always-editable** IDE-style editor in the shared `#diff-view` container — there is no separate read/edit mode. A transparent `<textarea>` (`.editor-ta`) overlays a syntax-highlighted `<pre>` (`.editor-pre`) pixel-for-pixel: the textarea owns native editing (caret, selection, undo/redo), its text is `color: transparent` with a visible `caret-color`, and the highlight behind it shows through. A line-number gutter (`.editor-gutter`) sits left and is `position: sticky; left: 0` so it stays pinned during horizontal scroll. On every `input`, `paint()` re-highlights the whole text (capped at 5000 lines, then plain) and rebuilds the gutter so the overlay stays aligned.

**Edits live in memory until saved.** A `buffers` `Map<file, {value, saved}>` holds each file's current text and last-known on-disk content; `dirty = value !== saved`. Switching files or selecting a session never writes to disk and never discards the buffer — reopening a file restores its unsaved edits. A non-dirty buffer is re-synced from disk on open, so external changes (e.g. a Claude session editing the file) show up. **Save** (the `#diff-save` button in the head, or **Ctrl/Cmd+S**) writes `value` over `write-text` (main guards the path against escaping the repo), sets `saved = value`, and calls `refreshGit()` so the new diff appears immediately. The button is disabled when clean, accented when dirty, and the title shows a `•` while unsaved; it is shown only while a file is mounted (`hideDiff()` retires it, so the diff view never shows it). `Tab` inserts a tab and `Enter` carries the current line's leading indent, both via `document.execCommand('insertText', …)` to preserve the native undo stack.

## Syntax highlighting

Both the diff (`renderDiff`) and the file editor (`paint`) colour code with **highlight.js** — loaded as the prebuilt browser bundle `@highlightjs/cdn-assets` via plain `<script>`/`<link>` in `index.html` (no bundler), exposing `window.hljs`. The `vs2015` theme stylesheet supplies the `.hljs-*` token colours; rows keep their existing `.diff-code` background, so add/del/hunk colouring still shows through.

`langFor(file)` maps a file extension to an hljs language via `EXT_LANG` (covers python, js/ts, c#, c/c++, rust, go, swift, java, kotlin, ruby, php, bash, sql, json, yaml, xml, css, markdown, objc, …). Two grammars aren't in the common bundle and are loaded as extra self-registering scripts: **dos** (`.cmd`/`.bat`) and **powershell** (`.ps1`). Godot is handled without a dedicated grammar: `.gd` → python (GDScript is python-shaped) and `.import`/`.tscn`/`.tres`/`.cfg`/`.godot` → ini. Unknown extensions return `null` → the file viewer auto-detects (`hljs.highlightAuto`); the diff viewer leaves them plain.

- **File editor** highlights the whole text once, then `hlLines()` splits the HTML into per-line fragments, re-opening any span left open across a newline so each line stays balanced (correct for multi-line strings/comments); the fragments are re-joined with `\n` into the highlight `<pre>`.
- **Diff viewer** highlights each line in isolation via `hlLine()` — a hunk is a fragment with no whole-file context, so multi-line constructs may colour imperfectly; hunk headers (`@@`) stay plain.

## Asset viewer

Binary assets have no meaningful text diff, so clicking one in the git pane opens `#asset-view` instead. The renderer fetches the file's bytes as base64 over `read-asset` (`explorer.js` reads it from `repoPath`, returns `{ base64, mime }`) and builds a `data:` URL, then picks one of three views by type:

- **Pixel editor** — PNGs under 200×200. Drawn onto a `<canvas>`, blown up by an integer scale; left-drag paints/erases single pixels with the current palette/`<input type="color">` colour (the colour wheel). The −/+ toolbar buttons darken/lighten the selected colour by lerping toward black/white. Mouse wheel changes the integer zoom; middle-button drag pans (via `assetBody` scroll), while a middle click without dragging eyedrops the pixel underneath. Undo/redo (↶/↷ buttons or Ctrl+Z / Ctrl+Y, capped at 50 `ImageData` snapshots) is pushed once per stroke. **Save** sends `canvas.toDataURL()` bytes back over `write-asset` and refreshes git.
- **Image zoom** — any other image. An `<img>` with −/+/reset buttons scaling its width.
- **Audio player** — `wav`/`ogg`/`mp3`. A fully custom player (the native `<audio controls>` is hidden and driven programmatically): a circular play/pause transport, current/duration readout, a speed-cycle button, and a mute toggle + volume slider. Above it an interactive mirrored bar waveform, decoded once from `AudioContext.decodeAudioData` (base64 → `ArrayBuffer`, no network) and reduced to one normalised peak per bar. The played portion fills with an accent gradient as it plays (driven by a `requestAnimationFrame` loop only while playing); clicking or dragging the waveform seeks, and hovering shows a time bubble. Bars recompute on panel resize via a `ResizeObserver`. The view registers a cleanup that cancels the rAF loop, disconnects the observer, and pauses playback.

`data:` URLs for `<img>`/`<audio>` require the CSP in `index.html` to allow `img-src`/`media-src 'self' data:`.

## Run toolbar

The native Electron application menu is removed (`Menu.setApplicationMenu(null)` in `index.js`); a full-width **run toolbar** (`#toolbar`) sits at the very top of the window instead (`body` is a flex column: toolbar then the `#app` grid). It surfaces VS Code run configs as buttons: **one per launch config** (`.vscode/launch.json`), a separator, then **one per task** (`.vscode/tasks.json`). Clicking a button runs that config/task in a **git-pane terminal tab** (see [Git-pane consoles](#git-pane-consoles)) and focuses it — there is no in-process debugger.

`run-configs.js` parses both files as **JSONC** (`parseJsonc()` strips `//` and `/* */` comments outside strings, then trailing commas, then `JSON.parse`). `get-run-configs` returns the launch names (configs + compounds) and task labels for the toolbar; `run-config({kind, name})` **re-reads** the file each time (so edits are picked up), finds the entry by name, and returns one or more **run specs** `{command, cwd, env, name}` (a compound yields one per referenced configuration). It builds each command line via:

- **Launch** (`buildLaunchCommand`): `node`/`python` types map to `node`/`python <program> <args>` (honouring `runtimeExecutable`/`runtimeArgs`). On **Windows** an extensionless build-tool wrapper used as `runtimeExecutable` is rewritten to its Windows variant (`winExe`: `mvnw`→`mvnw.cmd`, `gradlew`→`gradlew.bat`) — PowerShell can't run the bare Unix script inline, so without this Windows shell-executes it by file association into an **external** window instead of the in-app terminal (the Spring-Boot-via-`mvnw` case). A `godot` type (which carries `project`/`scene` instead of a `program`) maps to `godot --path <project> [<scene>]` — the engine binary defaults to `godot` on PATH (overridable via `runtimeExecutable`/`editor_path`, since the Godot Tools extension keeps it in a VS Code setting, not launch.json), and a `scene` of `main`/`current`/absent runs the project's main scene while an explicit `res://` path runs that scene; other interpreter types in `TYPE_RUNTIME` (`go`→`go run`, `php`, `ruby`, `perl`, `lua`, …) map to their runtime when a `program` is present; anything with a `runtimeExecutable` or bare `program` falls back to running that. A config with none of these (e.g. a browser or attach config) can't be translated, so `run-config` returns `{ok:false, error}` naming the config and its type, and the renderer shows it in the `#run-error-dialog` modal rather than failing silently in the dev console.
- **Task** (`buildTaskCommand`): the `command` (verbatim for `shell` tasks, which may be a whole command line) followed by its quoted `args`. **Compound tasks** are supported via `resolveTask`: a task's `dependsOn` (a label or array of labels) is resolved recursively (cycles broken by a `seen` set) and combined per `dependsOrder` — `"sequence"` collapses the steps into **one** terminal whose commands are chained so each runs only if the previous succeeded (bash/zsh `&&`; Windows PowerShell 5.1 has no `&&`, so `chainCommands` gates each later step on the `$?` success variable instead, and `stepCommand` prepends a `cd`/`Set-Location` for any step with its own `options.cwd`), while `"parallel"`/`"any"` (the default) yields **one terminal per** referenced task. A compound may also carry its own `command`, which runs after its dependencies. A sequence shares a single terminal, so the steps' `options.env` are merged onto it (last wins) rather than applied per step.

`substVars()` resolves the editor-free VS Code variables (`${workspaceFolder}`, `${workspaceFolderBasename}`, `${workspaceRoot}`, `${cwd}`, `${pathSeparator}`, `${env:NAME}`); other `${…}` placeholders are left as-is (best effort). `cwd`/`env` come from the launch config or the task's `options`. A launch config's **`envFile`** (a `.env` path, e.g. Spring Boot's `dev.env`) is read in `run-configs.js` (file IO, kept out of the pure lib) and parsed by `parseEnvFile` (Electron-free, unit-tested: `KEY=VALUE` lines, `#` comments and blanks skipped, `export ` prefix and matching surrounding quotes stripped); those vars seed the spec's env with the config's inline `env` taking precedence, and a missing file is ignored (the config still launches).

The renderer's `runSpecInConsole(spec)` opens (or **reuses** — matched by config name) a terminal tab for each spec, spawning its shell in the spec's `cwd`/`env` and running its `command`; relaunching a config restarts its existing tab with a fresh shell rather than piling up duplicates. The renderer rebuilds the toolbar (`loadToolbar()`) on startup, after **Open folder**, and whenever the `.vscode` files change underneath it: `run-configs.js` keeps an `fs.watchFile` (2s poll) on `.vscode/launch.json` and `.vscode/tasks.json` — robust even before the files exist, firing on create/edit/delete — and pushes a `run-configs-changed` event that the renderer answers with `loadToolbar()`. The watch re-points to the new folder on **Open folder** via `repo.js`'s `onRepoChange` listener registry (fired from `setRepoPath`).

## Resizable panes

Four drag-gutters (`.gutter` divs in `index.html`) let the user resize, **not reorder**, the panes. The column layout is a CSS grid whose left/right tracks are the vars `--left`/`--right` on `#app` (`grid-template-columns: var(--left) 5px minmax(0,1fr) 5px var(--right)`); the center is the remaining `1fr`. The sidebar split is the var `--sess-h` on `#sidebar` (`#sessions-pane`'s flex-basis; `#files-pane` takes the rest). The git pane is itself split vertically: `#git-main` (status/commit) fills the top and `#git-console` takes a `--console-h` flex-basis at the bottom (default `33%`); see [Git-pane console](#git-pane-console).

`panes.js`'s `resizer(gutter, axis, sign, read, write, min, max)` is one generic pointer-drag handler wired four times — left column, right column, sidebar split, git-pane console split. It pointer-captures the gutter, then on move clamps `base + sign*delta` to `[min, max()]` and writes the var. Clamps keep every pane within bounds: columns can't shrink the center below `CENTER_MIN` (200px) or themselves below their mins; the sessions pane stays ≥80px and leaves ≥140px for the explorer; the console stays ≥80px and leaves ≥160px for the git content. `sign` is `-1` for the right and console gutters (their pane is on the far side, so dragging toward it shrinks it). Each move calls `fit()` so the active terminal reflows live, plus `fitConsole()` for the console terminal. Sizes are session-only — not persisted across restarts.

## Git-pane consoles

The bottom of the git pane (`#git-console`) holds a **multi-terminal panel**: a tab strip (`#term-tabs`) over a stack of terminal hosts (`#console-hosts`), each tab owning its own xterm.js `Terminal` + shell PTY. These are independent of the per-session Claude PTYs. In the main process the PTYs live in a `consoles` `Map<id, pty>` (`spawnConsole(id, opts)`); IPC is keyed by id — `term-create` (returns `{id}`), `term-restart` (kill + respawn under the **same** id), `term-input`, `term-resize`, `term-kill`, and the `term-data`/`term-exit` events (both carry `{id}`). `term-shells` lists the shells the user can pick (`availableShells()`: `ps`/`cmd` on Windows, `$SHELL`'s basename otherwise). A terminal can run a command — `spawnConsole` writes `command + '\r'` into the freshly-spawned shell — used by the run toolbar.

In `consoles.js` each tab is a record in the `consoles` `Map` (`{term, fit, host, tab, name, kind}`). `createConsole(opts)` builds the xterm/host/tab and calls `term-create`; `selectConsole(id)` toggles which host is visible (and the active tab) and focuses it; `closeConsole(id)` kills the PTY, disposes the term, drops the tab, and selects a sibling — recreating a default shell if none remain, so a terminal is always open. `term-exit` (e.g. the user typed `exit`) routes to `closeConsole`. Reused id is what lets `term-restart` swap in a fresh shell while `term-data` keeps routing to the same tab; the matching guard in `spawnConsole`'s `onExit` keeps a replaced PTY from reporting its tab as closed.

Tabs are **named**: a manually-created one (the **+** button, `#term-add` — a shell-picker menu when more than one shell is available) takes its shell's name (`cmd`/`ps`); a toolbar-launched one takes the config/task name. Labels are truncated (`truncName`, full name in the tab `title`). The header also carries a **Clear** button (`#term-clear`) that calls `term.clear()` on the active terminal (renderer-only; doesn't touch the shell).

## Terminal links & inline browser

Ctrl+click (Cmd on mac) in a terminal opens what's under the cursor — a web URL in an inline browser, a file path in the explorer's viewer (VS Code's gesture). `registerTerminalLinks(term)` wires each session's xterm via `registerLinkProvider`. The provider **always provides links** (decorated only while the modifier is held) rather than gating `provideLinks` on the key state — xterm only re-queries the provider on mouse movement, so gating there would force the user to press Ctrl *before* hovering and keep the terminal focused. Instead a window-level **capture-phase** `keydown`/`keyup` pair tracks `linkModDown` (capture so a focused panel that stops propagation can't hide the key), the link's `hover`/`leave` callbacks record the currently hovered link, and on any modifier change we flip that link's **live `decorations.underline`/`pointerCursor` setters** (xterm swaps these in once a link is hovered) — so a link lights up the instant Ctrl is held over it, regardless of focus or which happened first. Activation is gated on the modifier (`event.ctrlKey`/`metaKey`) so plain clicks pass through and normal hover/drag-to-select are untouched. `findTerminalLinks(lineText)` scans a row for `http(s)://` URLs first, then path-ish tokens (`PATH_RE`), claiming character ranges so the two never overlap. A path token only counts if it has a separator or a known extension (`looksLikePath` against `PATH_EXT`, built from `EXT_LANG`/`IMG_EXT`/`AUDIO_EXT`); a trailing `:line[:col]` is split off on activation and passed to the viewer's jump.

`openTerminalLink` routes by kind: a URL → `showWeb()`; a path → `resolve-link-path` (main resolves an absolute path as-is, else against `repoPath` = the session cwd, and reports `isFile`/`inRepo`/`rel`), then in-repo files open via `openFromTree(rel, {line})` and anything else goes to the OS through `open-external`. `showWeb()` is a center overlay (`#web-view`, peer of the diff/asset overlays, hidden by the shared `closeOverlay()` and on session select) holding a back/forward/reload bar, an **editable address bar**, an open-externally button, a **⋮ menu**, a **Terminate** button, and an Electron **`<webview>`** — a separate guest process, so the host-page CSP doesn't restrict the loaded site. It requires `webviewTag: true` in the window's `webPreferences`.

**One persistent instance.** There is exactly **one** `<webview>`, reused for every open — the browser is never re-instantiated. Besides Ctrl+click, it's opened from the **browser button** in the top toolbar (`#browser-btn`, left of the settings gear, both pinned to the toolbar's right edge), which calls the center coordinator's `openWeb()` → `web.js`'s `openWeb()`: it just reveals the overlay over the *already-loaded* page (focusing the address bar only when nothing is loaded yet). Crucially, hiding the overlay (session switch, `closeOverlay()`, opening another center view) **no longer unloads the page** — `hideWeb()` only sets `display:none`, so the site keeps running in the background and is shown again as-is on the next open. The toolbar button carries an **`.active`** accent tint whenever a real (non-`about:blank`) page is loaded, so it doubles as a browser-is-alive indicator; `web.js` toggles it from the `did-navigate(-in-page)` events and on terminate. The only thing that unloads the page is the **Terminate** button (`#web-terminate`, ⏻): a destructive action, so it uses the same **two-click arm** confirm as the git pane's discard/revert (armed red on the first click, kills the page — `terminateWeb()` sets the webview back to `about:blank`, clears the active tint — on the second), then returns to the active session via `closeOverlay()`. The terminate/close buttons and the browser button are all wired in the center coordinator (`center.js`), which owns the overlay hand-off.

The webview uses a **persistent partition** (`partition="persist:browser"`), so its cookies and site data live on disk under the instance's `userData` and survive restarts (the `persist:` prefix is what makes a partition on-disk rather than in-memory). The **⋮ menu** is the "clean cookies control": its single **Reset cookies** item confirms (danger `confirmDialog`) then calls `clear-web-data` → `session.fromPartition('persist:browser').clearStorageData({ storages: ['cookies'] })` in main, wiping just that partition's cookies and reloading the page; the app's own session is untouched.

The **address bar** (`#web-url`, an `<input>`) is editable: Enter navigates (`normalizeUrl` keeps a real scheme, else prepends `https://`). It offers **suggestions** from a most-recent-first visited-URL list (`web.history` in `localStorage`, capped at 30): focusing the empty bar shows the latest addresses, typing substring-filters them; a suggestion is chosen on `mousedown` (before the input's blur hides the list). `did-navigate`/`did-navigate-in-page` keep the bar in sync (without clobbering mid-typing) and push each address into the history. The ↗ button hands the current URL to `open-external` (→ `shell.openExternal`).

## preload/

The entire IPC surface (`contextBridge.exposeInMainWorld('api', …)`), split into per-subsystem modules that `preload/index.js` composes. Add a channel to the matching module when wiring any new main↔renderer call. `resolve-link-path` and `open-external` back the terminal link feature above.

## Files

Vanilla JS, no bundler, no test framework. The code is split into per-subsystem modules, not three monolithic files; `index.html` + `styles.css` are the only non-JS UI assets. Within `src/`:

- **`main/`** — the main process, one module per concern, each registering its own `ipcMain` handlers as a load side-effect (`index.js` just requires them in order):
  - `index.js` — app bootstrap: command-line switches, require order, `app` lifecycle.
  - `instance.js` — per-instance `userData` profile redirect (see [platform-notes.md](platform-notes.md)); must load first.
  - `window.js` — the `BrowserWindow` + window title. Owns `sendToRenderer(channel, payload)` — the one path every main→renderer push must use: it no-ops when the renderer is gone/crashed and swallows the dispose race, so a dead renderer can't turn a `webContents.send` into an unhandled main-process crash. On `render-process-gone` it reloads the window (throttled) so the UI recovers instead of freezing.
  - `sessions.js` — the PTY manager (Claude sessions), hook activity recording (text edits + the [filesystem-change diff](#tracking-filesystem-changes) for binaries/renames/deletes), session naming.
  - `hook-server.js` — the localhost hook → status HTTP server (see [status-detection.md](status-detection.md)).
  - `repo.js` — the open repo path, folder picker, last-folder + recent-folders persistence. The Open-folder button is a reverse-combobox: it pops up the last 7 opened projects (`get-recent-folders`) above the button plus a "Browse…" entry; picking a recent one re-points via `open-folder-path`, the dialog via `open-folder`. The capping/dedup logic lives in the Electron-free `recent-folders.js` (`addRecent`), unit-tested (see [testing.md](testing.md)).
  - `git.js` — all `git` porcelain handlers and the shared `git()` helper; the pure stdout parsers (`git-parse.js`: `parsePorcelain`/`parseLog`, plus `filterCommits` for the History search) are split out so they're unit-tested (see [testing.md](testing.md)).
  - `session-commit.js` + `edit-ops.js` — per-session commit/revert (replay/inverse of recorded ops; see [Per-session commit](#per-session-commit)). `edit-ops.js` is pure (Electron-free) and unit-tested.
  - `explorer.js` — file-tree, file-search, and file CRUD handlers.
  - `run-configs.js` + `run-configs-lib.js` — the run toolbar. `run-configs.js` owns file IO, IPC, and the `.vscode` watcher; `run-configs-lib.js` is the Electron-free JSONC parse + launch/task → command translation, unit-tested (see [testing.md](testing.md)).
  - `consoles.js` — the git-pane terminal PTYs.
  - `claude.js` — the shared `runHaiku()` helper (see [Haiku generation](#haiku-generation)).
- **`preload/`** — the `contextBridge` IPC surface, split to mirror the main modules (`index.js` composes them). Add a channel here when wiring any new main↔renderer call.
- **`renderer/`** — pure UI, no Node/OS access. `index.js` is the entry; the rest is grouped by area: `sessions.js`, `git-pane.js`, `toolbar.js`, `consoles.js`, `panes.js`, `settings.js`, `terminal-links.js`, plus `explorer/` (tree + search), `viewer/` (diff, file editor, asset views, web overlay), and `shared/` (highlight, prompt/confirm modals, terminal, ext maps, bootstrap).
- **`i18n/`** — the translation engine (`index.js`) and per-language `locales/` (see [settings.md](settings.md)).
